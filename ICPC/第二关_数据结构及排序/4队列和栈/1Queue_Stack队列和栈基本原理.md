## 1. 什么是「操作受限」的数据结构

前面学的数组和链表，可以对**任意位置**的元素进行增删查改。
这件事很重要，因为它代表“自由度很高”，你想改哪里就改哪里。
但自由度高，也意味着你更容易写出和题意不匹配的操作。
后面学队列和栈时，你会发现“限制”反而是一种保护。

**第一步：最简单的例子（用你能想到的最日常的比喻）**

```cpp
int box[3] = {10, 20, 30};      // 准备3个盒子，先放好3个数字
box[1] = 99;                    // 改第2个盒子里的数字
// 假设有 insert/erase 接口          // 生活中相当于中途插队或让人离队
// 这里想表达：任意位置都能操作      // 这就是“自由操作”的核心
```

> 上面这个小例子只是在讲“任意位置可改可插可删”的感觉，下面的代码和它做的是同一类事，只是写法更贴近你在数据结构课里见到的 API。

```cpp
arr[3] = 99;           // 直接修改任意索引位置的值
arr.insert(2, 666);    // 在任意位置插入一个元素
arr.erase(5);          // 删除任意位置的元素
```

队列和栈是在数组/链表的基础上，**故意删掉了大部分操作，只保留特定的几个**。
你可以把它理解成“给工具加护栏”。
护栏不是让你变慢，而是让你不容易跑偏。
写题时，操作越受限，代码意图通常越清晰。

```
数组/链表：可以操作任意位置
队列：     只能在一端插入，另一端删除
栈：       只能在同一端插入和删除
```

**为什么要「限制」操作？**

限制操作不是缺点，而是刻意设计的。有些场景只需要特定的访问模式，限制操作可以：

- 防止误操作
- 让代码意图更清晰
- 底层实现可以针对性地优化

> 💡 **老师提醒：** 刚入门时最常见错误是“能用数组就全用数组乱改位置”。如果题目是 FIFO/LIFO 语义，你就应该优先用队列/栈，不要绕路。

## 2. 队列

### 特点：先进先出（FIFO）

队列就像**排队买奶茶**。
先到的人先处理，后到的人排在后面。
这个顺序在很多算法里非常关键，比如 BFS 分层遍历。
因为你总是先处理“更早进入队列”的状态，层次才不会乱。

```
入队（队尾插入）：新来的排到最后
出队（队头删除）：排在最前面的先离开

队头                        队尾
 ↓                           ↓
[先来的] → [  ] → [  ] → [后来的]
  ↑ 出队                      ↑ 入队
```

操作示例：

```
初始：空队列 []

push(1)：[1]
push(2)：[1, 2]
push(3)：[1, 2, 3]

pop()：返回1，队列变成 [2, 3]   ← 先进来的1先出去
pop()：返回2，队列变成 [3]
peek()：返回3，但不删除，队列还是 [3]
```

*就像你在窗口叫号，先取号的人先被叫到，而不是看谁跑得快。*

## 3. 栈

### 特点：先进后出（LIFO）

栈就像**一摞盘子**。
你最后放上去的那个盘子，最容易先拿走。
所以栈天然适合处理“最近的状态优先回退”这类问题。
比如函数调用栈、括号匹配，本质上都在用 LIFO。

```
只能从顶部操作（放入/取出）

    ↑ push/pop 都在这里
  ┌─────┐
  │  3  │  ← 最后放的，最先被取走
  ├─────┤
  │  2  │
  ├─────┤
  │  1  │  ← 最先放的，最后被取走
  └─────┘
```

操作示例：

```
初始：空栈 []

push(1)：[1]
push(2)：[1, 2]
push(3)：[1, 2, 3]

pop()：返回3，栈变成 [1, 2]   ← 最后进来的3先出去
pop()：返回2，栈变成 [1]
peek()：返回1，但不删除，栈还是 [1]
```

*就像撤销操作（Undo），你总是先撤销“刚刚那一步”。*

> 💡 **老师提醒：** 很多同学会把队列和栈的 pop 行为记混。你就盯住一句话：队列看“先来先走”，栈看“后进先出”。

> ✅ 能把 FIFO 和 LIFO 用生活场景讲出来，说明你已经理解了，不是死记。

## 4. 基本API

队列和栈的 API 完全一样，只是行为不同。
你可以把它看成“同名接口，不同语义”。
这也是抽象的价值：调用方式一致，内部规则不同。
写题时先想语义，再选具体实现。

**第一步：最简单的例子（用你能想到的最日常的比喻）**

```cpp
struct Bag {                         // 准备一个“袋子”类型
    void put(int x);                 // 往袋子里放东西
    int take();                      // 从袋子里拿东西
};                                   // 这里只看接口，不看内部
```

> 上面这个极简例子强调的是“接口先行”，下面的代码也是同样思路，只是把袋子细分成队列和栈，并且用了模板写成通用类型。

```cpp
// 队列接口：同样是 push/pop/peek/size，但语义是 FIFO
template<typename E>                 // ⚠️ 模板：E 表示元素类型占位符
class MyQueue {
public:
    void push(const E& e);  // ⚠️ const E&：按引用传参且只读，避免拷贝
    E pop();                // 从队头删除并返回元素，O(1)
    E peek() const;         // ⚠️ 函数后 const：承诺不修改成员变量
    int size() const;       // 返回元素个数，O(1)
};

// 栈接口：函数名一致，但语义变成 LIFO
template<typename E>                 // ⚠️ 模板：同一套代码支持 int/string 等类型
class MyStack {
public:
    void push(const E& e);  // 向栈顶插入元素，O(1)
    E pop();                // 从栈顶删除并返回元素，O(1)
    E peek() const;         // 查看栈顶元素（不删除），O(1)
    int size() const;       // 返回元素个数，O(1)
};
```

> 两个问题分开讲。
>
> ---
>
> # 第一个问题：`const E& e` 里的 `&` 为什么在 E 后面
>
> 这里的 `&` 是 **引用** ，声明参数是引用类型。
>
> 你之前学过，`&` 在类型后面就是引用：
>
> **第一步：最简单的例子（用你能想到的最日常的比喻）**
>
> ```cpp
> int apple = 8;                 // 桌上有8个苹果
> int &basket = apple;           // 给这堆苹果起个“别名篮子”
> basket = 10;                   // 改篮子里的数，其实就是改 apple
> // 引用不是复制，是同一个对象的另一个名字
> ```
>
> > 上面的例子在讲“引用=别名”，下面这段代码就是把这个别名机制用到函数参数里。
>
> ```cpp
> int &ref = a;      // 引用变量：ref 和 a 绑定为同一个对象
> void func(int &x)  // 引用参数：函数里对 x 的改动会影响实参
> ```
>
> 所以 `const E& e` 就是 `const E &e`，两种写法完全等价：
>
> **第一步：最简单的例子（用你能想到的最日常的比喻）**
>
> ```cpp
> const int& seatA = number;     // & 靠近类型写法，表示只读引用
> const int &seatB = number;     // & 靠近变量写法，含义完全相同
> // 两行都表示：不拷贝 number，只读访问它
> ```
>
> > 上面两行在语义上完全一致，下面的函数声明也是这个关系，只是把 int 换成了模板类型 E。
>
> ```cpp
> void push(const E& e);   // & 靠近类型，常见写法
> void push(const E &e);   // & 靠近变量名，也对
> ```
>
> 就像之前讲指针时说的：
>
> **第一步：最简单的例子（用你能想到的最日常的比喻）**
>
> ```cpp
> int candy = 5;                 // 手里有5颗糖
> int* p1 = &candy;              // 写法1：* 靠近类型
> int *p2 = &candy;              // 写法2：* 靠近变量
> // 两种写法都表示 p 指向 candy
> ```
>
> > 这个对比和引用写法是同一种“代码风格差异”，下面原代码表达的也是这个点。
>
> ```cpp
> int* p;   // * 靠近类型
> int *p;   // * 靠近变量名
> // 两种写法完全一样，只是风格不同
> ```
>
> ---
>
> # 第二个问题：`const` 是什么
>
> 这里有两个地方出现了 `const`，含义不一样。
> 你只要记位置，就不容易混。
> 放在参数类型前，保护的是“传进来的值”。
> 放在函数后面，保护的是“对象自身状态”。
>
> ---
>
> ## ① 参数里的 const：`const E& e`
>
> 意思是：**这个参数传进来后，函数内部不能修改它。**
>
> **第一步：最简单的例子（用你能想到的最日常的比喻）**
>
> ```cpp
> void readName(const string& name) {   // 只读“姓名标签”
>     // name = "Tom";                // 不能改标签内容
> }                                     // 只读取，不改写
> ```
>
> > 这个小例子表示“只读参数”，下面原代码就是用故意写错的方式告诉你：改 const 参数会报错。
>
> ```cpp
> void push(const E& e) {         // 参数 e 被 const 保护
>     e = 999;  // ❌ 报错！const 参数不能修改
> }
> ```
>
> **为什么要加 const？**
>
> `push` 函数只是把 `e` 插入队列，不需要修改它。加 `const` 是一种承诺：
>
> ```
> "我保证只读这个参数，不会改它"
> ```
>
> 这样调用者就放心了，传进去的数据不会被意外修改。
>
> ---
>
> ## ② 函数后面的 const：`E peek() const`
>
> 意思是：**这个函数不会修改类的任何成员变量。**
>
> 就是上次讲过的 **常量成员函数** ：
>
> **第一步：最简单的例子（用你能想到的最日常的比喻）**
>
> ```cpp
> int watch(const int x) {          // 参数只读
>     // x = 7;                     // 不能改 x
>     return x;                      // 只看不改
> }
> ```
>
> > 上面在讲“只读行为”，下面原代码把这个规则放到成员函数语境：`peek` 只能看，不能改对象状态。
>
> ```cpp
> E peek() const {                 // ⚠️ 函数后 const：this 被视为只读对象
>     size++;  // ❌ 报错！const 函数不能修改成员变量
>     return data;  // ✅ 只读返回，没问题
> }
> ```
>
> `peek` 只是看一下队头元素，不删除也不修改任何东西，所以加 `const` 表示"我保证只读，不写"。
>
> ---
>
> # 对比两种 const 的位置
>
> **第一步：最简单的例子（用你能想到的最日常的比喻）**
>
> ```cpp
> void feed(const int& food) {}      // 参数只读：不改“食物”
> int check() const { return 1; }    // 函数只读：不改“仓库状态"
> // 前者保护参数，后者保护对象成员
> ```
>
> > 上面这两行和下面原代码一一对应，核心就是“位置不同，保护对象不同”。
>
> ```cpp
> void push(const E& e);   // const 在参数前：参数不可修改
> E peek() const;          // const 在函数后：成员变量不可修改
> ```
>
> ---
>
> # 一句话总结
>
> ```
> const E& e     → 传引用（不复制），但保证不修改传入的值
> E peek() const → 这个函数只读数据，保证不修改类的成员变量
>
> 两个 const 位置不同，保护的对象也不同：
>   参数前的 const → 保护传入的参数
>   函数后的 const → 保护类自己的成员变量
> ```

**push 和 pop 的区别：**

|               |   队列   |     栈     |
| ------------- | :------: | :--------: |
| push 插入位置 |   队尾   |    栈顶    |
| pop 删除位置  |   队头   |    栈顶    |
| 规律          | 两端操作 | 同一端操作 |

*就像电梯和抽屉的区别：电梯按“先到楼层先服务”，抽屉永远先拿最上面那层。*

**peek 和 pop 的区别：**

```
pop()：取出元素，同时删除它
peek()：只看一眼，不删除

就像奶茶店：
pop  = 队头的人买完离开
peek = 只是看看队头是谁，人还在
```

> 💡 **老师提醒：** 做题时如果你需要“查看但暂时不移除”，要优先想到 `peek`/`top`/`front`，别一上来就 `pop` 把数据删掉。

### C++ STL 中的对应

**第一步：最简单的例子（用你能想到的最日常的比喻）**

```cpp
#include <queue>                      // 用排队容器
queue<int> line;                      // 建一个“叫号队伍”
line.push(7);                         // 新同学排到队尾
int first = line.front();             // 看看最前面是谁
```

> 上面是最小化队列动作，下面真实代码把队列和栈都列出来，帮助你对照记忆。

```cpp
#include <queue>   // 引入 queue 容器
#include <stack>   // 引入 stack 容器

// 队列：先进先出（FIFO）
queue<int> q;      // 定义一个 int 队列
q.push(1);    // 入队：元素从队尾进入
q.pop();      // 出队：删除队头元素（无返回值）
q.front();    // 查看队头（相当于 peek）
q.size();     // 查看当前元素个数

// 栈：先进后出（LIFO）
stack<int> s;  // 定义一个 int 栈
s.push(1);    // 入栈：元素放到栈顶
s.pop();      // 出栈：删除栈顶元素（无返回值）
s.top();      // 查看栈顶（相当于 peek）
s.size();     // 查看当前元素个数
```

> ⚠️ C++ STL 的 `pop()` 没有返回值，需要先用 `front()`/`top()` 取值，再用 `pop()` 删除。

> ✅ 你现在已经能把“自定义 API”和“STL API”建立一一映射了，这一步非常关键。

## 5. 总结

```
队列和栈都是「操作受限」的数据结构
底层还是数组或链表实现的，只是限制了可以用哪些操作

队列：先进先出（FIFO）
      只能队尾入，队头出
      类比：排队

栈：  先进后出（LIFO）
      只能栈顶入，栈顶出
      类比：一摞盘子

两者的API：push（插入）/ pop（删除）/ peek（查看）/ size（大小）
```

**竞赛中的使用场景：**

| 数据结构 | 典型场景                              |
| -------- | ------------------------------------- |
| 队列     | BFS（广度优先搜索）                   |
| 栈       | DFS（深度优先搜索）、括号匹配、单调栈 |

*就像“按层扩散”选队列，“沿路回退”选栈，场景不同，工具就不同。*
