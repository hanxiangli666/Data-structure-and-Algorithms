# 哈希集合的原理及代码实现 — 学习笔记

---

## 一、一句话搞懂哈希集合

> **哈希集合 = 只用了"键"的哈希表，把"值"扔掉不管。**

就这么简单。如果你已经理解了哈希表，那哈希集合就是白送的知识点。为什么是"白送"？因为哈希集合的所有操作，底层都是原封不动地调用哈希表——只是忽略了 value 这一列。没有任何新的算法，没有新的数据结构，只是一个视角的转换。

---

## 二、哈希集合是干什么用的？

核心场景：**去重**。

它有三个特性：

| 特性 | 说明 |
| ------ | ------ |
| 不会有重复元素 | 加入已存在的元素时，什么都不会发生 |
| O(1) 增删 | 添加和删除元素都是常数时间 |
| O(1) 查找 | 判断一个元素是否存在也是常数时间 |

"不会有重复元素"这个特性不是魔法，而是 `add` 的实现保证的——加入时先检查 key 是否已存在，已存在就不做任何事。这意味着你不需要自己维护"是否重复"的逻辑，集合替你承包了。O(1) 增删查的代价来自哈希表本身：数组随机访问 + 哈希函数，时间复杂度和集合里有多少元素无关。

> **类比**：哈希集合就像一个"签到簿"。每个人只能签一次名（去重），你可以瞬间签到（O(1) 添加），瞬间划掉自己的名字（O(1) 删除），瞬间查看某个人有没有签过到（O(1) 查找）。
> ✅ 三个操作全是 O(1)，这是哈希集合最大的优势——不管集合里有一百个还是一百万个元素，增删查的速度几乎一样。

---

## 三、为什么说它就是哈希表的封装？

回忆哈希表的 API：

```cpp
// 哈希表
mp.put(key, value);    // 插入键值对，O(1)
mp.get(key);           // 通过 key 查 value，O(1)
mp.remove(key);        // 删除键值对，O(1)
```

哈希集合的 API：

```cpp
// 哈希集合
set.add(key);          // 添加元素，O(1)
set.contains(key);     // 判断元素是否存在，O(1)
set.remove(key);       // 删除元素，O(1)
```

对应关系一目了然：

| 哈希集合操作 | 实际上做的事 |
| ------------- | ------------ |
| `add(key)` | `map.put(key, 随便一个占位值)` |
| `contains(key)` | `map.get(key) != null` |
| `remove(key)` | `map.remove(key)` |

value 放什么？**随便放，比如 `true`**。反正我们根本不关心 value 是什么，只关心 key 存不存在。

用 `true` 作为 value 是 C++ 里的惯用写法，bool 类型只占 1 字节，空间浪费最小。有些实现用 `int` 填 1，有些用 `char` 填 'x'，本质都一样——这个 value 永远不会被读取，只是为了满足"哈希表必须有 value"这个格式要求。

> **类比**：哈希表是一本"通讯录"，每个人名（key）对应一个电话号码（value）。哈希集合是一份"签到表"，你只关心名字（key）有没有出现，不需要写电话号码，但格式要求你必须填点什么，那就随便写个"到"（占位符）。
> 💡 **老师提醒：** 这里的"封装"是真正的面向对象意义上的封装——把底层实现（哈希表）隐藏起来，只暴露符合"集合"语义的接口（add/contains/remove）。这样上层代码不需要关心底层是哈希表还是其他结构，只要接口一样，随时可以替换实现。

## 四、代码实现

### 4.1 用 C++ 标准库的 unordered_map 封装

```cpp
#include <unordered_map>

template<typename K>
class MyHashSet {
private:
    // 底层就是一个哈希表
    // value 用 bool 类型当占位符，存什么无所谓
    unordered_map<K, bool> map;

public:
    // 添加元素
    void add(const K& key) {
        map[key] = true;  // value 填 true 作占位
    }

    // 删除元素
    void remove(const K& key) {
        map.erase(key);
    }

    // 判断元素是否存在
    bool contains(const K& key) {
        return map.find(key) != map.end();
    }

    // 返回集合大小
    int size() {
        return map.size();
    }
};
```

**C++ 语法说明**：

- `const K& key`：常量引用传参。`const` 表示不会修改 key，`&` 表示不拷贝直接引用，提升效率
- `map.find(key) != map.end()`：`find` 返回一个迭代器，如果找到了就指向那个元素，没找到就等于 `end()`（表示"超出末尾"的位置）
- `map.erase(key)`：删除指定 key 的键值对，如果 key 不存在也不会报错

`contains` 用 `find` 而不是直接用 `map[key]` 来判断，是有原因的：`map[key]` 在 key 不存在时**会自动插入一个默认值**，悄悄往集合里加了一个元素，完全破坏了"只查不改"的语义。`find` 是纯读操作，不会有任何副作用。

> 💡 **老师提醒：** `map[key]` 这个方括号操作符在 C++ 里是"找不到就创建"的语义，这在 `add` 里是好事，在 `contains` 里是灾难。一定要分清楚：写入用 `map[key] = value`，只查不写用 `map.find(key)`。这是 C++ 中最容易踩的坑之一。

### 4.2 使用示例

```cpp
MyHashSet<int> s;
s.add(1);
s.add(2);
s.add(3);
s.add(2);          // 2 已存在，什么都不发生

s.contains(2);     // true
s.contains(4);     // false

s.remove(2);
s.contains(2);     // false

s.size();           // 2（剩下 1 和 3）
```

注意 `s.add(2)` 被调用两次，`size()` 仍然是 3 而不是 4。集合的去重语义在这里体现得很清楚：第二次 `add(2)` 执行的是 `map[2] = true`，而 map 里已经有 key=2 了，只是更新了一遍相同的 value，元素数量没有变化。

> ✅ 自己写一遍这个封装类，哈希表和哈希集合的关系就彻底清楚了——两者之间的距离，只有一个 value 被忽略的距离。

## 五、竞赛中直接用标准库

实际写题时不需要自己封装，直接用 `unordered_set`：

```cpp
#include <unordered_set>
using namespace std;

unordered_set<int> s;

s.insert(10);          // 添加
s.erase(10);           // 删除
s.count(10);           // 判断是否存在（返回 0 或 1）

// 遍历（顺序不确定）
for (int x : s) {
    cout << x << " ";
}
```

`s.count(10)` 返回 0 或 1，不会返回 2 或更大的数——因为集合里每个元素最多只有一份，count 的含义就是"这个元素存在几个"。这和 `multiset`（允许重复的有序集合）不同，`multiset` 的 count 可以返回任意正整数。

### unordered_set vs set

| | `unordered_set` | `set` |
| --- | ---------------- | ----- |
| 底层实现 | 哈希表 | 红黑树 |
| 增删查 | 平均 O(1) | O(log N) |
| 元素有序？ | 无序 | 自动排序 |
| 适用场景 | 只需要去重/判断存在 | 需要有序遍历 |

选哪个？只有一个判断标准：**你需不需要有序**。需要有序（比如输出结果要从小到大），用 `set`；只需要去重和 O(1) 查找，用 `unordered_set`。别因为"哈希表听起来更高级"就无脑选 `unordered_set`——如果需求是有序的，`set` 的 O(log N) 比乱序的 O(1) 更实用。

*就像快递和普通邮寄的区别——快递（unordered_set）速度快但不保证顺序，挂号信（set）慢一点但你知道它会按规矩来。*

> 💡 **老师提醒：** 竞赛中如果题目要求"按顺序输出"或"找第 k 小的元素"，`set` 是更好的选择。反之，如果只是"判断某个值有没有出现过"，`unordered_set` 更快。先想清楚需求，再选数据结构。

## 六、哈希集合的限制（继承自哈希表）

因为底层就是哈希表，所以哈希表的那些限制都适用：

| 限制 | 原因 |
| ------ | ------ |
| 遍历顺序不可依赖 | 扩容后元素位置会变 |
| 元素必须是不可变类型 | 如果元素变了，hash 值变了，就找不到了 |
| 遍历时不要增删元素 | 可能触发扩容导致混乱 |

"遍历顺序不可依赖"不只是说顺序是随机的，而是说同一个程序每次运行，甚至同一次运行中扩容前后，遍历顺序都可能不同。如果你的算法依赖集合的遍历顺序，结果将是不可预测的。

"元素必须是不可变类型"在 C++ 里尤其要注意。如果你把一个 `vector<int>` 放进 `unordered_set`，再修改那个 vector，哈希值就变了，集合再也找不到这个元素——但它还占着槽位，既删不掉也找不到，内存泄漏且逻辑错误。

> 💡 **老师提醒：** C++ 的 `unordered_set` 不允许存放默认可哈希以外的自定义类型，必须手动提供 `std::hash` 的特化版本。如果你试图 `unordered_set<vector<int>>` 会直接编译报错。这是编译器在保护你，避免你存入不可哈希的类型。

## 七、常见应用场景

| 场景 | 示例 |
| ------ | ------ |
| **去重** | 从数组中找出不重复的元素 |
| **快速判断存在性** | 判断一个单词是否在字典中 |
| **集合运算** | 求两个数组的交集、并集、差集 |
| **标记已访问** | BFS/DFS 中记录已经访问过的节点 |

"标记已访问"是竞赛中最常见的用法之一。BFS/DFS 遍历图时，如果不记录哪些节点已经访问过，就会无限循环。用 `unordered_set` 记录已访问节点，每次入队/入栈前用 `count` 检查一下，O(1) 的代价换来了算法的正确性。

### 经典例题：两个数组的交集

```cpp
// LeetCode 349: 给两个数组，找出它们的交集
vector<int> intersection(vector<int>& nums1, vector<int>& nums2) {
    unordered_set<int> s(nums1.begin(), nums1.end()); // 把 nums1 去重放入集合
    vector<int> result;

    for (int x : nums2) {
        if (s.count(x)) {       // nums2 的元素在集合中存在
            result.push_back(x);
            s.erase(x);         // 避免重复加入结果
        }
    }
    return result;
}
```

这道题的关键思路：先把 nums1 转成集合（去重 + O(1) 查找），再遍历 nums2，每个元素去集合里查一下。`s.erase(x)` 是为了防止 nums2 里有重复元素时，同一个值被加进结果多次——找到一次就删掉，保证结果里每个值只出现一次。

> 💡 **老师提醒：** `unordered_set<int> s(nums1.begin(), nums1.end())` 这种构造方式是 C++ 的范围构造，把一个迭代器范围内的所有元素一次性插入集合，同时完成去重。比写 `for` 循环一个个 `insert` 更简洁，竞赛中常用。
> ✅ 哈希集合的应用场景比你想象的广——只要题目里出现"去重"、"判断有没有"、"已经访问过"这类字眼，第一反应应该是 `unordered_set`。

## 八、知识图谱

```text
哈希集合
├── 本质：只用了 key 的哈希表（value 用占位符填充）
├── 核心能力：去重 + O(1) 增删查
├── API 对应关系
│   ├── add(key)      → map[key] = true
│   ├── contains(key) → map.find(key) != map.end()
│   └── remove(key)   → map.erase(key)
├── 限制（继承自哈希表）
│   ├── 遍历顺序不可依赖
│   ├── 元素必须不可变
│   └── 遍历时不增删
├── C++ 标准库
│   ├── unordered_set → 哈希表实现，O(1)，无序
│   └── set → 红黑树实现，O(log N)，有序
└── 常见应用
    ├── 去重
    ├── 判断存在性
    ├── 集合运算（交集/并集/差集）
    └── BFS/DFS 标记已访问
```
