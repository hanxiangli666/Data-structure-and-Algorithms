## 一、BST 的核心优势

### 1.1 对比：BST 查找 vs 普通二叉树查找

**普通二叉树**找一个值：只能遍历所有节点 → O(N)

**BST** 找一个值：每次和当前节点比大小，排除一半 → O(log N)

你可以把这里的关键抓成一句话：BST 的效率来自“有序性”，不是来自“树形”本身。普通二叉树虽然也是树，但节点没有大小规则，所以每次都不知道该往哪边剪枝。BST 恰好相反，只要比较一次大小，就能排除半边子树。数据规模一大，这个差距就会越来越明显。

> 💡 **老师提醒：** 看到 O(log N) 时，先问自己“是不是每次都能稳定排除一半”。如果做不到，往往就不是对数复杂度。

```
BST 中查找 9：
        7
       / \
      4   9    ← 9 > 7，往右走
     / \    \
    1   5   10

步骤：7 → 9，找到了！只看了 2 个节点

普通二叉树中查找 9：可能要看完所有 6 个节点
```

**类比**：在一本按字母排序的字典里找单词（BST），vs 在一堆散乱的纸条里找单词（普通二叉树）。

*就像“有目录的教科书”和“打乱页码的讲义”的区别。*

### 1.2 BST 所有操作的复杂度

| 操作 | 复杂度（平衡时） | 原理 |
|------|-----------------|------|
| 查找 `get` | O(log N) | 每次排除一半 |
| 插入 `put` | O(log N) | 先查找位置，再改指针 |
| 删除 `remove` | O(log N) | 先查找位置，再改指针 |
| 遍历所有键 `keys` | O(N) | 中序遍历（结果有序！） |

**前提**：树必须是**平衡的**。不平衡的 BST 会退化（后面会讲）。

这句前提非常重要。很多同学看到 BST 就默认 O(log N)，这是不严谨的。真正稳妥的说法是“平衡时是 O(log N)，极端退化时会变成 O(N)”。你现在先把这个条件记住，后面学红黑树就会很顺。

> ✅ 你已经抓住了 BST 最核心的效率来源，继续保持这个“先问前提”的思考习惯。

## 二、TreeMap = 把键值对存在 BST 里

### 2.1 TreeMap vs HashMap

| 对比 | HashMap（哈希表） | TreeMap（BST） |
|------|------------------|---------------|
| 底层结构 | 数组 + 哈希函数 | 二叉搜索树 |
| 键的顺序 | 无序 | **有序（按大小排列）** |
| 增删查改 | O(1) | O(log N) |
| 找最大/最小键 | O(N) 遍历 | **O(log N)** |
| 找某范围内的键 | O(N) 遍历 | **O(log N + M)**，M 为结果数量 |
| 找第 k 小的键 | 不支持 | **O(log N)** |

**关键认知**：

> HashMap 增删查改更快（O(1) vs O(log N)），但**不能处理键的大小关系**。  
> TreeMap 稍慢，但提供了一堆和大小相关的强大操作。

这里你要建立“按需求选工具”的意识。题目只要判断存在性、计数、映射关系时，HashMap 往往更合适。题目一旦出现“第 k 小、前驱后继、区间”这些关键词，TreeMap 的价值就出来了。快不快是一方面，能不能直接支持目标操作更关键。

*就像“快递柜取件”和“图书馆索引检索”的区别。前者拿件快，后者更擅长按顺序和范围查找。*

> 💡 **老师提醒：** 别把 O(1) 当成永远更优。需要有序能力时，强行用哈希表反而会让代码更绕、更容易错。

**类比**：
- HashMap 像一个抽屉柜，每个抽屉上贴了标签，找东西很快，但抽屉之间没有顺序
- TreeMap 像一个从小到大排好序的书架，找东西稍慢一点，但你可以轻松找到「第 3 本书」「比这本厚的下一本」等

### 2.2 TreeSet

TreeSet 和 TreeMap 的关系，就像 HashSet 和 HashMap 的关系：

> **TreeSet = TreeMap 的简单封装，只存 key，不存 value。**

你可以把它理解成“把 value 固定成一个占位符”的 TreeMap。因为很多场景只关心元素是否存在，不需要附带信息，所以单独做成 Set 更直观。接口也更简洁，代码可读性会更好。比赛里如果你只做去重、查找、上下界，优先考虑 set。

> ✅ 工具越用越顺手的标志，就是你能在 map 和 set 之间快速切换。

## 三、TreeMap 的节点结构

这里是从“概念理解”走向“数据结构实现”的桥。你会发现 TreeMap 并不神秘，本质还是树节点，只是把原来的单个值拆成了 key 和 value。这样做的意义是：key 负责排序和查找路径，value 负责承载业务数据。职责拆开以后，结构就清楚了。

**第一步：最简单的例子（用你能想到的最日常的比喻）**

```cpp
int boxLabel = 7;          // 给盒子贴一个编号（像 key）
string boxItem = "apple"; // 盒子里装的东西（像 value）
cout << boxLabel;          // 先按编号找盒子
cout << boxItem;           // 再取出盒子里的内容
```

> 上面这个小例子就是“编号和内容分开存”，下面的真实代码做的是同一件事，只是把它放进树节点里管理。

```cpp
// 普通二叉树节点（LeetCode 风格）
class TreeNode {
  int val;         // 只有一个值，既当比较依据也当数据
  TreeNode* left;  // ⚠️ 指针：指向左子节点
  TreeNode* right; // ⚠️ 指针：指向右子节点
};

// TreeMap 的节点：把 val 换成 key + value
template <typename K, typename V>
class TreeNode {
  K key;          // ⚠️ 模板类型 K：键，决定走左还是走右
  V value;        // ⚠️ 模板类型 V：值，真正存储的业务数据
  TreeNode* left; // 左子节点指针
  TreeNode* right;// 右子节点指针
};
```

和 LeetCode 的 TreeNode 几乎一样，只是把 `val` 拆成了 `key` 和 `value`。

这一步拆分会让后面的 API 设计更自然。比如查找时按 key 比较，返回时拿到 value；更新时通常只改 value，不动 key 对应的树结构。你会发现，很多容器设计都是这个思路。先把“索引字段”和“数据字段”分离。

> 💡 **老师提醒：** key 一旦参与排序，通常不建议在节点就位后随便改它。改 key 往往等价于“删除再插入”。

## 四、TreeMap 的 API 及实现思路

### 4.1 基础操作

#### get(key) —— 查找

从根节点开始，和当前节点的 key 比大小：
- 目标 < 当前 → 往左走
- 目标 > 当前 → 往右走
- 目标 = 当前 → 找到了！

```
查找 key=5：
        7
       / \
      4   9
     / \
    1   5  ← 找到！

路径：7 → 4 → 5（只看了 3 个节点）
```

#### put(key, value) —— 插入/更新

先用 get 的逻辑找位置：
- 找到了 → 更新 value
- 没找到（走到了 nullptr）→ 在那个位置插入新节点

这就是 put 同时负责“新增”和“覆盖”的原因。你不需要先 contains 再决定 insert 还是 update，一次 put 就够了。这样接口更简洁，调用方也不容易写出重复查找。容器设计里，这是一种很常见的模式。

#### remove(key) —— 删除

先找到目标节点，然后分情况处理指针。（具体细节后面习题阶段再学，现在了解思路即可）

删除难点不在“找到节点”，而在“如何保持 BST 结构不乱”。尤其是删除有两个孩子的节点，通常要找前驱或后继来顶替。你现在先记住：删除是局部重连指针，不是简单清空值。后面做题时再把三种情况拆开练。

> 💡 **老师提醒：** 删除题最容易错在漏更新父子指针，画图再写代码，错误率会明显下降。

#### containsKey(key) —— 判断是否存在

就是 get，看返回的是不是空。

这个 API 看起来简单，但特别常用。很多逻辑分支都由“是否存在”决定，比如首次出现计数初始化、重复元素处理等。把它写成单独接口，代码可读性会比直接比较 get 返回值更好。短，但很实用。

> ✅ 到这里你已经把 TreeMap 的基础 CRUD 思路串起来了。

### 4.2 有序性相关操作（TreeMap 独有的优势）

#### firstKey() —— 最小键

从根节点出发，**一直往左走**，最后一个非空节点就是最小的。

```
        7
       / \
      4   9
     / \    \
    1   5   10
    ↑
  一直往左走，1 就是最小键
```

**类比**：BST 是一棵「左小右大」的树，最左边的节点自然最小，最右边的自然最大。

*就像“班级成绩单第一页和最后一页”的区别。第一页最小分，最后一页最大分，不用全表重排。*

#### lastKey() —— 最大键

从根节点出发，**一直往右走**。上面的例子中最大键是 10。

#### floorKey(key) —— ≤ key 的最大键

类似 get，但找不到精确匹配时，返回最接近的**小于等于**目标的键。

```
floorKey(6)：没有 6，返回 5（≤ 6 的最大键）
floorKey(7)：有 7，返回 7
floorKey(0)：没有 ≤ 0 的键，返回 null
```

floorKey 的本质是“向下取整到集合里的最近键”。这在离散化、区间边界、时间轴回溯里都很常见。它并不要求命中原值，只要不超过目标且尽量接近就行。理解这个语义后，你就不容易把它和 lower_bound 搞混。

> 💡 **老师提醒：** floorKey 和 ceilingKey 是一对。做题时常常要成对考虑边界，少写一个就容易漏情况。

#### ceilingKey(key) —— ≥ key 的最小键

floorKey 的反面，返回最接近的**大于等于**目标的键。

你可以把它理解成“向上取整到集合中的最近键”。在调度类题目里，ceilingKey 常用于找“下一班车”“下一个可用资源”。和 floorKey 搭配后，很多“最近点”问题就能直接落地。两者都是有序结构给你的红利。

#### keys() —— 返回所有键（有序）

利用 BST 的中序遍历自带排序的特性：

```
        7
       / \
      4   9
     / \    \
    1   5   10

中序遍历（左→根→右）：[1, 4, 5, 7, 9, 10]  ← 自动从小到大！
```

#### rangeKeys(low, high) —— 区间查询

返回 [low, high] 范围内的所有键。利用 BST 性质剪枝：

```
rangeKeys(4, 9)：

        7      ← 7 在 [4,9] 内，左右都要看
       / \
      4   9    ← 4 和 9 都在范围内
     / \    \
    1   5   10 ← 1 < 4，跳过；5 在范围内；10 > 9，跳过

结果：[4, 5, 7, 9]
```

当前节点 < low → 整棵左子树都不用看（肯定更小）  
当前节点 > high → 整棵右子树都不用看（肯定更大）

这段剪枝思想非常关键。你不是“遍历全树再筛选”，而是“根据范围提前跳过不可能区域”。这就是有序数据结构在区间问题上的效率来源。把这种思维迁移到别的数据结构，你会写出更快的代码。

> ✅ 你已经在用算法思维做减法了：少看不必要的节点，就是优化的起点。

### 4.3 排名相关操作（进阶）

#### selectKey(k) —— 找排名第 k 的键

#### rank(key) —— 找某个键的排名

这两个操作需要在节点中**额外维护一个 size 字段**：

这一步叫“用空间换能力”。多维护一个 size，看起来多了更新成本，但换来了 O(log N) 的排名查询。很多高级数据结构都在做这件事：在节点里维护辅助信息，换取更强查询功能。你现在先建立这个意识，后面学线段树会更顺。

**第一步：最简单的例子（用你能想到的最日常的比喻）**

```cpp
int leftBoxCount = 2;                 // 左边有 2 个盒子
int myRank = leftBoxCount + 1;        // 当前盒子的名次=左边数量+1
cout << myRank;                       // 输出自己的排名
```

> 上面这个例子展示了“左侧数量决定排名”，下面的真实代码做的是同样的事，只是把数量放进树节点的 size 字段。

```cpp
template <typename K, typename V>
class TreeNode {
  K key;          // 键：负责比较大小
  V value;        // 值：负责存储数据
  int size;       // ⚠️ 额外信息：当前子树节点总数（含自己）
  TreeNode* left; // 左子节点
  TreeNode* right;// 右子节点
};
```

```
        7 (size=6)      ← 整棵树 6 个节点
       / \
 (3) 4   9 (size=2)
     / \    \
(1) 1   5  10 (size=1)
       (1)
```

**selectKey(3) 的思路**（找排名第 3 的键）：

```
根节点 7：左子树 size=3，说明 7 排第 4
  → 排名 3 < 4，目标在左子树中

左子节点 4：左子树 size=1，说明 4 排第 2（在左子树中）
  → 排名 3 > 2，目标在右子树中，且在右子树中排名 3-2=1

右子节点 5：左子树 size=0，说明 5 排第 1（在右子树中）
  → 排名 1 == 1，找到了！答案是 5
```

每次只走一边，复杂度 O(log N)。

你可以把 selectKey 看成“按名次找人”的二分过程。每次比较目标名次和左子树规模，决定去左边还是右边。路径始终是一条根到叶的链，不会分叉遍历。复杂度自然就是树高级别。

**rank(9) 的思路**（找键 9 的排名）：

```
根节点 7：9 > 7，目标在右子树
  7 的左子树有 3 个节点 + 7 本身 = 前 4 名确定
  → rank(9) = 4 + rank_in_right_subtree(9)

右子节点 9：9 == 9，找到了！
  9 的左子树有 0 个节点 → 9 在右子树中排第 1
  → rank(9) = 4 + 1 = 5
```

rank 的思路和 selectKey 是互逆关系。一个是“给名次找键”，一个是“给键找名次”。把这对关系理解透，你会对有序统计树的设计非常有感觉。后面遇到 order statistic 题，迁移很快。

## 五、性能问题：平衡 vs 不平衡

### 5.1 平衡的 BST → O(log N)

```
        7           高度 = 3
       / \          节点数 = 6
      4   9         log₂(6) ≈ 2.6 ≈ 3 ✓
     / \    \
    1   5   10
```

  这里的重点是“高度决定操作上限”。查找、插入、删除都沿着根到叶路径走，所以路径最长多长，复杂度就有多大。平衡时高度接近 log N，操作就稳定。你可以把它当成所有树结构复杂度判断的通用模板。

### 5.2 不平衡的 BST → 退化为链表 O(N)

```
    1               高度 = 5
     \              节点数 = 5
      2             这就是一条链表！
       \            所有操作变成 O(N)
        3
         \
          4
           \
            5
```

**什么时候会退化？** 当你按顺序插入 1, 2, 3, 4, 5 时，每个新节点都会被放到右边，树就变成了链表。

这就是输入分布对数据结构性能的影响。算法不是只看代码，还要看数据长什么样。顺序数据在很多结构里都是“天然坏例子”，这一点你要敏感起来。做题前先想最坏输入，常常能提前规避坑。

> 💡 **老师提醒：** 测试时别只喂随机数据，至少再测一组有序和逆序数据。

**类比**：BST 像一棵真的树。如果修剪得当（平衡），树冠丰满，查找路径短；如果放任不管（不平衡），树可能长歪，变成一根藤蔓，失去了树的优势。

### 5.3 解决方案：自平衡

通过在插入/删除时做「**旋转**」操作，让树保持平衡。

```
不平衡：          旋转后（平衡）：
    1                 3
     \               / \
      2             2   4
       \           /     \
        3         1       5
         \
          4
           \
            5
```

最经典的自平衡 BST 是**红黑树**，C++ 标准库的 `map` 和 `set` 底层就是红黑树。

> 红黑树的实现很复杂，目前阶段**不需要掌握**。只要知道：C++ 的 `map`/`set` 已经帮你实现好了自平衡 BST，直接用就行。

学习节奏要对。你现在重点是“会用 + 知道为什么快”，不是“手撕红黑树细节”。先把 map/set 用熟，等你二叉树题量上来，再回头看旋转和染色会轻松很多。该深入的时候再深入，效率更高。

> ✅ 你已经站在正确节奏上了：先用起来，再逐步吃透实现。

## 六、C++ 标准库中的 TreeMap / TreeSet

| 概念 | C++ 标准库 | 底层实现 |
|------|-----------|---------|
| TreeMap | `std::map` | 红黑树（自平衡 BST） |
| TreeSet | `std::set` | 红黑树（自平衡 BST） |
| HashMap | `std::unordered_map` | 哈希表 |
| HashSet | `std::unordered_set` | 哈希表 |

**竞赛中常用的 `map`/`set` 操作**：

**第一步：最简单的例子（用你能想到的最日常的比喻）**

```cpp
map<int, string> seat;              // 像“座位号 -> 同学名字”的登记表
seat[2] = "Li";                    // 2 号座位坐了 Li
cout << seat.begin()->first;        // 看最小座位号
```

> 上面这个小例子是最小化的 map 用法，下面的代码是更完整的竞赛常用操作清单，思路完全一致。

```cpp
#include <map>
#include <set>

// map 基本操作（键值对，按键有序）
map<int, string> m;          // ⚠️ map<键类型, 值类型>：底层红黑树，键自动有序
m[1] = "one";               // 插入或更新 key=1 对应的 value
m.count(1);                 // 查询 key=1 是否存在（结果是 0 或 1）
m.erase(1);                 // 删除 key=1
m.begin()->first;           // begin() 指向最小键，取 first 得到键
m.rbegin()->first;          // rbegin() 指向最大键（反向迭代器）

// 和 TreeMap 的 floorKey / ceilingKey 对应的操作
auto it = m.lower_bound(5);  // ⚠️ auto 自动推导迭代器类型；lower_bound 找到第一个 >=5 的键
auto it = m.upper_bound(5);  // upper_bound 找到第一个 >5 的键
// floorKey 需要稍微绕一下：
auto it = m.upper_bound(5);
if (it != m.begin()) {
  --it;  // 迭代器前移一位，现在 it 指向 <=5 的最大键（近似 floorKey）
}

// set 基本操作（只存键，按键有序）
set<int> s;                 // set 只有键，没有 value
s.insert(3);                // 插入元素 3
s.count(3);                 // 判断 3 是否存在
s.erase(3);                 // 删除元素 3
*s.begin();                 // 取最小元素
*s.rbegin();                // 取最大元素
s.lower_bound(5);           // 找第一个 >=5 的元素
```

**C++ 知识点**：
- `map` 和 `set` 内部元素**自动按键排序**（默认从小到大）
- `begin()` 返回指向最小元素的迭代器
- `rbegin()` 返回指向最大元素的反向迭代器
- `lower_bound(x)` 返回指向 ≥ x 的第一个元素的迭代器
- `upper_bound(x)` 返回指向 > x 的第一个元素的迭代器
- 所有操作 O(log N)

这组 API 你不用一次性硬背。建议你先记住最常用的四个：插入、删除、查找、lower_bound。做题中遇到边界再补 upper_bound、rbegin 等细节。边用边记，比死记快很多。

> 💡 **老师提醒：** 对迭代器解引用前，先判断是否等于 end()，这是防止运行时错误的基本习惯。

## 七、什么时候用 map/set，什么时候用 unordered_map/unordered_set？

| 需求 | 选择 | 原因 |
|------|------|------|
| 只要增删查改，不关心顺序 | `unordered_map` / `unordered_set` | O(1) 更快 |
| 需要按大小顺序遍历 | `map` / `set` | 自动有序 |
| 需要找最大/最小 | `map` / `set` | O(log N) |
| 需要找 ≤ x 或 ≥ x 的最近元素 | `map` / `set` | `lower_bound` / `upper_bound` |
| 需要区间查询 | `map` / `set` | 迭代器范围遍历 |

**竞赛实战建议**：默认用 `unordered_map`/`unordered_set`（更快），只有当题目涉及**排序、排名、区间查询、最大最小**时，才换成 `map`/`set`。

这个建议很实战。你在比赛里先用更快的哈希容器起手，通常没问题。只要需求出现“有序”两个字，就要立刻切换到 map/set。容器选型越快，写题节奏越稳。

*就像“短跑鞋和登山鞋”的区别。平路冲刺选短跑鞋，碰到山路就要换登山鞋。*

> ✅ 你已经具备了“按题意选容器”的工程思维，这比死记 API 更重要。

## 八、关键概念速查表

| 概念 | 说明 |
|------|------|
| **BST** | 左小右大的二叉树，查找效率 O(log N) |
| **TreeMap** | 基于 BST 的键值映射，键自动有序 |
| **TreeSet** | TreeMap 的简单封装，只存键 |
| **自平衡** | 插入/删除时自动旋转保持平衡 |
| **红黑树** | 最经典的自平衡 BST，C++ `map`/`set` 的底层 |
| **退化** | 不平衡的 BST 退化为链表，操作变 O(N) |
| **size 字段** | 额外维护子树节点数，支持排名操作 |
| `lower_bound(x)` | C++ 中 ≥ x 的最小元素（对应 ceilingKey） |
| `upper_bound(x)` | C++ 中 > x 的最小元素 |
