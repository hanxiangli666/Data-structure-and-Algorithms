## 一、BST 的核心优势

### 1.1 对比：BST 查找 vs 普通二叉树查找

**普通二叉树**找一个值：只能遍历所有节点 → O(N)

**BST** 找一个值：每次和当前节点比大小，排除一半 → O(log N)

```
BST 中查找 9：
        7
       / \
      4   9    ← 9 > 7，往右走
     / \    \
    1   5   10

步骤：7 → 9，找到了！只看了 2 个节点

普通二叉树中查找 9：可能要看完所有 6 个节点
```

**类比**：在一本按字母排序的字典里找单词（BST），vs 在一堆散乱的纸条里找单词（普通二叉树）。

### 1.2 BST 所有操作的复杂度

| 操作 | 复杂度（平衡时） | 原理 |
|------|-----------------|------|
| 查找 `get` | O(log N) | 每次排除一半 |
| 插入 `put` | O(log N) | 先查找位置，再改指针 |
| 删除 `remove` | O(log N) | 先查找位置，再改指针 |
| 遍历所有键 `keys` | O(N) | 中序遍历（结果有序！） |

**前提**：树必须是**平衡的**。不平衡的 BST 会退化（后面会讲）。

## 二、TreeMap = 把键值对存在 BST 里

### 2.1 TreeMap vs HashMap

| 对比 | HashMap（哈希表） | TreeMap（BST） |
|------|------------------|---------------|
| 底层结构 | 数组 + 哈希函数 | 二叉搜索树 |
| 键的顺序 | 无序 | **有序（按大小排列）** |
| 增删查改 | O(1) | O(log N) |
| 找最大/最小键 | O(N) 遍历 | **O(log N)** |
| 找某范围内的键 | O(N) 遍历 | **O(log N + M)**，M 为结果数量 |
| 找第 k 小的键 | 不支持 | **O(log N)** |

**关键认知**：

> HashMap 增删查改更快（O(1) vs O(log N)），但**不能处理键的大小关系**。  
> TreeMap 稍慢，但提供了一堆和大小相关的强大操作。

**类比**：
- HashMap 像一个抽屉柜，每个抽屉上贴了标签，找东西很快，但抽屉之间没有顺序
- TreeMap 像一个从小到大排好序的书架，找东西稍慢一点，但你可以轻松找到「第 3 本书」「比这本厚的下一本」等

### 2.2 TreeSet

TreeSet 和 TreeMap 的关系，就像 HashSet 和 HashMap 的关系：

> **TreeSet = TreeMap 的简单封装，只存 key，不存 value。**

## 三、TreeMap 的节点结构

```cpp
// 普通二叉树节点（LeetCode 风格）
class TreeNode {
    int val;
    TreeNode* left;
    TreeNode* right;
};

// TreeMap 的节点：把 val 换成 key + value
template <typename K, typename V>
class TreeNode {
    K key;          // 键（用来比较大小、决定左右位置）
    V value;        // 值（存储的数据）
    TreeNode* left;
    TreeNode* right;
};
```

和 LeetCode 的 TreeNode 几乎一样，只是把 `val` 拆成了 `key` 和 `value`。

## 四、TreeMap 的 API 及实现思路

### 4.1 基础操作

#### get(key) —— 查找

从根节点开始，和当前节点的 key 比大小：
- 目标 < 当前 → 往左走
- 目标 > 当前 → 往右走
- 目标 = 当前 → 找到了！

```
查找 key=5：
        7
       / \
      4   9
     / \
    1   5  ← 找到！

路径：7 → 4 → 5（只看了 3 个节点）
```

#### put(key, value) —— 插入/更新

先用 get 的逻辑找位置：
- 找到了 → 更新 value
- 没找到（走到了 nullptr）→ 在那个位置插入新节点

#### remove(key) —— 删除

先找到目标节点，然后分情况处理指针。（具体细节后面习题阶段再学，现在了解思路即可）

#### containsKey(key) —— 判断是否存在

就是 get，看返回的是不是空。

### 4.2 有序性相关操作（TreeMap 独有的优势）

#### firstKey() —— 最小键

从根节点出发，**一直往左走**，最后一个非空节点就是最小的。

```
        7
       / \
      4   9
     / \    \
    1   5   10
    ↑
  一直往左走，1 就是最小键
```

**类比**：BST 是一棵「左小右大」的树，最左边的节点自然最小，最右边的自然最大。

#### lastKey() —— 最大键

从根节点出发，**一直往右走**。上面的例子中最大键是 10。

#### floorKey(key) —— ≤ key 的最大键

类似 get，但找不到精确匹配时，返回最接近的**小于等于**目标的键。

```
floorKey(6)：没有 6，返回 5（≤ 6 的最大键）
floorKey(7)：有 7，返回 7
floorKey(0)：没有 ≤ 0 的键，返回 null
```

#### ceilingKey(key) —— ≥ key 的最小键

floorKey 的反面，返回最接近的**大于等于**目标的键。

#### keys() —— 返回所有键（有序）

利用 BST 的中序遍历自带排序的特性：

```
        7
       / \
      4   9
     / \    \
    1   5   10

中序遍历（左→根→右）：[1, 4, 5, 7, 9, 10]  ← 自动从小到大！
```

#### rangeKeys(low, high) —— 区间查询

返回 [low, high] 范围内的所有键。利用 BST 性质剪枝：

```
rangeKeys(4, 9)：

        7      ← 7 在 [4,9] 内，左右都要看
       / \
      4   9    ← 4 和 9 都在范围内
     / \    \
    1   5   10 ← 1 < 4，跳过；5 在范围内；10 > 9，跳过

结果：[4, 5, 7, 9]
```

当前节点 < low → 整棵左子树都不用看（肯定更小）  
当前节点 > high → 整棵右子树都不用看（肯定更大）

### 4.3 排名相关操作（进阶）

#### selectKey(k) —— 找排名第 k 的键

#### rank(key) —— 找某个键的排名

这两个操作需要在节点中**额外维护一个 size 字段**：

```cpp
template <typename K, typename V>
class TreeNode {
    K key;
    V value;
    int size;       // 以当前节点为根的子树的节点总数（含自身）
    TreeNode* left;
    TreeNode* right;
};
```

```
        7 (size=6)      ← 整棵树 6 个节点
       / \
 (3) 4   9 (size=2)
     / \    \
(1) 1   5  10 (size=1)
       (1)
```

**selectKey(3) 的思路**（找排名第 3 的键）：

```
根节点 7：左子树 size=3，说明 7 排第 4
  → 排名 3 < 4，目标在左子树中

左子节点 4：左子树 size=1，说明 4 排第 2（在左子树中）
  → 排名 3 > 2，目标在右子树中，且在右子树中排名 3-2=1

右子节点 5：左子树 size=0，说明 5 排第 1（在右子树中）
  → 排名 1 == 1，找到了！答案是 5
```

每次只走一边，复杂度 O(log N)。

**rank(9) 的思路**（找键 9 的排名）：

```
根节点 7：9 > 7，目标在右子树
  7 的左子树有 3 个节点 + 7 本身 = 前 4 名确定
  → rank(9) = 4 + rank_in_right_subtree(9)

右子节点 9：9 == 9，找到了！
  9 的左子树有 0 个节点 → 9 在右子树中排第 1
  → rank(9) = 4 + 1 = 5
```

## 五、性能问题：平衡 vs 不平衡

### 5.1 平衡的 BST → O(log N)

```
        7           高度 = 3
       / \          节点数 = 6
      4   9         log₂(6) ≈ 2.6 ≈ 3 ✓
     / \    \
    1   5   10
```

### 5.2 不平衡的 BST → 退化为链表 O(N)

```
    1               高度 = 5
     \              节点数 = 5
      2             这就是一条链表！
       \            所有操作变成 O(N)
        3
         \
          4
           \
            5
```

**什么时候会退化？** 当你按顺序插入 1, 2, 3, 4, 5 时，每个新节点都会被放到右边，树就变成了链表。

**类比**：BST 像一棵真的树。如果修剪得当（平衡），树冠丰满，查找路径短；如果放任不管（不平衡），树可能长歪，变成一根藤蔓，失去了树的优势。

### 5.3 解决方案：自平衡

通过在插入/删除时做「**旋转**」操作，让树保持平衡。

```
不平衡：          旋转后（平衡）：
    1                 3
     \               / \
      2             2   4
       \           /     \
        3         1       5
         \
          4
           \
            5
```

最经典的自平衡 BST 是**红黑树**，C++ 标准库的 `map` 和 `set` 底层就是红黑树。

> 红黑树的实现很复杂，目前阶段**不需要掌握**。只要知道：C++ 的 `map`/`set` 已经帮你实现好了自平衡 BST，直接用就行。

## 六、C++ 标准库中的 TreeMap / TreeSet

| 概念 | C++ 标准库 | 底层实现 |
|------|-----------|---------|
| TreeMap | `std::map` | 红黑树（自平衡 BST） |
| TreeSet | `std::set` | 红黑树（自平衡 BST） |
| HashMap | `std::unordered_map` | 哈希表 |
| HashSet | `std::unordered_set` | 哈希表 |

**竞赛中常用的 `map`/`set` 操作**：

```cpp
#include <map>
#include <set>

// map 基本操作（键值对，按键有序）
map<int, string> m;
m[1] = "one";              // 插入/更新
m.count(1);                // 判断是否存在（返回 0 或 1）
m.erase(1);                // 删除
m.begin()->first;          // 最小键（firstKey）
m.rbegin()->first;         // 最大键（lastKey）

// 和 TreeMap 的 floorKey / ceilingKey 对应的操作
auto it = m.lower_bound(5);  // ≥ 5 的最小键 → ceilingKey(5)
auto it = m.upper_bound(5);  // > 5 的最小键
// floorKey 需要稍微绕一下：
auto it = m.upper_bound(5);
if (it != m.begin()) {
    --it;  // 现在 it 指向 ≤ 5 的最大键
}

// set 基本操作（只存键，按键有序）
set<int> s;
s.insert(3);               // 插入
s.count(3);                // 判断是否存在
s.erase(3);                // 删除
*s.begin();                // 最小元素
*s.rbegin();               // 最大元素
s.lower_bound(5);          // ≥ 5 的最小元素
```

**C++ 知识点**：
- `map` 和 `set` 内部元素**自动按键排序**（默认从小到大）
- `begin()` 返回指向最小元素的迭代器
- `rbegin()` 返回指向最大元素的反向迭代器
- `lower_bound(x)` 返回指向 ≥ x 的第一个元素的迭代器
- `upper_bound(x)` 返回指向 > x 的第一个元素的迭代器
- 所有操作 O(log N)

## 七、什么时候用 map/set，什么时候用 unordered_map/unordered_set？

| 需求 | 选择 | 原因 |
|------|------|------|
| 只要增删查改，不关心顺序 | `unordered_map` / `unordered_set` | O(1) 更快 |
| 需要按大小顺序遍历 | `map` / `set` | 自动有序 |
| 需要找最大/最小 | `map` / `set` | O(log N) |
| 需要找 ≤ x 或 ≥ x 的最近元素 | `map` / `set` | `lower_bound` / `upper_bound` |
| 需要区间查询 | `map` / `set` | 迭代器范围遍历 |

**竞赛实战建议**：默认用 `unordered_map`/`unordered_set`（更快），只有当题目涉及**排序、排名、区间查询、最大最小**时，才换成 `map`/`set`。

## 八、关键概念速查表

| 概念 | 说明 |
|------|------|
| **BST** | 左小右大的二叉树，查找效率 O(log N) |
| **TreeMap** | 基于 BST 的键值映射，键自动有序 |
| **TreeSet** | TreeMap 的简单封装，只存键 |
| **自平衡** | 插入/删除时自动旋转保持平衡 |
| **红黑树** | 最经典的自平衡 BST，C++ `map`/`set` 的底层 |
| **退化** | 不平衡的 BST 退化为链表，操作变 O(N) |
| **size 字段** | 额外维护子树节点数，支持排名操作 |
| `lower_bound(x)` | C++ 中 ≥ x 的最小元素（对应 ceilingKey） |
| `upper_bound(x)` | C++ 中 > x 的最小元素 |
