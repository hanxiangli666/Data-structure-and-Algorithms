# 第八课：竞赛输入输出优化

## 为什么需要优化？

有些题目输入数据量非常大，比如读入100万个数字。默认的 `cin/cout` 比较慢，可能导致超时。

`cin` 慢的根本原因是它默认和 C 语言的 `scanf` 保持同步，而且每次输出前会先刷新缓冲区——这两个机制在保证安全性的同时也带来了明显的性能开销。数据量小的时候感觉不出来，一旦到了 100 万 ~ 1000 万级别就会出问题。

---

## 一行代码解决：加速cin/cout

```cpp
ios::sync_with_stdio(false);
cin.tie(nullptr);
```

把这两行加在 `main` 函数开头就行了，速度能提升 **3-5倍** 。

```cpp
int main() {
    ios::sync_with_stdio(false);  // 关闭cin和scanf的同步
    cin.tie(nullptr);              // 解除cin和cout的绑定

    // 后面正常用cin/cout就行
    int n;
    cin >> n;
}
```

就这两行。不需要改任何其他代码，后面正常用 `cin/cout` 就行，完全透明。

> 💡 **老师提醒：** 加了这两行之后，**不能再混用 `scanf/printf` 和 `cin/cout`**。混用会导致输入输出顺序错乱。竞赛里统一用 `cin/cout` 就没问题，不要两套混用。

---

## 两行代码的含义（了解即可）

**`ios::sync_with_stdio(false)`**

C++有两套输入输出：

* C语言的：`scanf / printf`
* C++的：`cin / cout`

默认情况下两套是同步的（可以混用），同步本身有开销。关掉同步后 `cin/cout` 会快很多。

代价是： **关掉之后不能混用 `scanf/printf` 和 `cin/cout`** ，竞赛中统一用 `cin/cout` 就没问题。

**`cin.tie(nullptr)`**

默认情况下每次用 `cin` 读取前，会先强制刷新 `cout` 的缓冲区（确保输出先显示出来）。解除绑定后就不会自动刷新了，速度更快。

这个设计的初衷是交互式终端——你输出了一个提示语"请输入："，希望它在用户输入前先显示出来。但竞赛里不需要这种实时刷新，解绑之后纯粹提速。

---

## 二、用 `\n` 代替 `endl`

```cpp
cout << "hello" << endl;   // 慢：输出并强制刷新缓冲区
cout << "hello" << "\n";   // 快：只输出换行，不刷新
```

`endl` 每次都会刷新缓冲区，输出量大时很慢。竞赛中统一用 `"\n"`。

`endl` 的刷新操作相当于告诉系统"立刻把缓冲区里的内容写到屏幕/文件上"，这个 I/O 操作本身很耗时。`"\n"` 只是往缓冲区里加一个换行符，等缓冲区满了或程序结束时再一次性写出，效率高很多。*就像 `endl` 和 `"\n"` 的区别，就像每写一行就马上打印一次，和把整篇文章写完再一起打印的区别——后者省的就是那些中途启动打印机的时间。*

> 💡 **老师提醒：** 加了 `ios::sync_with_stdio(false)` 之后，`endl` 的影响更显著——因为 cin/cout 现在更快了，`endl` 的刷新开销占比反而更大。所以这两个优化要一起用：加速头 + 换掉 `endl`。

> ✅ 两行加速代码 + `\n` 替代 `endl`，三招合一，I/O 效率直接上一个台阶！

## 三、完整的竞赛模板

把前面所有课学的东西整合起来，这就是你以后每道题的起点：

```cpp
#include <iostream>
#include <vector>
#include <algorithm>
#include <string>
#include <numeric>
#include <cmath>
using namespace std;

typedef long long ll;  // 以后写ll就是long long

const int MOD = 1e9 + 7;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    // 在这里写题目逻辑

    return 0;
}
```

把这个模板背下来，以后每道题直接从这里开始写。这个模板包含了你目前学过的所有常用头文件、long long 简写、MOD 常量和 I/O 优化——它是你进入算法阶段的标准起跑线。

> 💡 **老师提醒：** 竞赛中如果用到 `gcd`，还需要加 `#include <numeric>`（已在模板里）。如果用到 `unordered_map`，需要 `#include <unordered_map>`。模板里的头文件不是全包的，遇到编译报错"找不到某函数"，先检查是不是缺头文件。

## 四、scanf/printf（了解即可）

有些老选手喜欢用C语言风格的输入输出，你可能在别人代码里看到：

```cpp
scanf("%d", &n);        // 读一个int
scanf("%lld", &n);      // 读一个long long
printf("%d\n", n);      // 输出一个int
printf("%lld\n", n);    // 输出一个long long
```

这个你不需要主动用，看懂别人代码就行。`scanf/printf` 需要手动指定格式字符串（`%d`、`%lld` 等），容易写错格式符，现代竞赛基本都用加了优化的 `cin/cout`。
