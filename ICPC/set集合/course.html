集合：天生就去重

为什么需要集合？它解决了什么问题？

想象一个签到表

比赛里有人反复提交同一个答案。你需要统计有多少人参与了，而不是提交了多少次。这时候，普通数组会忠实地记录每一次——包括重复的。

💡 关键洞察

set 的"不重复"特性是自动的，你不需要手动检查。这让去重问题从 O(n) 的手动遍历变成了一行代码。在竞赛里，这能省掉大量时间和出错机会。

insert、count、erase、size、遍历——逐行读懂代码

CODE (set.cpp) 
set<int> s;

s.insert(30);
s.insert(10);
s.insert(50);
s.insert(10);  // 重复插入，自动忽略

cout << s.size() << "\n";   // 3

打印"10 存在"

从集合里删掉 10

现在只剩 {30, 50}，size 变成 2

💡 count 的特别之处

在 set 里，count(x) 只返回 0 或 1，本质上是"存在性检查"。在竞赛中，if (s.count(x)) 是判断元素是否在集合内的标准写法。

03

set 的超能力：边界查找

lower_bound 与 upper_bound——找到"最接近"的元素

比 count 更强的查找

count 只能告诉你某个值存不存在。但在竞赛里，你经常需要找"第一个不小于 x 的元素"或"第一个大于 x 的元素"。这就是 lower_bound 和 upper_bound 的用武之地。

📚 比喻：书架查找

想象书架上按编号排好的书：{20, 30, 40, 50}。
lower_bound(30)：找第一本"编号 ≥ 30"的书 → 指向 30。
upper_bound(30)：找第一本"编号 > 30"的书 → 指向 40。

可视化：set 中查找 30

当前 set：{20, 30, 40, 50}，目标值 = 30

20
30
40
50
lower_bound(30) → 第一个 ≥ 30，指向 30
upper_bound(30) → 第一个 > 30，指向 40

代码 × 白话对照

CODE (set.cpp) 
s.insert(20);
s.insert(40);
// 现在 set：{20, 30, 40, 50}

auto it1 = s.lower_bound(30);
cout << *it1 << "\n";  // 30

auto it2 = s.upper_bound(30);
cout << *it2 << "\n";  // 40
白话解释

插入 20 和 40，集合变成 {20, 30, 40, 50}

auto 让编译器自动推断类型，it1 是一个迭代器

lower_bound(30)：找第一个 ≥ 30 的元素，it1 指向 30

用 * 解引用迭代器，打印出 30

upper_bound(30)：找第一个 > 30 的元素，it2 指向 40

打印出 40

⚡ 竞赛常用技巧

lower_bound 和 upper_bound 的时间复杂度是 O(log n)，远快于手动遍历查找的 O(n)。在数据量大时，这个差距会非常显著——100万个元素，前者约 20 次，后者需要 100 万次。

04

unordered_set：速度优先

哈希表的魔法：O(1) 的查找，代价是放弃顺序

储物柜编号系统

哈希表的工作原理就像体育馆的储物柜：

1
计算"柜号"
插入元素时，对数值进行哈希运算，得到一个"柜号"（数组下标）
2
直接放进对应柜子
不需要和其他元素比大小，直接定位——这就是为什么 O(1) 比 O(log n) 快
3
查找时也用同样的柜号
查找 10？计算 hash(10)，直接去那个柜子看——找不到就是不存在
⚠️ 代价

柜子是按"柜号"分布的，不是按元素大小排序的。所以 unordered_set 遍历时顺序是不可预测的，也没有 lower_bound / upper_bound。

和 set 接口几乎一样

这是 unordered_set.cpp 的对照：

CODE (unordered_set.cpp) 
unordered_set<int> s;

s.insert(30);
s.insert(10);
s.insert(50);
s.insert(10);  // 重复插入，自动忽略

cout << s.size() << "\n";  // 3

if (s.count(10)) {
    cout << "10 存在\n";
}

// 遍历：顺序随机
for (int x : s) {
    cout << x << " ";
}
白话解释

创建无序集合 s，底层用哈希表

插入 30, 10, 50；第二次插入 10 被忽略

大小为 3，与 set 行为相同

count(10) 返回 1，说明 10 在里面

for 循环遍历——但输出顺序不定！

可能是"50 30 10"也可能是其他顺序

两者的"性格差异"

05

如何选择？

场景决策指南——竞赛里什么时候用哪个

一张表看清差异

特性 set unordered_set
底层结构 红黑树 哈希表
元素顺序 自动升序 随机，不保证
insert / erase / count O(log n) 平均 O(1)
lower_bound / upper_bound ✓ 支持 ✗ 不支持
最坏时间复杂度 O(log n)，稳定 O(n)（哈希冲突时）
内存占用 略高（树节点指针） 略低（数组 + 链表）

竞赛决策流程

?
需要找"最接近某值"的元素？
比如"大于 x 的最小值"、范围查询 → 用 set（lower_bound / upper_bound）
?
只需要去重 + 判断存在？
不在乎顺序，只关心"这个数出现过吗" → 用 unordered_set（更快）
?
需要按顺序遍历所有元素？
比如输出所有不同数的升序列表 → 用 set（自动排好序）
?
数据有"反哈希"风险？
竞赛题有时会有专门卡 unordered_set 的数据 → 安全起见用 set
🏆 竞赛经验

ICPC 中 set 更常见，因为它稳定可靠且支持边界查找。当题目数据量极大（>10⁶）且只需存在性检查时，unordered_set 能帮你踩在时间限制线内通过。
