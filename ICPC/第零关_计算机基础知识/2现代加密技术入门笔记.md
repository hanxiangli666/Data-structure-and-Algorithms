# 现代加密技术入门笔记

> **核心概览**：现代加密技术是信息安全的基石，主要包括三种核心技术：**对称加密**、**哈希函数**、**非对称加密**。本笔记结合 `openssl` 命令实操，帮你理解它们的原理、边界和协作方式。

---

## 零、三种技术的适用场景速查

先看全局，再看细节。

| 技术 | 核心作用 | 速度 | 是否可逆 | 最适合做什么 | 不适合做什么 |
|------|----------|------|----------|--------------|--------------|
| **对称加密**（AES） | 用同一密钥加解密，保护数据机密性 | 快 | 可逆 | 大文件/大量数据加密（磁盘、数据库、通信数据体） | 大规模密钥分发 |
| **哈希函数**（SHA256） | 生成固定长度摘要，做完整性校验 | 很快 | 不可逆 | 文件校验、密码存储摘要、数字签名前摘要 | 直接当作“加密”来保密数据 |
| **非对称加密**（RSA/ECC） | 公私钥分离，解决身份与密钥交换问题 | 慢 | 可逆（加密场景） | 密钥交换、数字签名、证书体系（HTTPS/SSH） | 直接加密大体量业务数据 |

## 一、对称加密：一把钥匙开一把锁

### 1.1 核心思想

加密和解密使用**同一个密钥**。

> **类比**：对称加密就像你和朋友约定了一个“暗号规则” - 比如每个字母往后移 3 位。你们都知道这个规则，所以你能加密，朋友能解密。但问题是，你怎么安全地把这个“规则”告诉朋友？

### 1.2 从凯撒密码说起

最古老的对称加密算法 - **凯撒密码（Caesar Cipher）**：

```text
明文:  h  e  l  l  o
       ↓  ↓  ↓  ↓  ↓    每个字母向后移动 3 位（密钥 = 3）
密文:  k  h  o  o  r
```

| 术语 | 对应内容 |
|------|---------|
| 明文 | `hello`（原始信息） |
| 密文 | `khoor`（加密后的信息） |
| 密钥 | `3`（移动的位数） |

凯撒密码太简单，很容易被暴力破解。现代对称加密算法（如 **AES**、**DES**）复杂得多，但本质一样：通信双方**共享同一个密钥**。

### 1.3 实操：用 openssl 体验 AES 加密

```bash
# 1. 创建测试文件
echo "This is a secret message" > secret.txt

# 2. 加密：使用 AES-256-CBC 算法，密码为 mypassword
openssl enc -aes-256-cbc -salt -in secret.txt -out secret.enc -k mypassword

# 3. 解密
openssl enc -aes-256-cbc -d -in secret.enc -out decrypted.txt -k mypassword

# 4. 查看解密结果
cat decrypted.txt
# 输出: This is a secret message
```

**命令参数解读**：

| 参数 | 含义 |
|------|------|
| `enc -aes-256-cbc` | 使用 AES-256-CBC 算法加密 |
| `-salt` | 添加随机盐值，增加安全性 |
| `-k mypassword` | 指定加密密码 |
| `-d` | 执行解密操作（decrypt） |
| `-in` / `-out` | 输入文件 / 输出文件 |

注意：加密和解密使用的是**同一个密码** `mypassword`，这就是“对称”的含义。

### 1.4 什么是盐值（Salt）？

盐值是加密前混入密码的一个**随机字符串**。

#### 为什么需要盐值？

> **类比**：你用同一个密码锁了两个一模一样的箱子。没有盐值时，两把锁长得完全一样；攻击者就能利用预计算对照表快速猜出密码。加盐后，即使密码和内容相同，结果也不同，对照表就失效。

**技术细节**：
- 不加 `-salt`：同一密码 + 同一文件 -> 每次生成**相同密文**（危险）
- 加了 `-salt`：同一密码 + 同一文件 -> 每次生成**不同密文**（更安全）
- openssl 会把盐值写在密文开头，解密时自动读取

### 1.5 对称加密的优缺点

| 维度 | 说明 |
|------|------|
| **优点：速度快** | 计算效率高，适合加密大量数据 |
| **优点：实现简单** | 算法相对简单 |
| **缺点：密钥分发问题** | 必须先把密钥安全共享，密钥若被截获，加密失效 |
| **缺点：密钥管理困难** | 人越多，需要管理的密钥数量爆炸增长 |

### 1.6 为什么不能用对称加密解决所有问题？

假设有 $N$ 个人，任意两人都要安全通信，每对通信都需要一把独立密钥。

密钥总数公式：

$$
K = \frac{N(N-1)}{2}
$$

代入看规模：

- $N = 10$：$K = 10 \times 9 / 2 = 45$
- $N = 1000$：$K = 1000 \times 999 / 2 = 499500$（约 50 万）

这还不包括密钥轮换、泄露吊销、权限分级等现实管理成本。也就是说，对称加密本身很快，但在大规模网络中，**密钥分发和管理会先崩溃**。

> 结论：对称加密非常重要，但它不能单独解决互联网级别的安全通信问题。

## 二、哈希函数：数据的“指纹”

### 2.1 哈希函数是什么？

哈希函数（Hash Function）把**任意长度**输入转换为**固定长度**输出（哈希值），相当于数据指纹。

注意：哈希函数**不是加密算法**，它是单向过程，不能解密还原。

### 2.2 哈希函数的四大特性

| 特性 | 说明 | 例子 |
|------|------|------|
| **单向性** | 从数据可算哈希值，但难以反推原文 | 知道指纹不能还原人 |
| **固定长度** | 输入任意长，输出固定长 | SHA256 输出 256 位 |
| **雪崩效应** | 输入改一点，输出大变 | `Hello` 和 `Hello!` 哈希完全不同 |
| **抗碰撞性** | 很难找到两个不同输入得到同一哈希 | 安全哈希核心要求之一 |

### 2.3 实操：用 openssl 计算 SHA256

```bash
# 创建测试文件
echo 'Hello World' > test.txt

# 计算 SHA256
openssl dgst -sha256 test.txt
```

只改一个字符再计算，会得到完全不同结果，这就是雪崩效应。

### 2.4 哈希函数的应用场景

| 场景 | 工作方式 | 例子 |
|------|---------|------|
| 数据完整性校验 | 文件哈希与官方值比对 | 下载镜像后比对 SHA256 |
| 数字签名 | 对哈希值签名而非对大文件签名 | 合同签名 |
| 密码存储 | 存密码摘要而非明文 | 登录比对摘要 |
| 版本控制 | 内容摘要作为对象 ID | Git 提交对象 |
| 数据去重 | 比较摘要判断重复 | 云盘秒传 |

### 2.5 哈希碰撞：为什么 MD5 不再用于安全校验？

**哈希碰撞**：找到两份不同数据，却得到相同哈希值。

关键结论：

- **MD5 已被证明可构造碰撞**，不适合安全场景（证书签名、防篡改证明等）。
- MD5 仍可用于**非安全目的的快速校验**（如传输过程中的粗略完整性检查、去重索引）。
- **SHA256 目前没有实用的碰撞攻击**，是现代安全场景常用基线。

### 2.6 彩虹表攻击与“加盐”防御

很多人以为“数据库里存的是哈希值，不是明文密码，所以安全”。这只对了一半。

**彩虹表攻击原理**：

1. 攻击者提前计算大量“常见密码 -> 哈希值”对照表。
2. 数据库泄露后，直接拿用户哈希值查表。
3. 查到就等于找回原密码。

为什么加盐有效？

- 给每个用户密码加上随机盐再哈希：`hash(password + salt)`。
- 同样的密码，在不同用户上得到不同哈希。
- 彩虹表无法复用到所有用户，攻击成本大幅上升。

实践建议：

- 不要用 MD5/SHA1 直接存密码。
- 使用带盐且可调成本的密码哈希算法（如 bcrypt、scrypt、Argon2）。

### 2.7 哈希函数的边界

哈希能做完整性校验，但不能单独保证“数据来自谁”。

要同时保证“没被改”和“确实是你发的”，要结合非对称加密做**数字签名**。

## 三、非对称加密：锁和钥匙分离

### 3.1 核心思想

非对称加密使用两把不同密钥：

- 公钥（Public Key）：可公开
- 私钥（Private Key）：必须保密

> 类比：公钥像公开挂在门口的锁，任何人都能上锁；私钥在你手里，只有你能开锁。

### 3.2 实操准备：先生成密钥对

#### 第一步：生成私钥

```bash
openssl genpkey -algorithm RSA -out private_key.pem -pkeyopt rsa_keygen_bits:2048
```

#### 第二步：从私钥导出公钥

```bash
openssl rsa -pubout -in private_key.pem -out public_key.pem
```

`private_key.pem` 必须严格保密，`public_key.pem` 可以公开分发。

### 3.3 场景一：加密通信（公钥加密，私钥解密）

目标：让“只有私钥持有者能读到消息”。

#### 实操

```bash
# Bob 创建消息
echo "My name is Bob." > message.txt

# Bob 用你的公钥加密
openssl pkeyutl -encrypt -pubin -inkey public_key.pem -in message.txt -out encrypted_message.bin

# 你用私钥解密
openssl pkeyutl -decrypt -inkey private_key.pem -in encrypted_message.bin -out decrypted_message.txt
```

#### 流程图

```text
发送方                         接收方
  |                               |
  |--(获取接收方公钥)------------>| 
  |--(公钥加密消息)-------------->| 
  |                               |--(私钥解密)
```

结论：攻击者即使拿到密文和公钥，没有私钥也无法解密。

### 3.4 场景二：数字签名（私钥签名，公钥验签）

目标：证明“消息是我发的，而且没被改”。

#### 实操

```bash
# 创建文件
echo "I agree." > contract.txt

# 私钥签名
openssl dgst -sha256 -sign private_key.pem -out signature.bin contract.txt

# 公钥验签
openssl dgst -sha256 -verify public_key.pem -signature signature.bin contract.txt
# 通过时输出: Verified OK
```

#### 为什么先哈希再签名？

- 签名对象变成固定长度摘要，效率更高。
- 任何改动都会改变摘要，验签立即失败。

结论：验签成功同时说明两件事：

1. 身份真实性：签名来自私钥持有者。
2. 数据完整性：内容在签名后未被改动。

### 3.5 非对称加密的性能局限

非对称算法（如 RSA）计算代价高，尤其不适合直接加密大块数据。

粗略量级对比（工程直觉层面）：

- 同样处理 1MB 数据，**AES 往往比 RSA 快 1000 倍以上量级**。
- 实际实现中，RSA 通常只用于加密“很小的数据”（如会话密钥），而不是加密整段业务报文。

这就是为什么 HTTPS 不是“全程 RSA 加密数据体”，而是：

1. 用非对称加密安全交换对称密钥。
2. 后续大流量数据用对称加密传输。

### 3.6 ECC 简介（一句话）

**ECC（椭圆曲线密码）**能在更短密钥长度下提供与 RSA 相当的安全强度，因此在移动端和现代协议中越来越常见。

## 四、三种技术全面对比

| 维度 | 对称加密 | 哈希函数 | 非对称加密 |
|------|---------|---------|-----------|
| 密钥特点 | 同一个密钥加解密 | 无密钥（单向） | 公钥/私钥分离 |
| 主要用途 | 大数据加密 | 完整性校验、摘要 | 密钥交换、数字签名 |
| 速度 | 快 | 很快 | 慢 |
| 典型算法 | AES | SHA256 | RSA、ECC |
| 可逆性 | 可逆 | 不可逆 | 加密场景可逆 |
| 适用场景 | 文件/磁盘/流量加密 | 文件校验、密码摘要 | HTTPS、证书、SSH |

## 五、三者如何协作：以 HTTPS 为例

真实系统里三者是组合拳：

1. 服务器提供公钥（证书体系保证可信）。
2. 客户端生成随机对称密钥。
3. 客户端用服务器公钥加密该对称密钥并发送。
4. 双方拿到同一对称密钥后，用 AES 加密后续数据。
5. 使用哈希/消息认证机制校验完整性。

结论：

- 非对称加密负责“安全建立信任与密钥”。
- 对称加密负责“高性能传输数据”。
- 哈希负责“完整性与摘要能力”。
