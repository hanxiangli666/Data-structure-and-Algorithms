# 现代加密技术入门笔记

> **核心概览**：现代加密技术是信息安全的基石，主要包括三种核心技术——**对称加密**、**哈希函数**、**非对称加密**。本笔记结合 `openssl` 命令实操，帮你彻底理解它们的原理、区别和应用场景。

---

## 一、对称加密：一把钥匙开一把锁

### 1.1 核心思想

加密和解密使用**同一个密钥**。

> **类比**：对称加密就像你和朋友约定了一个"暗号规则"——比如每个字母往后移 3 位。你们都知道这个规则，所以你能加密，朋友能解密。但问题是，你怎么安全地把这个"规则"告诉朋友呢？

### 1.2 从凯撒密码说起

最古老的对称加密算法——**凯撒密码（Caesar Cipher）**：

```
明文:  h  e  l  l  o
       ↓  ↓  ↓  ↓  ↓    每个字母向后移动 3 位（密钥 = 3）
密文:  k  h  o  o  r
```

| 术语 | 对应内容 |
|------|---------|
| 明文 | `hello`（原始信息） |
| 密文 | `khoor`（加密后的信息） |
| 密钥 | `3`（移动的位数） |

凯撒密码太简单，很容易被暴力破解（总共只有 25 种可能）。现代对称加密算法（如 **AES**、**DES**）要复杂得多，但本质一样——加密方和解密方**共享同一个密钥**。

### 1.3 实操：用 openssl 体验 AES 加密

```bash
# 1. 创建测试文件
echo "This is a secret message" > secret.txt

# 2. 加密：使用 AES-256-CBC 算法，密码为 mypassword
openssl enc -aes-256-cbc -salt -in secret.txt -out secret.enc -k mypassword

# 3. 解密
openssl enc -aes-256-cbc -d -in secret.enc -out decrypted.txt -k mypassword

# 4. 查看解密结果
cat decrypted.txt
# 输出: This is a secret message
```

**命令参数解读**：

| 参数 | 含义 |
|------|------|
| `enc -aes-256-cbc` | 使用 AES-256-CBC 算法加密 |
| `-salt` | 添加随机盐值，增加安全性 |
| `-k mypassword` | 指定加密密码 |
| `-d` | 执行解密操作（decrypt） |
| `-in` / `-out` | 输入文件 / 输出文件 |

注意：加密和解密使用的是**同一个密码** `mypassword`——这就是"对称"的含义。

### 1.4 什么是盐值（Salt）？

盐值是加密前混入密码的一个**随机字符串**。

#### 为什么需要盐值？

> **类比**：假设你用同一个密码锁了两个一模一样的箱子。如果没有盐值，两个箱子的锁看起来完全一样——攻击者发现了这个规律，就可以用"彩虹表"（预先计算好的密码-密文对照表）快速破解。加了盐值后，即使密码和内容都相同，每次锁出来的样子都不一样，彩虹表就失效了。

**技术细节**：
- 不加 `-salt`：同一密码 + 同一文件 → 每次生成**相同密文**（危险！）
- 加了 `-salt`：同一密码 + 同一文件 → 每次生成**不同密文**（安全）
- openssl 会把盐值存储在密文文件的**开头**（前 8 字节），解密时自动读取，用户无需额外保存

> 你可以自己试试：去掉 `-salt` 参数多次加密同一文件，会发现密文完全相同；加上 `-salt` 后，每次密文都不同。

### 1.5 对称加密的优缺点

| 维度 | 说明 |
|------|------|
| **优点：速度快** | 计算效率高，适合加密大量数据 |
| **优点：实现简单** | 算法相对简单 |
| **缺点：密钥分发问题** ⭐ | 加密方和解密方必须共享密钥，但在不安全的信道上，密钥可能被截获，加密形同虚设 |
| **缺点：密钥管理困难** | N 个人两两通信需要 N×(N-1)/2 个密钥。例：10 个人需要 45 个密钥！ |

**适用场景**：本地加密（磁盘加密、文件加密等），不涉及密钥传输的场景。

> **关键问题**：怎么把密钥安全地传给对方？这个"密钥分发问题"催生了**非对称加密**技术。但在讲非对称加密之前，我们先了解另一个重要工具——哈希函数。

## 二、哈希函数：数据的"指纹"

### 2.1 哈希函数是什么？

哈希函数（Hash Function）可以把**任意长度**的数据转换成一个**固定长度**的字符串（哈希值），相当于数据的"指纹"。

> **类比**：哈希函数就像一台"指纹提取机"——你把任何东西放进去（一篇文章、一部电影、一个软件），它都会吐出一个固定长度的指纹。不同的东西指纹不同，但你无法通过指纹反推出原始物品。

**注意**：哈希函数**不是加密算法**！它是单向的，不能"解密"还原原始数据。

### 2.2 哈希函数的四大特性

| 特性 | 说明 | 例子 |
|------|------|------|
| **单向性** | 从数据可以轻松算出哈希值，但**无法**从哈希值反推出原始数据 | 知道指纹无法还原出人 |
| **固定长度** | 无论输入数据多大，输出长度固定 | SHA256 永远输出 64 个十六进制字符（256 位） |
| **雪崩效应** | 输入哪怕只改一个字节，输出就完全不同 | "Hello World" 和 "Hello World!" 的哈希值天差地别 |
| **唯一性** | 不同数据几乎不可能产生相同哈希值（哈希冲突概率极低） | 两个不同文件的"指纹"不会撞车 |

### 2.3 实操：用 openssl 计算 SHA256 哈希值

```bash
# 创建测试文件
echo 'Hello World' > test.txt

# 计算 SHA256 哈希值
openssl dgst -sha256 test.txt
# 输出: SHA256(test.txt)= d2a84f4b8b650937ec8f73cd8be2c74add5a911ba64df27458ed8229da804a26
```

只改一个字符（加了感叹号）：

```bash
echo 'Hello World!' > test.txt

openssl dgst -sha256 test.txt
# 输出: SHA256(test.txt)= 03ba204e50d126e4674c005e04d82e84c21366780af1f43bd54a37816b6ab340
```

两个哈希值**完全不同**——这就是"雪崩效应"。

### 2.4 哈希函数的五大应用场景

| 应用场景 | 工作方式 | 具体例子 |
|---------|---------|---------|
| **数据完整性验证** | 下载文件后计算哈希值，与官方提供的值对比 | 下载 Ubuntu ISO 后，对比官网给出的 SHA256 值，确认文件未被篡改 |
| **数字签名** | 先算哈希值，再对哈希值签名（效率更高） | 对一个 1GB 的合同文件签名，只需对 64 字符的哈希值签名 |
| **密码存储** | 数据库存密码的哈希值，而非明文 | 用户登录时，对输入的密码算哈希，与数据库中的哈希值比对 |
| **版本控制** | Git 用 SHA-1 哈希值作为 commit ID | 每次提交代码，Git 根据内容生成唯一的 commit ID |
| **数据去重** | 比较文件哈希值判断是否重复 | 云盘上传前先算哈希值，如果服务器已有相同哈希值的文件就跳过上传（秒传） |

### 2.5 哈希函数的局限

> **重要**：哈希函数只能计算"指纹"，**不能加密数据**，也**不能防篡改**。

为什么不能防篡改？

> **类比**：你给朋友寄一封信，信封上贴了这封信的指纹。但如果快递员拆开信改了内容，然后重新算一个新指纹贴上去，你朋友收到后对比指纹——完美匹配，根本发现不了问题！

**解决方案**：需要结合非对称加密的**数字签名**功能——用私钥对哈希值签名，攻击者没有私钥就无法伪造签名。

## 三、非对称加密：锁和钥匙分离

### 3.1 核心思想

加密和解密使用**不同的密钥**——公钥（Public Key）和私钥（Private Key）。

> **类比**：
> - **对称加密** = 密码锁保险箱：密码和锁是一体的，寄保险箱就必须同时寄密码，不安全。
> - **非对称加密** = 锁+钥匙保险箱：接收方自制一把锁和钥匙，把**锁**寄给发送方（公钥公开），发送方用锁锁住保险箱寄回，钥匙始终在接收方手里（私钥保密），其他人没有钥匙就打不开。

### 3.2 公钥和私钥的关系

| 密钥 | 谁持有 | 能做什么 |
|------|-------|---------|
| **公钥（Public Key）** | 任何人（公开发布） | 加密数据、验证签名 |
| **私钥（Private Key）** | 仅自己（严格保密） | 解密数据、生成签名 |

**两大应用场景**：

| 场景 | 流程 | 目的 |
|------|------|------|
| **加密通信** | 公钥加密 → 私钥解密 | 保证只有私钥持有者能读取信息 |
| **数字签名** | 私钥签名 → 公钥验证 | 证明信息确实来自私钥持有者，且未被篡改 |

### 3.3 实操一：生成密钥对

#### 第一步：生成私钥

```bash
openssl genpkey -algorithm RSA -out private_key.pem -pkeyopt rsa_keygen_bits:2048
```

| 参数 | 含义 |
|------|------|
| `genpkey` | 生成密钥 |
| `-algorithm RSA` | 使用 RSA 算法（最经典的非对称加密算法） |
| `-out private_key.pem` | 私钥保存到 `private_key.pem` |
| `-pkeyopt rsa_keygen_bits:2048` | 密钥长度 2048 位（当前推荐安全长度） |

> ⚠️ `private_key.pem` 是你的私钥文件，**一定要保管好，绝不能泄露！**

#### 第二步：从私钥生成公钥

```bash
openssl rsa -pubout -in private_key.pem -out public_key.pem
```

| 参数 | 含义 |
|------|------|
| `rsa` | 处理 RSA 密钥 |
| `-pubout` | 输出公钥 |
| `-in private_key.pem` | 从私钥文件读取 |
| `-out public_key.pem` | 公钥保存到 `public_key.pem` |

`public_key.pem` 就是你的公钥，可以放心分发给任何人。

#### 为什么先生成私钥，再从私钥推导公钥？

> **类比**：想象私钥是一个复杂的数学迷宫的完整地图，公钥是这个迷宫的入口照片。从完整地图（私钥）可以轻松确定入口在哪（推导公钥），但看着入口照片（公钥）无法反推出整个迷宫的地图（私钥）。
>
> **技术原理**：以 RSA 算法为例，私钥文件包含两个巨大的质数和私钥指数等数字。公钥是从私钥中的某些数字经过数学运算得到的新数字。由于大质数分解的数学难题，无法从公钥反推出私钥。

### 3.4 实操二：加密与解密

**场景**：Bob 想给你发一条机密消息。

```bash
# Bob 创建消息
echo "My name is Bob." > message.txt

# Bob 用你的【公钥】加密消息
openssl pkeyutl -encrypt -pubin -inkey public_key.pem -in message.txt -out encrypted_message.bin
```

| 参数 | 含义 |
|------|------|
| `-encrypt` | 执行加密 |
| `-pubin -inkey public_key.pem` | 使用公钥加密 |
| `-in message.txt` | 待加密的文件 |
| `-out encrypted_message.bin` | 输出密文文件（内容是乱码） |

```bash
# 你用自己的【私钥】解密
openssl pkeyutl -decrypt -inkey private_key.pem -in encrypted_message.bin -out decrypted_message.txt

# 查看结果
cat decrypted_message.txt
# 输出: My name is Bob.
```

**流程图**：

```
Bob                                          你
 │                                            │
 │  ① 获取你的公钥 (public_key.pem)            │
 │ ←─────────────────────────────────────────  │
 │                                            │
 │  ② 用公钥加密消息 → encrypted_message.bin    │
 │ ──────────────────────────────────────────→ │
 │                                            │
 │                          ③ 用私钥解密 → 原文  │
```

即使攻击者截获了 `encrypted_message.bin` 和 `public_key.pem`，没有私钥也无法解密。

### 3.5 实操三：签名与验证

**场景**：你要发布一份公开声明（合同），需要证明确实是你本人签署的。

#### 签名过程

```bash
# 创建合同文件
echo "I agree." > contract.txt

# 用【私钥】对合同签名
openssl dgst -sha256 -sign private_key.pem -out signature.bin contract.txt
```

| 参数 | 含义 |
|------|------|
| `dgst -sha256` | 先用 SHA256 计算文件的哈希值 |
| `-sign private_key.pem` | 用私钥对哈希值签名 |
| `-out signature.bin` | 签名保存到 `signature.bin` |

> **为什么先算哈希再签名？** 两个好处：
> 1. **效率高**：哈希值长度固定（256 位），对小数据签名比对大文件签名快得多
> 2. **数据完整性**：哈希值能唯一代表原始数据

#### 验证过程

你公开发布 `contract.txt`（原文）+ `signature.bin`（签名）。任何持有你公钥的人都可以验证：

```bash
# 用【公钥】验证签名
openssl dgst -sha256 -verify public_key.pem -signature signature.bin contract.txt
```

```
# 验证成功输出:
Verified OK
```

**验证成功意味着两件事**：

| 验证结论 | 含义 |
|---------|------|
| **身份真实性** | 签名确实由与该公钥配对的私钥生成（证明是你签的） |
| **数据完整性** | `contract.txt` 从签名后一个字都没被改过 |

**验证失败的情况**：
- 修改了 `contract.txt` 的内容 → `Verification Failure`
- 用其他私钥生成的签名 → `Verification Failure`

#### 签名流程图

```
你（签名方）                                    验证方
 │                                              │
 │  ① contract.txt → SHA256 → 哈希值             │
 │  ② 私钥对哈希值签名 → signature.bin            │
 │                                              │
 │  ③ 公开发布: contract.txt + signature.bin      │
 │ ────────────────────────────────────────────→ │
 │                                              │
 │                ④ contract.txt → SHA256 → 哈希值 │
 │                ⑤ 公钥验证签名 → Verified OK ✅   │
```

## 四、三种技术全面对比

| 维度 | 对称加密 | 哈希函数 | 非对称加密 |
|------|---------|---------|-----------|
| **密钥特点** | 加密解密用**同一个**密钥 | **无密钥**，单向计算 | **公钥**加密/验证，**私钥**解密/签名 |
| **主要用途** | 大量数据加密 | 数据完整性验证、密码存储 | 密钥交换、数字签名 |
| **速度** | 快 ⚡ | 很快 ⚡⚡ | 慢 🐢 |
| **典型算法** | AES, DES | SHA256, MD5 | RSA, ECC |
| **可逆性** | ✅ 可逆（能解密还原） | ❌ 不可逆（单向） | ✅ 可逆（能解密还原） |
| **适用场景举例** | 磁盘加密、文件加密 | 密码存储、文件校验、Git | HTTPS、数字证书、SSH 登录 |

## 五、三者如何协作？实际场景串联

在真实世界中，这三种技术通常**组合使用**，各取所长：

> **类比**：想象你要给朋友寄一份机密文件。
>
> 1. 你用一把**随机密码锁**（对称加密密钥）锁住文件——因为密码锁效率高，适合锁大件
> 2. 你用朋友给你的**挂锁**（朋友的公钥，非对称加密）锁住那把密码锁的密码——解决了密钥传输问题
> 3. 你在文件上按了一个**指纹**（哈希值），并用你的**私章**（私钥签名）盖在指纹上——朋友收到后能验证文件没被篡改，且确实是你寄的

这就是 **HTTPS 协议**的基本工作原理：

```
┌─────────────────────────────────────────────────┐
│                  HTTPS 握手过程                    │
│                                                  │
│  ① 服务器发送公钥（非对称加密）                      │
│  ② 浏览器生成随机对称密钥                           │
│  ③ 浏览器用公钥加密对称密钥，发给服务器（非对称加密）   │
│  ④ 服务器用私钥解密，得到对称密钥（非对称加密）        │
│  ⑤ 双方用对称密钥加密后续通信（对称加密，速度快）       │
│  ⑥ 通信过程中用哈希函数校验数据完整性（哈希函数）      │
└─────────────────────────────────────────────────┘
```

**为什么这样组合？**
- 非对称加密解决了**密钥分发问题**，但速度慢
- 对称加密**速度快**，适合加密大量数据
- 哈希函数确保数据**完整性**

三者互补，各取所长。
