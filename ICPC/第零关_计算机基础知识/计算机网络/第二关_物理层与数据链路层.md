# 第二关：物理层与数据链路层

---

## 一、物理层——数据到底是怎么"跑"出去的？

### 1.1 核心问题

你在键盘上敲了一个字母"A"，电脑内部把它变成了一串 0 和 1（二进制数据）。但 0 和 1 是抽象概念，它们没有腿，不会自己跑到另一台电脑上去。

所以必须把 0 和 1 **"翻译"成物理世界里可以传播的东西**，才能发出去。

### 1.2 三种传输介质

翻译成什么，取决于传输介质：

```
网线（双绞线）  →  用电压的高低来表示 0 和 1
                    高电压 = 1，低电压 = 0

光纤            →  用光的亮灭来表示 0 和 1
                    有光 = 1，无光 = 0

WiFi / 蓝牙     →  用无线电波的变化来表示 0 和 1
                    通过频率或波形的变化来编码
```

> **类比**：就像古代烽火台传军情——"有烟 = 敌人来了，没烟 = 平安"。本质都是把抽象信息翻译成物理世界能传播的信号。

### 1.3 三种介质对比

| 介质 | 速度 | 距离 | 抗干扰 | 成本 | 常见场景 |
|------|------|------|--------|------|----------|
| 双绞线（网线） | 中等 | 短（~100m） | 一般 | 低 | 家里/办公室网线 |
| 光纤 | 极快 | 远（几十km） | 很强 | 高 | 骨干网、机房 |
| 无线电波 | 看情况 | 中等 | 较弱 | 中等 | WiFi、手机上网 |

生活中的对应：

- 墙上网口到路由器那根线 → 双绞线
- 小区连到电信机房那根 → 很可能是光纤
- 手机连 WiFi → 无线电波

### 1.4 物理层只管一件事

> **把比特流（0 和 1）变成物理信号发出去，或者把收到的物理信号还原成比特流。**

它完全不关心这些 0 和 1 代表什么意思——是一张图片还是一封邮件，物理层一概不管。

> **类比**：物理层就像公路——只负责让车能跑，不关心车里装的是家具还是水果。

## 二、MAC 地址——每台设备的"身份证号"

### 2.1 问题引出

你家里有好几个设备连在同一个路由器上——电脑、手机、iPad、智能电视……路由器收到一个数据包，怎么知道该给谁？

光靠物理层没法解决——物理层只管传信号，不管"传给谁"。数据链路层用来区分设备的核心工具就是 **MAC 地址**。

### 2.2 MAC 地址是什么？

每一块网卡（有线网卡和 WiFi 网卡）出厂时，厂家就给它烧录了一个**全球唯一的编号**。

```
MAC 地址格式：  AA:BB:CC:DD:EE:FF

                6组，每组2个十六进制字符，共48位

前3组（AA:BB:CC）= 厂商编号（IEEE 分配给每个厂商的唯一标识）
后3组（DD:EE:FF）= 设备编号（厂商自己分配，保证不重复）
```

### 2.3 MAC 地址 vs IP 地址

这两个概念容易混淆，区别非常重要：

| 维度 | MAC 地址 | IP 地址 |
|------|----------|---------|
| 本质 | 硬件地址，烧在网卡里 | 逻辑地址，由网络分配 |
| 变化 | 出厂就有，一般不变 | 换个网络就可能变 |
| 作用范围 | 局域网内识别"你是谁" | 整个互联网上定位"你在哪" |
| 类比 | 身份证号（走到哪都是这个号） | 收货地址（搬家了就变） |

> **类比强化**：快递小哥在小区里送件，靠的是门牌号（MAC）；但快递公司规划跨城路线，靠的是城市+街道地址（IP）。

### 2.4 怎么查看自己的 MAC 地址

```bash
# Linux
ip link show

# 或者
ifconfig

# 输出中 link/ether 后面的就是 MAC 地址
# 例如：link/ether aa:bb:cc:dd:ee:ff
```

## 三、以太网与交换机——局域网里的"快递小哥"

### 3.1 以太网：局域网的通信规则

目前绝大多数有线局域网用的协议都叫**以太网（Ethernet）**。它规定了局域网里数据怎么打包、怎么发送。

以太网传输数据的基本单位叫**帧（Frame）**——就是数据链路层封装后的数据包裹。

### 3.2 以太网帧结构（简化版）

```
┌──────────────┬──────────────┬──────┬─────────┬──────────┐
│ 目标MAC地址   │ 源MAC地址     │ 类型  │ 数据     │ 校验码   │
│ 我要送给谁    │ 我是谁        │      │ 实际内容  │ 有没有损坏│
└──────────────┴──────────────┴──────┴─────────┴──────────┘
```

> **类比**：帧就是一个写好了"收件人"和"寄件人"的快递包裹。收件人和寄件人写的都是 MAC 地址。

### 3.3 集线器（Hub）——已淘汰的"大喇叭"

最早期的局域网用**集线器**连接设备。工作方式非常粗暴：

> 收到一个帧，原封不动地转发给**所有**连接的设备。

```
电脑A 发给 电脑C 的数据：

        电脑A ──→ Hub ──→ 电脑B（不是给我的，丢弃）
                   │──→ 电脑C（是给我的，收下！）
                   │──→ 电脑D（不是给我的，丢弃）
```

> **类比**：就像在办公室里大喊"张三，你的快递到了！"——所有人都听到了，但只有张三会来拿。

**问题**：浪费带宽、效率低、不安全（别人都能"听到"你的数据）。

### 3.4 交换机（Switch）——现代主流的"精准投递"

交换机内部维护一张 **MAC 地址表**，记录了"哪个 MAC 地址连在我的哪个口上"。

```
MAC 地址表示例：
┌──────────────────┬────────┐
│ MAC 地址          │ 端口    │
├──────────────────┼────────┤
│ AA:BB:CC:11:22:33│ 端口1   │  ← 电脑A
│ AA:BB:CC:44:55:66│ 端口2   │  ← 电脑B
│ AA:BB:CC:77:88:99│ 端口3   │  ← 电脑C
└──────────────────┴────────┘
```

收到一个帧时：

```
1. 看帧里的"目标MAC地址"
2. 查表：这个MAC在我的哪个口？
3. 只从那个口发出去，其他口不发
```

> **类比**：交换机就像小区里的**智能快递柜管理员**——他记得每户人家住哪个门，你的快递来了，直接放到你的格子里，不会挨家挨户去敲门。

### 3.5 交换机的自学习机制

交换机刚开机的时候，MAC 地址表是空的。它通过**自学习**来填表：

```
1. 电脑A（MAC: AA:...:33）从端口1发了一个帧
2. 交换机记下："AA:...:33 在端口1"
3. 目标MAC还不在表里？那先广播给所有口
4. 电脑C回复了，从端口3进来
5. 交换机又记下："AA:...:99 在端口3"
6. 下次电脑A再发给电脑C，直接从端口3送出，不用广播了
```

> 用得越久，表越完整，投递越精准。

### 3.6 集线器 vs 交换机总结

```
集线器（Hub）  ：收到数据 → 所有口都转发（大喇叭广播）→ 已淘汰
交换机（Switch）：收到数据 → 查MAC表 → 只从目标口转发（精准投递）→ 现在主流
```

## 四、ARP 协议——知道 IP 怎么找到 MAC？

### 4.1 问题引出

交换机靠 MAC 地址精准投递数据。但实际使用中：

> **你通常只知道对方的 IP 地址，不知道对方的 MAC 地址。**

比如你的电脑要发数据给 192.168.1.5，你知道目标 IP，但数据链路层打包帧的时候需要填**目标 MAC 地址**——这个你不知道。

ARP（Address Resolution Protocol，地址解析协议）就是来解决这个问题的。

### 4.2 ARP 工作方式

> **类比**：你新搬到一个小区，想给 302 房间的邻居送东西，但你不知道 302 的人长什么样。你的做法是——站在小区广场上大喊一声："请问谁是 302 的住户？" 只有 302 的人会回应："我是，我长这样。"

ARP 协议几乎一模一样：

```
第1步：电脑A 广播一个 ARP 请求（发给局域网内所有人）
       "谁的 IP 是 192.168.1.5？请告诉我你的 MAC 地址！
        顺便说一下，我是 192.168.1.2，我的 MAC 是 AA:...:33"

       ┌────────┐
       │ 电脑A  │──广播──→ 所有设备都收到
       └────────┘

第2步：所有设备都检查一下
       电脑B（192.168.1.3）："不是找我的，忽略"
       电脑C（192.168.1.5）："是找我的！"

第3步：电脑C 单播回复一个 ARP 响应（只发给电脑A）
       "我是 192.168.1.5，我的 MAC 是 AA:...:99"

       ┌────────┐            ┌────────┐
       │ 电脑C  │──单播──→   │ 电脑A  │
       └────────┘            └────────┘

第4步：电脑A 收到回复，记下来
       "192.168.1.5 的 MAC 是 AA:...:99"
       现在可以填上目标MAC，打包成帧，交给交换机精准投递了
```

### 4.3 ARP 缓存

每次都广播太浪费了。所以每台电脑都会把查到的结果**缓存**一段时间，下次再发给同一个 IP 就直接用缓存。

```bash
# 查看 ARP 缓存表
arp -a

# 输出示例：
# 192.168.1.1    AA:BB:CC:11:22:33   ← 你的路由器
# 192.168.1.5    AA:BB:CC:77:88:99   ← 之前查过的设备
```

缓存有过期时间（通常几分钟），过期后下次使用时会重新发 ARP 请求。

### 4.4 完整流程串联

你的电脑要给局域网内 192.168.1.5 发数据，完整过程：

```
1. 应用层产出数据，逐层封装，到了数据链路层
2. 数据链路层需要填目标 MAC 地址
3. 先查 ARP 缓存：有没有 192.168.1.5 的 MAC？
   ├── 有 → 直接用，跳到第5步
   └── 没有 → 第4步
4. 发送 ARP 广播请求，拿到对方 MAC，存入缓存
5. 填上目标 MAC，打包成帧
6. 帧交给交换机
7. 交换机查 MAC 地址表，从对应端口精准送达
```

## 五、第二关知识总结

```
物理层与数据链路层
├── 物理层
│   ├── 职责：把比特流变成物理信号发出去 / 把物理信号还原成比特流
│   ├── 传输介质
│   │   ├── 双绞线（网线）→ 电压高低表示 0 和 1
│   │   ├── 光纤 → 光的亮灭表示 0 和 1
│   │   └── 无线电波 → 频率/波形变化表示 0 和 1
│   └── 类比：公路只负责让车跑，不关心车里装什么
│
├── MAC 地址
│   ├── 每块网卡出厂烧录的全球唯一编号（48位）
│   ├── 格式：AA:BB:CC:DD:EE:FF（前3组=厂商，后3组=设备）
│   └── vs IP 地址
│       ├── MAC = 身份证号（不变，局域网内识别"你是谁"）
│       └── IP = 收货地址（可变，互联网上定位"你在哪"）
│
├── 以太网与交换机
│   ├── 以太网帧：[目标MAC | 源MAC | 类型 | 数据 | 校验码]
│   ├── 集线器（Hub）：大喇叭广播，已淘汰
│   ├── 交换机（Switch）：查MAC表精准投递，现代主流
│   └── 交换机自学习：用得越久，MAC表越完整，投递越精准
│
└── ARP 协议
    ├── 解决问题：知道对方 IP，怎么查到对方 MAC？
    ├── 方式：广播请求 → 目标单播回复
    ├── ARP 缓存：查过的结果缓存几分钟，避免重复广播
    └── 类比：在小区广场大喊"谁是302？"，302的人回应"我是"
```

## 六、下一关预告

**第三关：网络层（IP 协议族）** —— 数据怎么跨越多个网络、从源头找到目的地。

将会学习：

- IP 地址的结构与含义
- 子网划分：怎么把一个大网络切成小块
- 路由：数据经过十几个路由器，每一跳怎么决定方向
- NAT：家里几台设备怎么共用一个公网 IP
