# Linux 进程、线程、文件描述符 完整笔记

---

## 一、计算机内存空间的整体模型

在深入进程之前，先建立一个整体视角：

```
┌──────────────────────────────────────────┐
│              内存空间                    │
├──────────────────────────────────────────┤
│                                          │
│   [ 进程A ]   [ 进程B ]   [ 进程C ]      │  ← 用户空间
│                                          │
├──────────────────────────────────────────┤
│                                          │
│            内核空间                      │  ← 用户不可直接访问
│                                          │
└──────────────────────────────────────────┘
       ↑                        ↑
     磁盘                  输入输出设备
   （存储文件）           （鼠标、键盘、显示器）
```

| 空间               | 说明                                       |
| ------------------ | ------------------------------------------ |
| **用户空间** | 用户进程使用的资源（如程序中的数组、变量） |
| **内核空间** | 内核进程的系统资源，一般不允许用户直接访问 |

> 💡 有些用户进程会共享部分内核空间资源，例如动态链接库。

## 二、进程是什么

### 可执行文件 ≠ 进程

> **类比理解：**
>
> 可执行文件就像一张乐谱，而进程就像正在演奏的乐队。乐谱放在那里什么都不会发生，只有乐队（操作系统）把乐谱（程序）读进来开始演奏，音乐（程序运行）才真正发生。

```
编译 C 程序
    ↓
得到可执行文件（只是磁盘上的一个文件）
    ↓
运行（操作系统介入）
    ↓
可执行文件被读入内存 → 包装成进程 → 开始执行
```

### 进程的本质：一个数据结构

在 Linux 内核中，进程用 `task_struct` 结构体表示，也叫 **进程描述符** ：

```c
struct task_struct {
    long              state;      // 进程状态（运行/等待/停止...）
    struct mm_struct  *mm;        // 虚拟内存结构体（程序代码和数据在这里）
    pid_t             pid;        // 进程号（PID）
    struct task_struct *parent;   // 指向父进程的指针
    struct list_head  children;   // 子进程列表
    struct fs_struct  *fs;        // 文件系统信息
    struct files_struct *files;   // ⭐ 该进程打开的所有文件指针数组
};
```

其中最关键的两个字段：

| 字段      | 说明                                                    |
| --------- | ------------------------------------------------------- |
| `mm`    | 指向进程的虚拟内存（存放程序代码、数据、堆栈）          |
| `files` | **文件指针数组** ，存放该进程打开的所有文件的指针 |

## 三、文件描述符是什么

### `files` 数组：一切的核心

`files` 是一个 **文件指针数组** ，每个进程创建时，前三个位置默认填入：

```
files[0] → 标准输入流（stdin）  ← 默认：键盘
files[1] → 标准输出流（stdout） ← 默认：显示器
files[2] → 标准错误流（stderr） ← 默认：显示器
```

**文件描述符（File Descriptor）** = 这个数组的下标（索引）

```
文件描述符 0 = 输入
文件描述符 1 = 输出
文件描述符 2 = 错误
```

> **类比理解：**
>
> `files` 数组就像一个插线板，每个插孔（文件描述符）可以接不同的设备或文件。默认情况下 0 号插孔接着键盘，1 号接着显示器，2 号也接着显示器。你可以随时把某个插孔接到别的地方（重定向）。

### 进程与内核的连接图（初始状态）

```
┌──────────────────┐
│     进程          │
│                  │
│  files[0] ───────┼──── 内核 ──── 键盘（标准输入）
│  files[1] ───────┼──── 内核 ──── 显示器（标准输出）
│  files[2] ───────┼──── 内核 ──── 显示器（标准错误）
└──────────────────┘
```

> 💡 硬件由内核管理，进程需要通过**系统调用**让内核代为访问硬件资源。

### 打开新文件时

当程序打开一个额外的文件，内核将其放入 `files` 数组的下一个空位：

```
┌──────────────────┐
│     进程          │
│                  │
│  files[0] ───────┼──── 键盘
│  files[1] ───────┼──── 显示器
│  files[2] ───────┼──── 显示器
│  files[3] ───────┼──── data.txt  ← 新打开的文件（fd=3）
└──────────────────┘
```

## 四、重定向和管道的原理

理解了 `files` 数组，重定向和管道的原理就一目了然。

### 输入重定向 `<`

```bash
$ command < file.txt
```

本质：把 `files[0]` 从键盘改指向 `file.txt`

```
files[0] ───→ file.txt（而不是键盘）
```

程序读取数据时仍然访问 `files[0]`，但读到的是文件内容，程序本身完全不需要改变。

---

### 输出重定向 `>`

```bash
$ command > output.txt
```

本质：把 `files[1]` 从显示器改指向 `output.txt`

```
files[1] ───→ output.txt（而不是显示器）
```

---

### 管道 `|`

```bash
$ cmd1 | cmd2 | cmd3
```

本质：把前一个进程的 `files[1]`（输出）与后一个进程的 `files[0]`（输入）连接起来：

```
cmd1 的 files[1] ──→ 管道 ──→ cmd2 的 files[0]
cmd2 的 files[1] ──→ 管道 ──→ cmd3 的 files[0]
```

> **类比理解：**
>
> 管道就像工厂的流水线——上一道工序的出口直接接到下一道工序的入口，数据在其中流动，每个工序只管接收输入、处理、输出，互不干扰。

---

### "Linux 一切皆文件"设计哲学的优雅之处

| 资源类型         | 在进程眼中 |
| ---------------- | ---------- |
| 键盘、显示器     | 文件       |
| 磁盘上的文件     | 文件       |
| 另一个进程的输出 | 文件       |
| 网络 socket      | 文件       |

所有这些都统一装进 `files` 数组，进程通过简单的文件描述符（整数下标）访问，具体细节交给操作系统处理， **有效解耦，优美高效** 。

## 五、线程是什么

### Linux 的核心观点

> **在 Linux 内核看来，线程和进程没有本质区别** ——线程和进程都用 `task_struct` 结构体表示。

唯一的区别在于： **共享的数据区域不同** 。

| 对比项                      | 进程（`fork()`）       | 线程（`pthread()`）  |
| --------------------------- | ------------------------ | ---------------------- |
| **内存（`mm`）**    | 拷贝副本（独立）         | 共享父进程的 `mm`    |
| **文件（`files`）** | 拷贝副本（独立）         | 共享父进程的 `files` |
| **本质**              | 数据不共享的 task_struct | 数据共享的 task_struct |

### 图示对比

**多进程：各自拥有独立的内存和文件**

```
父进程 task_struct          子进程 task_struct
├── mm ──→ [内存空间A]      ├── mm ──→ [内存空间A的副本]
└── files ──→ [文件表A]    └── files ──→ [文件表A的副本]
```

**多线程：共享内存和文件**

```
主线程 task_struct          子线程 task_struct
├── mm ──→ ┐               ├── mm ──→ ┘ 指向同一块内存！
└── files ──→ ┐            └── files ──→ ┘ 指向同一个文件表！
              └────────────────────┘
                    共享
```

> **类比理解：**
>
> 多进程就像把一份文件复印了多份，每人拿一份各自修改，互不影响；
>
> 多线程就像多人同时修改同一份原件，效率更高，但需要协调（加锁），否则会乱。

## 六、为什么多线程比多进程更常用？

| 对比项             | 多进程               | 多线程                 |
| ------------------ | -------------------- | ---------------------- |
| **数据隔离** | 完全隔离（各有副本） | 共享同一份数据         |
| **数据安全** | 天然安全，无需加锁   | 需要锁机制             |
| **通信成本** | 高（进程间通信复杂） | 低（直接读写共享内存） |
| **适用场景** | 数据独立的并发任务   | 需要共享数据的并发任务 |

**为什么现实中多线程更普遍？**

> 现实中数据共享的并发场景更多。
>
> **例子：** 银行账户并发取款
>
> * ✅ 多线程：10 个人同时从同一账户取钱，余额正确减少 100 元
> * ❌ 多进程：每人拿到账户的一份副本，各自减少 10 元，结果完全错误

---

## 七、多线程的注意事项

由于多线程共享内存， **必须使用锁机制** ，防止多个线程同时写入同一区域导致数据错乱。

> **类比理解：**
>
> 共享内存就像多人共用一块白板。如果两个人同时写，字会叠在一起，谁也看不清。锁就像"正在使用，请勿打扰"的牌子——一个人写完之前，其他人不能动。

---

## 八、Linux 多进程的性能优化：Copy-on-Write

你可能会担心：`fork()` 创建子进程时要拷贝整个内存空间，不会很慢吗？

Linux 使用了 **Copy-on-Write（写时复制）** 策略：

```
fork() 时：
  不真正复制父进程内存 → 父子进程先共享同一份内存（只读）

当某方需要写入时：
  才真正复制那一部分内存 → 各自拥有独立副本
```

效果：`fork()` 创建进程非常迅速，与 `pthread()` 创建线程效率相近。
