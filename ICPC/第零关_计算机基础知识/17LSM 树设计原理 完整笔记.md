# LSM 树设计原理 完整笔记

---

## 一、先看问题：每秒写入 100 万条日志，B+ 树会怎样？

假设你在做日志平台或时序数据库，写入模式是：

1. 写入量极大（每秒几十万到上百万条）。
2. 每条数据到来就要落盘，不能丢。
3. 查询有，但远少于写入。

如果直接用传统 B+ 树，问题会逐渐出现：

1. 写入是随机写，页分裂、页合并、锁竞争频繁。
2. 磁盘 I/O 变碎，吞吐上不去。
3. 高并发下尾延迟变大。

LSM 树的核心思路就是：
把“随机改”改成“顺序写 + 后台整理”，用读路径更复杂来换取写入吞吐。

---

## 二、LSM 树 vs B+ 树（先给全局图）

| 对比维度 | B+ 树（MySQL InnoDB 常见） | LSM 树（RocksDB/LevelDB/Cassandra 常见） |
| --- | --- | --- |
| 数据可变性 | 可变，原地修改 | 不可变，追加新版本 |
| 写入模式 | 随机写为主 | 顺序写为主 |
| 读路径 | 单棵树查找，路径稳定 | 可能查 memtable + 多层 SSTable |
| 写入吞吐 | 中等 | 高 |
| 写放大来源 | 页分裂、二级索引维护 | WAL + flush + compaction |
| 典型优势场景 | 读多写少、复杂事务 OLTP | 写多读少、日志流、时序、消息 |

一句话记忆：
B+ 树更像“精细整理好的档案柜”；LSM 更像“先快速记流水账，再批量归档”。

---

## 三、B 树和 B+ 树到底差在哪（补齐基础）

很多资料直接讲 B+ 树，但你需要先知道它和 B 树的区别。

| 对比项 | B 树 | B+ 树 |
| --- | --- | --- |
| 数据存放 | 内部节点和叶子节点都可存 value | 只有叶子节点存完整 value |
| 内部节点作用 | 既做索引也存数据 | 主要做索引导航 |
| 范围查询 | 需要中序遍历整棵树 | 叶子节点链表顺序扫，天然更友好 |
| 磁盘局部性 | 相对一般 | 通常更好，工程实现更主流 |

为什么数据库普遍用 B+ 树：

1. 内部节点只放 key，可容纳更多分支，树更矮。
2. 范围查询特别常见，叶子链表很实用。
3. 更适合页式存储和磁盘预读。

## 四、LSM 树的核心组件

LSM 不是单个结构，而是一整套写入和整理机制。

```text
写请求
    -> WAL（先记日志，保底恢复）
    -> memtable（内存有序结构：跳表/红黑树）
    -> 达阈值后 flush 为 SSTable（磁盘有序不可变文件）
```

### 1. WAL（Write-Ahead Log）

先写 WAL，再确认写成功。即使进程崩溃，也能通过 WAL 重放恢复 memtable 未落盘数据。

### 2. memtable

内存里的有序表，负责承接高并发写入。满了就冻结并刷盘。

### 3. SSTable

磁盘上的有序不可变文件。不可变意味着“不会原地改”，新值以新记录形式写入。

## 五、写入、删除、更新怎么做

```text
set(key, value):
    1) append WAL
    2) insert memtable
    3) memtable 满了 -> flush 到 L0 SSTable
```

删除不是物理立刻删除，而是写墓碑（tombstone）。

| 操作 | 底层行为 |
| --- | --- |
| 新增 | 写一条新 key |
| 更新 | 对同 key 写更新版本 |
| 删除 | 写 tombstone |

最终由 compaction 清理旧版本和墓碑。

---

## 六、层级结构与 Compaction

典型 leveled compaction 形态：

```text
L0: 最新文件，可能有重叠
L1: 较旧，key range 基本不重叠
L2/L3...: 更旧更大
```

当某层文件数量或大小超阈值，会触发 compaction：

1. 选出 key 范围重叠的 SSTable。
2. 多路归并排序。
3. 丢弃旧版本和墓碑。
4. 生成新 SSTable 下沉到下一层。

### 写放大（Write Amplification）是什么

写放大指的是：
应用只写入 1 次数据，但存储引擎内部可能因为 flush + 多轮 compaction 把同一条数据重写多次。

可粗略理解为：

$$
    ext{Write Amplification} = \frac{\text{设备实际写入字节}}{\text{应用写入字节}}
$$

例如应用写 1GB，磁盘监控显示实际写了 8GB，则写放大约为 $8$。

这是 LSM 的核心代价之一。

## 七、读取路径与布隆过滤器

读取某个 key 的常见路径：

1. 先查 memtable（最新）。
2. 再查 L0 到 Ln 的 SSTable。
3. 每个候选 SSTable 先过 Bloom Filter，再决定是否真的读文件块。

### Bloom Filter 机制（必须会）

Bloom Filter 是“概率型集合判断结构”。

它的结论只有两类：

1. 判断“不在集合里”：一定不在（无假阴性，no false negative）。
2. 判断“可能在集合里”：可能误判（有假阳性，false positive）。

这正好适合 LSM：

1. 如果 Bloom 说“不在”，直接跳过这个 SSTable，省一次磁盘读。
2. 如果 Bloom 说“可能在”，再去读索引和数据块验证。

因此 Bloom 的价值是大幅减少“无效读盘”。

## 八、性能量级直觉：RocksDB vs MySQL（NVMe 场景）

下面只给“量级直觉”，不是严格基准。
真实结果受硬件、schema、压缩、WAL 策略、批量大小、并发度、事务隔离级别影响很大。

在常见 NVMe + 多核服务器下，纯写入导向 workload 往往会看到：

1. RocksDB（LSM）单机写入吞吐常能到“几十万到百万级 ops/s”量级。
2. MySQL InnoDB（B+ 树）如果是强事务、二级索引较多、随机写明显，通常会低一个量级或更多。

要点不是具体数字，而是机制差异：

1. LSM 借助顺序写和后台 compaction 吃下突发写流。
2. B+ 树在高写压下要持续维护页结构和索引一致性。

所以工程选型常见结论是：

1. 写入优先、可接受读路径复杂度，优先考虑 LSM。
2. 强事务 + 高比例点查/范围查 + 成熟 SQL 生态，优先考虑 B+ 树数据库。

---

## 九、总结

LSM 树本质是“以读换写”的工程权衡：

1. 优点：写入吞吐高，适合日志流、时序、KV 高写场景。
2. 代价：读路径更复杂，存在写放大和 compaction 调优成本。
3. 关键优化：Bloom Filter、合理层级策略、压缩与 compaction 参数调优。

只要你抓住这条主线，LSM 的细节都会顺下来。
