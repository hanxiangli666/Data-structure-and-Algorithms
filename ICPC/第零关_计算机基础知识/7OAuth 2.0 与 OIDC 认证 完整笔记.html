01

空白白卡 vs 智能胸牌

为什么标准的 OAuth 2.0 无法用来做“登录”？

假设你暑期去联合实验室做 DSSI 的人工智能课题研究。实验室有一套极其严格的安保系统。

纯 OAuth 2.0（授权）

实验室发给你一张**空白的白色感应卡 (Access Token)**。刷卡能打开 AI 算力机房的门。

致命问题：当你想登录实验室内部电脑时，系统要求输入用户名。你举起白卡，系统说：“这卡只能证明你有权进门，但我怎么知道你是谁？”

OIDC（认证 + 授权）

实验室为你特制了一张**智能学生胸牌**。胸牌里面不仅内置了开门的芯片 (Access Token)，胸牌表面还印着你的照片、姓名、学号 (ID Token)。

完美解决：OIDC 不仅给你了开门的权限，还把“你是谁”这件事业界标准化了。这就是为什么它可以做**第三方登录**。

02

拆解智能胸牌

ID Token 里究竟装了什么秘密？

在 OIDC 中，这张“印着照片的胸牌”叫做 ID Token。它本质上是一段数字签名的 JSON 数据。

如果你把你应用的授权请求里加上 scope=openid 这个参数，服务器就会在返回 Access Token 的同时，附赠一张 ID Token 给你的后端。

ID Token Payload 
{
  "iss": "https://accounts.google.com",
  "sub": "student_james_001",
  "aud": "your_app_client_id",
  "exp": 1738278000,
  "nonce": "nonce_abc123",
  "name": "Hanxiang Li"
}
大白话翻译

胸牌签发方：Google 认证中心。

这名学生的唯一学号标识（Subject）。

这张胸牌是专门发给哪个应用看的（受众）。

胸牌过期时间戳。

为了防止造假添加的“今日防伪码”。

姓名：Hanxiang Li。

03

防重放：Nonce 的防伪艺术

为什么必须把“认证结果”和“本次请求”死死绑定？

在 OIDC 中有一个极其核心的安全参数：nonce。它和 OAuth 中的 `state` 不同：`state` 保护的是请求过程（防 CSRF），而 `nonce` 保护的是**返回的身份结果**（防重放）。

🦹
黑客 (身份截获者)

嘿嘿，我虽然破解不了 ID Token 的数字签名，但我从网络包里截获了你**昨天**登录成功后的那张 ID Token！我今天重新把这张发给你的服务器，伪装成你登录！这叫 重放攻击。

💻
你的应用后端

不好意思，我今天让你去授权时，悄悄在系统里记下了一个随机的防伪码：nonce_今日有效。我明确要求 Google 把这个防伪码印在你返回的 ID Token 里。

💻
你的应用后端

你发来的这张旧胸牌，虽然签名是真的，但里面的 `nonce` 还是昨天的！对不上今天的暗号，立刻拒绝访问！

05

架构师的决策时刻

什么时候该用 UserInfo 接口？

除了 ID Token，OIDC 还提供了一个标准的 /userinfo 接口。

情景测验：你的系统只需要知道用户的唯一 ID (`sub`) 和姓名 (`name`) 就能完成账号绑定。为了优化登录速度，你是否还需要在后端额外去请求一次 `/userinfo` 接口？
