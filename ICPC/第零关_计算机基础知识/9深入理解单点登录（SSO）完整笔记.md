# 深入理解单点登录（SSO）完整笔记

---

## 一、先从用户视角感受 SSO

先不讲协议，先看真实体验。

### 1.1 用户操作模拟

场景：你第一次打开 CRM（`crm.company.com`）。

```text
[步骤1] 打开 CRM 页面
  -> 页面提示“请登录”

[步骤2] 点击“使用公司账号登录”
  -> 浏览器跳转到统一登录页（如 login.company.com / Google / Entra）

[步骤3] 输入账号密码（或通过 MFA）
  -> 登录成功后自动跳回 CRM

[步骤4] 再打开 docs.company.com
  -> 只看到短暂跳转，自动登录成功，不再输入密码
```

这就是 SSO 给用户的核心价值：

- 第一次显式登录
- 之后跨系统“无感登录”

---

## 二、为什么需要 SSO？

### 2.1 多系统重复登录的痛点

企业常见系统：

- mail.company.com
- git.company.com
- pm.company.com
- docs.company.com
- hr.company.com

如果每个系统独立认证，用户每天都要重复登录，体验差且维护成本高。

### 2.2 共享 Cookie 为什么不是好方案

技术上可通过父域 Cookie 共享登录态，但问题明显：

1. 安全扩散：任一子域被攻破，可能威胁全域会话。
2. 架构耦合：所有系统要共享会话存储和格式。
3. 跨域受限：第三方 SaaS（不同域名）无法直接共享。

---

## 三、SSO 核心思想

引入独立认证中心（IdP/OP），业务系统统一信任它：

- 用户在认证中心登录一次
- 业务系统通过标准协议换取身份结果
- 各系统保留自身会话，避免直接共享业务 Cookie

---

## 四、企业里常见的 SSO 产品（不必从零自建）

在实际工作中，多数团队是“接入现成 IdP”，而不是自己实现完整身份系统。

常见产品：

1. Okta：企业身份管理和 SSO 产品，生态成熟。
2. Auth0：开发者友好，接入速度快。
3. Azure AD（现 Entra ID）：微软生态常见企业身份平台。
4. Keycloak（开源）：可自托管，灵活可定制。

选型一般看：

- 组织规模与合规要求
- 是否要私有化部署
- 与现有云平台/目录服务集成成本

---

## 五、SSO 核心角色

| 角色 | 说明 | OIDC 对应 |
|------|------|----------|
| 认证中心 | 统一登录、颁发身份凭证 | OP（OpenID Provider） |
| 业务应用 | 信任认证中心，消费身份结果 | RP（Relying Party） |
| 用户 | 发起登录，访问多个业务系统 | End User |

---

## 六、SSO 基本流程

### 6.1 第一次访问应用 A

```text
浏览器 -> 应用A：访问资源
应用A -> 浏览器：重定向到认证中心
浏览器 -> 认证中心：输入凭证登录
认证中心 -> 浏览器：回跳应用A并附带授权结果
应用A：验证结果并建立本地会话
```

### 6.2 再访问应用 B（SSO 生效）

```text
浏览器 -> 应用B：访问资源
应用B -> 浏览器：重定向到认证中心
认证中心：发现用户已登录（中心会话有效）
认证中心 -> 浏览器：直接回跳应用B（无需再输密码）
应用B：建立本地会话
```

关键点：

- 用户“只登录一次”依赖的是认证中心的会话状态。
- 每个业务系统仍维护自己的本地会话。

---

## 七、用 OIDC 实现 SSO（现代主流）

现代 Web/移动场景，OIDC 是最常见实现方式。

### 7.1 角色映射

| SSO 术语 | OIDC 术语 |
|---------|----------|
| 认证中心 | OP |
| 业务系统 | RP |
| 身份凭证 | ID Token |

### 7.2 安全要点

1. 每个应用拿到的 ID Token 都有自己的 `aud`。
2. 应用必须验证签名、`iss`、`aud`、`exp`、`nonce`。
3. Token 不能跨应用复用。

---

## 八、SAML 简介与 OIDC 对比

很多企业老系统历史上大量使用 SAML，现代新系统更偏向 OIDC。

### 8.1 SAML 是什么

- 基于 XML 的企业级联邦认证标准。
- 在传统企业软件、老门户、老 SaaS 集成中非常常见。

### 8.2 OIDC vs SAML（简要）

| 维度 | SAML | OIDC |
|------|------|------|
| 时代背景 | 更早，企业内网与传统门户时代 | 更现代，云原生与 API 时代 |
| 数据格式 | XML | JSON/JWT |
| 开发体验 | 相对复杂 | 更贴近现代 Web/移动开发 |
| 常见场景 | 老企业系统、遗留平台 | 新应用、移动端、前后端分离 |

实践上常见情况：

- 老系统继续跑 SAML。
- 新系统优先 OIDC。
- 身份平台同时支持两者做兼容过渡。

---

## 九、SSO 的核心权衡：集中化风险

SSO 带来统一登录体验，也带来“认证中心集中化”风险：

- 一旦认证中心被攻破，所有接入应用都可能受影响。
- 认证中心成为高价值目标，需要最高等级防护。

必须强化的安全措施：

1. 强制 MFA（至少对高风险账号和管理后台）。
2. 异常登录检测（地理异常、设备异常、IP 风险）。
3. 严格权限分层与最小权限原则。
4. 审计日志与告警闭环。
5. 密钥轮换、证书管理、灾备与演练。

一句话：SSO 的便利性来自集中，风险也来自集中。

---

## 十、SSO 登出

登录时是“中心认证 + 本地会话”，登出也要考虑两层状态。

### 10.1 仅退出当前应用

- 清除当前应用本地会话。
- 若不清除认证中心会话，用户可能很快又被自动登录。

### 10.2 单点登出（退出所有应用）

- 清除认证中心会话。
- 通知各业务应用清理本地会话。

#### 进阶（可选）：Back-Channel Logout 技术细节

这一段可第一遍跳过。

Back-Channel Logout 由认证中心通过后端通道通知各应用登出，常见做法是发送签名的 logout token，业务应用验证后删除对应用户会话。优点是不依赖浏览器前端状态，缺点是实现和运维复杂度更高。

---

## 十一、总结

```text
1) SSO 先解决的是用户体验：一次登录，多系统可用
2) 工程上依赖独立认证中心，业务系统与认证解耦
3) 现代实现主流是 OIDC；企业中仍常见 SAML
4) 不必从零开发，通常接入现成 IdP（Okta/Auth0/Entra/Keycloak）
5) 核心权衡是集中化风险，必须强化认证中心安全能力
```
