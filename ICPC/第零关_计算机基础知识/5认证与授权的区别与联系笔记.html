01

门卫与场控的职责

401 和 403 报错到底有什么区别？

想象你今晚要去一家顶级脱口秀俱乐部。这里有两道完全不同的关卡，对应着 Web 安全中最容易混淆的两个概念：

第一关在俱乐部大门外。门卫大哥需要查验你的电子票。他只关心一个问题：“你是谁？”
如果你没买票，或者票是假的，门卫大哥会把你拦在街上。在代码世界里，这叫做 认证失败，服务器会给你退回一个无情的 401 Unauthorized。

第二关在俱乐部内部。你顺利通过了门卫，坐在了台下。演出进行到一半，你突然兴奋地站起来，想冲上舞台抢过麦克风讲个段子。
这时候，舞台边的场控会一把按住你：“哥们，你是合法观众，但你没资格上台。”他关心的重点是：“你能做什么？”
在代码世界里，这叫 授权失败，服务器会给你退回一个 403 Forbidden。

一句话记忆法：

401 = 门卫说：“你谁啊？出去。”
403 = 场控说：“你是观众，但麦克风你不能碰。”

02

俱乐部的后台规矩

如何优雅地给不同的人分配权限？

当你向 AI 提需求：“给我的系统加个权限管理”时，你通常需要在这两种主流模型中做选择：

模型 A：RBAC（看角色发胸牌）
最简单粗暴。俱乐部有三种胸牌：观众、脱口秀演员、经理。
规定：带演员胸牌的可以进休息室；带经理胸牌的可以打开收银台。简单、清晰、好管理。

模型 B：ABAC（看条件动态判断）
随着俱乐部做大，规矩变复杂了。一个演员能随便上台吗？不行。
ABAC 允许你制定极其精细的规则：“只有当（你的身份=演员）且（当前时间=你的开放麦排期）时，你才能拿起麦克风。”

后端判断逻辑 (ABAC) 
if (user.role === 'comedian' && 
    time.now() === stage.slotted_time &&
    user.status === 'sober') {
  
  grantAccess(stage.mic);
} else {
  return 403;
}
大白话翻译

如果（这个人是个演员，并且...

现在正好轮到他上台表演的时间，并且...

他现在没喝醉（处于清醒状态））...


那就允许他拿走麦克风。

否则，

无情拒绝他（报 403 错误）。

03

偷外套的艺术：IDOR 漏洞

为什么光有登录还不够？提防最容易被忽视的“越权”。

如果说 401 和 403 是正面防守，那么 IDOR 漏洞（越权） 就是黑客最喜欢的后门偷袭。

在冬天，观众进俱乐部会把大衣存在前台，拿到一个号码牌（比如 12 号）。散场时，一个心术不正的观众拿到自己的 12 号牌，趁前台不注意，掏出笔把 12 改成了 13，然后递给前台说：“我要取 13 号大衣。”

04

安全高手的底层思维

如何在复杂系统中避免牵一发而动全身的灾难。

软件工程里有一条铁律，叫做 最小权限原则。回到俱乐部，这就好比：给灯光师一把只能开配电箱的钥匙，绝对不要把财务室保险箱的钥匙也图省事挂在一串上给他。这样即使灯光师的钥匙丢了，俱乐部的钱还在。

此外，为了防止有人拿捡来的票混进门，门卫不仅会查验你的票（密码），还会要求你掏出手机，出示购票订单的动态二维码（短信验证码/OTP）。这就是 多因素认证 (MFA)。

*注：MFA 只是让“门卫”的核查更严格（强化 AuthN），它并不会改变你在场内能做什么（不影响 AuthZ）。

实战决策：你发现你开发的一款写作助手里，用户只要在浏览器地址栏把 /docs/99 改成 /docs/100，就能偷看其他人的脱口秀手稿！作为技术主导，你应该如何修复？
