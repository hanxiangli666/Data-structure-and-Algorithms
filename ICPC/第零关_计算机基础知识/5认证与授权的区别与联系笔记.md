# 认证与授权的区别与联系 笔记

> 核心概览：认证（Authentication）解决“你是谁”，授权（Authorization）解决“你能做什么”。两者职责不同，但在系统中先后衔接。

---

## 零、最容易搞反：401 和 403（先记住）

这是开发里最常见误区，先放在最前面。

| 状态码 | 含义 | 常见场景 |
|--------|------|---------|
| **401 Unauthorized** | 认证失败（未登录、token 无效、凭证错误） | 没带 token、token 过期、密码错误 |
| **403 Forbidden** | 认证成功但授权失败（无权限） | 普通用户访问管理员接口 |

一句话记忆：

- `401` = “我不知道你是谁”
- `403` = “我知道你是谁，但你不允许做这件事”

---

## 一、核心概念辨析

### 1.1 认证（Authentication, AuthN）

一句话定义：证明“你是谁”。

系统通过凭证确认身份，得到用户唯一标识。

| 认证方式 | 凭证 | 系统确认的身份 |
|---------|------|--------------|
| 用户名 + 密码 | 正确密码 | 你是该用户名所有者 |
| 手机验证码 | 正确验证码 | 你是该手机号持有者 |
| 身份证 / 护照 | 有效证件 | 你是证件上的人 |
| 指纹 / 人脸识别 | 生物特征 | 你是该特征的拥有者 |

### 1.2 授权（Authorization, AuthZ）

一句话定义：确认“你能做什么”。

系统在知道你是谁之后，决定你拥有哪些资源访问权限。

### 1.3 鉴权（Auth）

“鉴权”常作为总称，通常指完整流程：认证 + 授权。

### 1.4 关系图

```text
认证（你是谁） -> 授权（你能做什么）
```

认证输出身份，授权基于身份做权限决策。

---

## 二、对比表

| 维度 | 认证（AuthN） | 授权（AuthZ） |
|------|--------------|--------------|
| 核心问题 | 你是谁？ | 你能做什么？ |
| 执行顺序 | 先执行 | 后执行（依赖认证结果） |
| 输入 | 凭证（密码、验证码、证件） | 身份与上下文信息 |
| 输出 | 用户 ID / 主体身份 | 允许的操作范围 |
| 失败结果 | 401 | 403 |

## 三、常见授权模型：RBAC vs ABAC

### 3.1 RBAC（基于角色的访问控制）

核心思想：权限绑定到角色，用户通过角色获得权限。

公司场景例子：

- 你是“管理员”角色，所以你能删帖。
- 你是“审计员”角色，所以你能查看日志但不能改数据。

优点：

- 规则简单，易落地。
- 适合组织架构明确、岗位固定的系统。

### 3.2 ABAC（基于属性的访问控制）

核心思想：根据“主体属性 + 资源属性 + 环境属性”动态决策。

公司场景例子：

- 这篇帖子是你写的，所以你能删。
- 当前时间在工作时段且你来自公司内网，才允许导出报表。

优点：

- 规则更细粒度、更灵活。
- 适合复杂业务和精细化权限控制。

### 3.3 一眼看懂区别

- RBAC：你是什么角色。
- ABAC：你在什么条件下对什么资源做什么操作。

可组合使用：先用 RBAC 做大框架，再用 ABAC 做细粒度补充。

## 四、例子一：酒店入住

```text
认证阶段：
前台核验身份证，确认“你是谁”。

授权阶段：
根据预订信息决定你能进哪间房、能否用健身房/SPA。
```

| 阶段 | 酒店场景 | 技术概念 |
|------|---------|---------|
| 认证 | 验身份证 | 验证用户名密码/验证码 |
| 授权 | 发房卡并限定可用设施 | 返回权限范围 |

## 五、例子二：购物网站登录

### 5.1 认证阶段

```text
POST /login
{ username, password }
-> 服务器验证通过，识别 user_id=12345
```

### 5.2 授权阶段

```text
GET /orders
Cookie: sid=abc
-> 服务器识别 user_id=12345
-> 仅返回 12345 的订单
```

关键点：认证通过不等于可以访问所有资源，授权必须继续校验资源归属。

## 六、越权漏洞（IDOR）案例与危害

IDOR（Insecure Direct Object Reference）是 OWASP Top 10 高频问题。

### 6.1 典型漏洞案例

接口：

```text
GET /orders/12345
```

错误实现：

- 只根据 URL 的订单 ID 查数据并返回。
- 没检查“当前登录用户是否是订单 12345 的所有者”。

攻击方式：

1. 用户 A 登录后访问 `GET /orders/12345`。
2. 把 URL 改成 `GET /orders/12346`。
3. 若后端未做所有权校验，就能看到用户 B 的订单。

### 6.2 危害

- 用户隐私泄露（姓名、地址、电话、消费信息）。
- 业务数据泄露，可能造成资金和合规风险。
- 进一步组合攻击（批量遍历 ID 拉取数据）。

### 6.3 防护要点

1. 服务端必须做对象级授权校验（owner check）。
2. 不能只信任前端传入的资源 ID。
3. 尽量使用不可预测资源标识（如 UUID）降低枚举成功率。
4. 关键接口加审计日志与异常访问监控。

## 七、最小权限原则（Principle of Least Privilege）

定义：默认只授予“完成当前任务所必需”的最小权限。

### 7.1 反例

错误做法：先给“大权限”，以后再慢慢收回。

问题：

- 权限回收经常滞后或遗漏。
- 一旦账号泄露，攻击面极大。

### 7.2 正确做法

1. 默认拒绝（deny by default）。
2. 按岗位/场景精确授权。
3. 临时高权限要有时效和审批。
4. 定期做权限审计与回收。

### 7.3 为什么重要

- 降低误操作和越权风险。
- 减少凭证泄露后的损害半径。
- 满足合规要求（审计可解释）。

## 八、多因素认证（MFA）简介

MFA（Multi-Factor Authentication）属于“认证增强”，不是授权概念。

常见两步验证：

- 第一因子：密码（你知道什么）
- 第二因子：短信/OTP/硬件令牌（你拥有什么）

结论：

- MFA 解决“更可靠地确认你是谁”。
- 它不会自动决定“你能访问哪些资源”。
- 所以 MFA 强化的是 AuthN，不是 AuthZ。

## 九、认证和授权在不同技术方案中的体现

| 技术方案 | 认证环节 | 授权环节 |
|---------|---------|---------|
| Session-Cookie | 登录时校验凭证并建立会话 | 请求时按会话身份检查权限 |
| JWT | 登录时校验凭证并签发 token | 验签 + claims + 资源权限校验 |
| OAuth 2.0 | 用户在授权服务器完成身份确认 | 通过 scope/claims 限定资源访问范围 |

## 十、知识点速查

| 概念 | 一句话解释 |
|------|-----------|
| 认证（AuthN） | 验证“你是谁” |
| 授权（AuthZ） | 决定“你能做什么” |
| 鉴权（Auth） | 认证 + 授权 的整体过程 |
| 401 | 认证失败 |
| 403 | 授权失败 |
| RBAC | 基于角色授予权限 |
| ABAC | 基于属性和上下文动态决策 |
| IDOR | 对象级授权缺失导致越权访问 |
| 最小权限原则 | 只给够用权限，不给多余权限 |
| MFA | 认证增强，不是授权 |
