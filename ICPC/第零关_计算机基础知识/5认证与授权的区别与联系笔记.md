# 认证与授权的区别与联系 笔记

> **核心概览**：认证（Authentication）解决"你是谁"，授权（Authorization）解决"你能做什么"。两者经常被混淆，但职责完全不同，是权限控制体系中**先后衔接**的两个阶段。

---

## 一、核心概念辨析

### 1.1 认证（Authentication, AuthN）

**一句话定义**：证明"**你是谁**"。

系统通过某种凭证来确认你的身份，获取你的**唯一标识**。

| 认证方式 | 凭证 | 系统确认的身份 |
|---------|------|--------------|
| 用户名 + 密码 | 正确的密码 | 你是该用户名的所有者 |
| 手机验证码 | 正确的验证码 | 你是该手机号的持有者 |
| 身份证 / 护照 | 有效的证件 | 你是该证件上记录的人 |
| 指纹 / 人脸识别 | 生物特征 | 你是该生物特征的主人 |

> **类比**：认证就像酒店前台让你出示身份证——目的是确认"你确实是预订房间的那个人"。

### 1.2 授权（Authorization, AuthZ）

**一句话定义**：确认"**你能做什么**"。

在知道你是谁之后，系统根据你的身份决定你有哪些权限。

| 授权场景 | 普通用户能做的 | 管理员能做的 |
|---------|--------------|------------|
| 操作系统文件 | 读取系统文件 | 读取 + 修改 + 删除 |
| 论坛帖子 | 修改**自己的**帖子 | 删除/修改**任何人的**帖子 |
| 购物网站订单 | 查看**自己的**订单 | 查看**所有用户的**订单 |

> **类比**：授权就像酒店前台确认你身份后给你房卡，告诉你"209 房间是你的，健身房也可以用，但 SPA 不在你的套餐里"。

### 1.3 鉴权（Auth）

**鉴权**是一个宽泛的概念，通常指整个权限控制过程，即 **认证 + 授权** 的组合动作。

工作中听到"鉴权"时，一般指的是完整的流程：先验证身份，再检查权限。

### 1.4 一张图理清关系

```
┌─────────────────────────────────────────────────┐
│                 鉴权（Auth）                      │
│                                                  │
│   ┌──────────────┐       ┌──────────────┐        │
│   │  认证 (AuthN)  │ ────→ │  授权 (AuthZ)  │       │
│   │  你是谁？      │       │  你能做什么？   │       │
│   │              │       │              │        │
│   │  输入：凭证    │       │  输入：用户身份  │       │
│   │  输出：用户 ID │       │  输出：权限列表  │       │
│   └──────────────┘       └──────────────┘        │
│                                                  │
│        先认证 ──────────→ 再授权                   │
└─────────────────────────────────────────────────┘
```

**总结公式**：

```
认证阶段 → 获取用户的唯一标识（你是谁）
授权阶段 → 通过唯一标识确认用户权限（你能做什么）
```

## 二、对比表

| 维度 | 认证（Authentication） | 授权（Authorization） |
|------|----------------------|---------------------|
| **缩写** | AuthN | AuthZ |
| **核心问题** | 你是谁？ | 你能做什么？ |
| **执行顺序** | **先**执行 | **后**执行（依赖认证结果） |
| **输入** | 凭证（密码、验证码、证件等） | 用户身份（认证阶段的输出） |
| **输出** | 用户唯一标识（User ID） | 权限列表（可访问的资源和操作） |
| **失败结果** | HTTP 401 Unauthorized（身份未验证） | HTTP 403 Forbidden（无权限） |
| **生活类比** | 出示身份证 | 拿到房卡，被告知能用哪些设施 |

> **常见混淆点**：HTTP 状态码 `401 Unauthorized` 的名字容易误导人——它实际上表示的是**认证失败**（未登录或凭证无效），而不是"未授权"。真正的授权失败应该返回 `403 Forbidden`。

## 三、例子一：酒店入住

用一个生活场景串联认证和授权的完整流程：

```
你走进酒店前台

┌─────────────────────────────────────────────────┐
│  认证阶段                                         │
│                                                  │
│  前台："请出示您的身份证件。"                        │
│  你：出示身份证 + 人脸验证                          │
│  前台：确认你就是"张三"（获取身份 ID）               │
└─────────────────────┬───────────────────────────┘
                      │ 身份确认 ✅
                      ↓
┌─────────────────────────────────────────────────┐
│  授权阶段                                         │
│                                                  │
│  前台通过身份证号查询预订信息                        │
│  前台："您的房间是 209，有效期到明天中午 12 点，       │
│        可以使用健身房，但 SPA 不在您的套餐内。"       │
│  给你一张房卡（令牌/凭证）                           │
└─────────────────────────────────────────────────┘
```

| 阶段 | 酒店场景 | 对应技术概念 |
|------|---------|-------------|
| 认证 | 出示身份证 + 人脸验证 | 输入用户名密码 / 验证码 |
| 获取标识 | 确认你是"张三" | 获取 User ID |
| 授权 | 给你房卡，限定房间和设施 | 返回权限列表（可访问的资源） |
| 令牌 | 房卡 | Session ID / JWT |

## 四、例子二：购物网站登录

用一个开发场景展示认证和授权的技术实现：

### 4.1 认证阶段

```
用户想查看自己的订单 → 网站要求登录

浏览器 ──── POST /login ──────→ 服务器
            { username, password }
                                  │
                                  │ 验证用户名密码
                                  │ 确认用户 ID = 12345
                                  │
浏览器 ←── Set-Cookie: sid=abc ── 服务器
```

**发生了什么**：
1. 用户输入用户名和密码
2. 服务器验证凭证，确认"你是用户 12345"——这就是**认证**
3. 服务器创建 Session，通过 `Set-Cookie` 下发 Session ID

### 4.2 授权阶段

```
浏览器 ──── GET /orders ──────→ 服务器
            Cookie: sid=abc
                                  │
                                  │ 通过 sid 查到用户 ID = 12345
                                  │ 去数据库查询：只返回 12345 的订单
                                  │ （不会返回其他人的订单）
                                  │
浏览器 ←── 你的订单列表 ───────── 服务器
```

**发生了什么**：
1. 浏览器带上 Session ID 请求订单页面
2. 服务器通过 Session ID 识别出你是用户 12345
3. 服务器**只返回属于你的订单**，确保你看不到别人的数据——这就是**授权**

> **如果没有授权会怎样？** 想象一下：你登录了购物网站（认证通过），但系统没有授权检查，你随手改一下请求中的用户 ID 就能看到别人的订单——这就是经典的**越权漏洞（IDOR）**。

## 五、认证和授权在不同技术方案中的体现

| 技术方案 | 认证环节 | 授权环节 |
|---------|---------|---------|
| **Session-Cookie** | 用户登录，服务器验证密码，创建 Session | 后续请求带 Session ID，服务器查 Session 确认权限 |
| **JWT** | 用户登录，服务器验证密码，签发 JWT 令牌 | 后续请求带 JWT，服务器验签后从 Payload 读取权限信息 |
| **OAuth 2.0** | 用户在第三方（如 Google）登录授权 | 第三方返回 Access Token，限定可访问的资源范围（scope） |

## 六、知识点速查

| 概念 | 一句话解释 |
|------|-----------|
| **认证（AuthN）** | 验证"你是谁"，获取用户唯一标识 |
| **授权（AuthZ）** | 确认"你能做什么"，基于身份分配权限 |
| **鉴权（Auth）** | 认证 + 授权的统称，完整的权限控制流程 |
| **401 Unauthorized** | 认证失败（未登录或凭证无效） |
| **403 Forbidden** | 授权失败（已认证但无权限） |
| **越权漏洞（IDOR）** | 授权缺失导致用户可以访问他人数据 |
| **先认证后授权** | 认证是授权的前提，不知道你是谁就无法判断你能做什么 |
