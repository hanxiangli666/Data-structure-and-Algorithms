01

中心档案馆的崩塌

为什么现代复杂的互联网架构，不再喜欢传统的 Session？

假设你到达了一个跨国机场。传统的 Session 机制，就像是海关**不发放任何实体签证**。你在北京总部的柜台登记身份后，柜台大妈在电脑里录入一条记录：人员ID:8899 -> 张三，允许登机。然后发给你一个写着 8899 的塑料牌（Session ID）。

当你飞到巴黎，巴黎的海关看到你的 8899 牌子，他不知道你是谁。他必须**打跨国长途电话**给北京总部的档案库查询：“喂？8899 是谁？有权限入境吗？”

你 ✈️ 巴黎海关 ☎️ 北京 Redis 档案中心

痛点：所有海关都极度依赖那个中心的档案库。
一旦档案库宕机，或者海关数量增加（微服务架构），长途电话的线路就会被彻底挤爆。

你 ✈️ 巴黎海关 ✅ 直接放行（无需打电话）

解法：海关不再依赖中心档案库，因为你出示的是一本**由官方防伪签发的电子护照 (JWT)**。

02

电子护照：签名，而不是加密

JWT 最大的误区：它从来没有把你的信息“藏起来”。

JWT 就像一本你随身携带的**电子护照**。护照上明明白白地印着你的名字、到期时间、可以去的国家。

任何人捡到你的护照，都可以翻开看（因为它是 Base64 编码 的）。但是，任何人都不敢篡改它。

如果你尝试用笔把护照上的“经济舱”划掉，改成“头等舱”，海关立刻就会发现。因为护照的最后一页，印着一个由政府防伪私钥盖下的**全息数字签名**。

🦹
伪造者 (黑客)

嘿嘿，我抓到了我的 JWT 令牌！我把它解码了，发现里面写着 {"role": "user"}。我直接改成 {"role": "admin"}，然后再重新 Base64 编码发给服务器！

👮
服务器 网关

不好意思，拒绝访问。我把你篡改后的内容，用我藏在机房里的**机密私钥**重新计算了一遍签名。发现算出来的签名，和你令牌尾巴上带的签名根本对不上！

👮
服务器 网关

JWT 的安全性来源于**签名不可伪造**，而不是内容加密。记住，千万别把银行卡号放进 JWT 里！

03

拆解 JWT 的三段解构

所有的 JWT 看起来都是三段用点（.）隔开的乱码串。

一个标准的 JWT 长这样：Header.Payload.Signature （头部 . 载荷 . 签名）。

让我们用代码直观地看看，这三段“乱码”解码后到底装了什么：

解码后的 JSON 
1. Header (封面)
{
  "alg": "RS256",
  "typ": "JWT"
}

2. Payload (个人信息页)
{
  "sub": "user-123",
  "role": "reader",
  "exp": 1758975478
}

3. Signature (防伪印章)
RS256_Sign(
  Base64(Header) + "." + Base64(Payload),
  Server_Private_Key
)
大白话翻译

告诉海关：本护照使用 JWT 格式，底部的防伪印章是用 RS256（非对称加密）算法盖的。




放具体的业务数据和 Claims(声明)：
你是谁（user-123）
你的角色（reader）
护照过期时间戳（exp）。


服务器把上面两部分拼起来，用自己的私钥“咔嚓”盖上一个谁也无法伪造的数学印章。海关只需用公开的公钥验一下印章真假即可放行。
