01

前端三剑客与运行环境

代码是如何变成你在屏幕上看到的一切的？

建造一座篮球馆

作为一名球队经理，如果你要建造一座数字化的篮球馆（也就是一个网页），你需要三种截然不同的工种配合。这就是前端的“三剑客”：HTML、CSS 和 JavaScript。

🏗️

HTML (骨架)

球馆的钢筋水泥。它定义了哪里是球场、哪里是记分牌、哪里是观众席。它只管“结构”，不管好看。

🎨

CSS (装修)

球馆的涂装。地板铺上实木，油漆区刷上球队专属配色，大屏幕调整到合适的亮度。它负责“样式”。

⚡

JavaScript (电路)

球馆的通电系统。进球时记分牌数字翻动，24秒违例时蜂鸣器响起。它负责一切“交互和行为”。

代码翻译：球员得分追踪器

让我们看看这三者是如何在一个真实的记分功能中协作的。这是一段由 JS 驱动的简单代码：

CODE 
let score = 0;

function addTwoPoints() {
  score = score + 2;
  const board = document.getElementById('scoreboard');
  board.textContent = score;
}
PLAIN ENGLISH

1. 在大脑中记住一个变量（球员分数），初始值为 0。

2. 定义一个动作叫“得两分”。

3. 当动作发生时，把大脑里的分数加 2。

4. 在 HTML 的骨架中，找到那个 ID 为“scoreboard”（记分牌）的区域。

5. 把记分牌上显示的文字，强行替换成我们刚刚计算出的新分数。

💡
这就是操控 DOM

刚才代码里寻找记分牌的动作，就是在操作 DOM。原生 JS 就像是一个亲力亲为的工人，每次数据变了，都要亲自跑去大屏幕前把数字换掉。

02

React 与状态管理

为什么我们需要现代前端框架？

从“手动推土机”到“城市规划局”

如果你玩过《城市：天际线 (Cities: Skylines)》，你一定懂这个道理：当城市很小的时候，你可以手动去建每一栋房子。但在几万人的大城市里，如果你还要手动去拆建每一栋楼，你会崩溃的。

原生 JavaScript 就喜欢“手动造房子”（手动操作 DOM）。而 React 就像是游戏里的“区域规划”功能：你只需要设定这块地是“商业区”还是“住宅区”（也就是设定状态 State），系统会自动帮你建好对应的建筑（更新网页画面）。

React 组件聊天室

看看在 React 的世界里，当我们点击“进球”按钮时，幕后发生了什么样的对话：

Btn
Button (按钮组件)

嘿，教练！用户刚刚点击了我，球员投进了一个三分球！

St
State (状态总管)

收到！我现在把记录簿上的总得分从 20 修改为 23。我更新完了，准备广播！

Re
React 引擎

大家注意，状态已经改变！我正在对比新旧蓝图（虚拟 DOM）... 发现只有“记分牌”区域需要更新。其他区域不用动！

?

这就叫 “状态驱动视图”。你再也不用写 document.getElementById 去手动修改画面了。你只管数据，React 管画面。
