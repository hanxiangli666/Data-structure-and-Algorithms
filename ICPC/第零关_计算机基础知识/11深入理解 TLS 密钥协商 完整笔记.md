# 深入理解 TLS 密钥协商 完整笔记

---

## 一、背景：有了公钥，怎么加密通信？

上一篇（数字证书和 CA）讲到，浏览器验证服务器证书后，就可以开始加密通信了。

最直接的想法：**直接用服务器公钥加密数据不就行了？**

问题：非对称加密计算开销大，加密大量数据效率很低。

**解决思路：** 用非对称加密保护一个"钥匙"，然后用这个"钥匙"做对称加密（速度快）。

---

## 二、早期方案：RSA 密钥交换

### 流程

```
1. 浏览器验证证书，提取服务器的 RSA 公钥
2. 浏览器生成一个随机数「预主密钥」（Pre-Master Secret）
3. 浏览器用服务器公钥加密「预主密钥」，发送给服务器
4. 服务器用私钥解密，双方由此推导出共享「会话密钥」
5. 后续通信全部用会话密钥进行对称加密（速度快）
```

> **类比理解：**
>
> 就像你把一把锁（会话密钥）用银行的保险柜（公钥加密）寄过去，只有银行能打开保险柜（私钥解密），拿到这把锁的钥匙。

### 致命缺陷：缺乏前向保密性（Forward Secrecy）

```
2024 年：黑客录制了你和银行的所有加密通信
          （包括加密的预主密钥 + 所有加密数据）

2026 年：银行服务器的私钥泄露

黑客可以：
  → 用泄露的私钥解密「预主密钥」
  → 推导出 2024 年的会话密钥
  → 解密 2024 年的所有通信 ← 历史数据全部暴露！
```

**根本问题：** 会话密钥的安全性完全依赖于服务器私钥的永久保密，一旦私钥泄露，**过去所有通信**都会被破解。

> 这对医疗记录、金融交易等需要长期保密的数据来说，是不可接受的风险。

 **现代 TLS（1.3）已废弃 RSA 密钥交换** ，转而使用  **Diffie-Hellman 密钥协商** 。

---

## 三、Diffie-Hellman（DH）密钥协商算法

### 核心目标

> 允许双方在 **完全公开的、被黑客监听的信道上** ，协商出一个 **只有双方知道的共享密钥** 。

这听起来很玄，但原理其实不难理解。

---

### 3.1 颜色混合比喻（理解核心思想）

> **类比理解：**
>
> Alice 和 Bob 想在黑客 Eve 的监听下，协商出一个只有他们俩知道的"秘密颜色"。

```
初始状态：
  黄色 = 公开的基础颜色（Eve 知道）
  红色 = Alice 的秘密颜色（Eve 不知道）
  蓝色 = Bob 的秘密颜色（Eve 不知道）

步骤 1：Alice 混合颜色
  黄 + 红 = 橙色 → 公开发给 Bob（Eve 看到橙色）

步骤 2：Bob 混合颜色
  黄 + 蓝 = 绿色 → 公开发给 Alice（Eve 看到绿色）

步骤 3：各自二次混合
  Alice：绿 + 红 = 黄 + 蓝 + 红 = 棕色 ✅
  Bob：  橙 + 蓝 = 黄 + 红 + 蓝 = 棕色 ✅

结果：
  Alice 和 Bob 都得到了棕色（共享密钥）
  Eve 只能看到：黄、橙、绿 → 无法得到棕色 ❌
```

**关键原理：颜色混合是不可逆的。** Eve 看到橙色，无法反推出是黄色加了什么颜色混合而成的。

这就是 DH 算法的核心：**利用数学上的「单向函数」，让监听者无法反推出秘密。**

---

### 3.2 先不谈数学：DH 的直觉版描述

先用一句“非数学语言”概括 DH：

> Alice 和 Bob 各自手里有一个秘密，只通过公开交换“处理过的结果”，最终都算出同一个共享秘密；监听者虽然看到了所有公开信息，但无法反推出双方各自的秘密。

可以把它理解为“三段式”：

1. 双方各自生成私密随机数（不公开）。
2. 双方各自把随机数经过某种单向处理后公开交换。
3. 双方再把“对方公开值 + 自己私密值”组合计算，得到同一个共享结果。

监听者缺失的是“任意一方的私密随机数”，因此无法完成最后一步。

---

### 3.3 DH 算法的数学原理

真实的 DH 算法使用 **模幂运算（Modular Exponentiation）** ，而不是颜色混合。

#### 单向函数：模幂运算

```
正向计算（容易）：已知 g、x、p，计算 g^x mod p = y
逆向计算（极难）：已知 g、p、y，反推 x ← 这就是「离散对数难题」
```

> **类比理解：**
>
> 单向函数就像绞肉机——正向（牛肉→肉馅）很容易，逆向（肉馅→牛肉）几乎不可能。

---

### 3.4 DH 算法完整流程（带具体数值）

#### 第 1 步：协商公开参数（Eve 可见）

```
p = 23（质数，作为模数）
g = 5（生成元）

注：实际应用中 p 是 2048 位或更长的大质数
```

#### 第 2 步：各自生成私钥（Eve 不可见）

```
Alice 的私钥：a = 6（随机生成，只存本地）
Bob 的私钥：  b = 15（随机生成，只存本地）
```

> ⚠️ 注意：这里的"私钥"是 **临时随机数** ，不是 RSA 私钥。

#### 第 3 步：计算并交换公钥（Eve 可见）

```
Alice 计算公钥：A = g^a mod p = 5^6 mod 23 = 8    → 发给 Bob
Bob 计算公钥：  B = g^b mod p = 5^15 mod 23 = 19   → 发给 Alice
```

#### 第 4 步：各自计算共享密钥

```
Alice 用 Bob 的公钥计算：s = B^a mod p = 19^6 mod 23 = 2
Bob 用 Alice 的公钥计算：s = A^b mod p = 8^15 mod 23 = 2

✨ 双方得到相同的共享密钥 s = 2！
```

#### 为什么结果相同？（数学解释）

```
Alice 计算：B^a mod p = (g^b)^a mod p = g^(b·a) mod p
Bob 计算：  A^b mod p = (g^a)^b mod p = g^(a·b) mod p

因为 a·b = b·a，所以结果必然相同。
```

---

### 3.5 完整流程示意图

```
公开参数：p=23, g=5（Eve 全程可见）

Alice                    Eve（监听）              Bob
  │                          │                    │
  │ 生成私钥 a=6             │  生成私钥 b=15      │
  │                          │                    │
  │ A = 5^6 mod 23 = 8       │                    │
  │──────────────────────────┼───────────────────▶│
  │                          │ 看到 A=8            │
  │                          │                    │ B = 5^15 mod 23 = 19
  │◀─────────────────────────┼────────────────────│
  │                          │ 看到 B=19           │
  │                          │                    │
  │ s = 19^6 mod 23 = 2      │    无法计算出 s ❌   │ s = 8^15 mod 23 = 2
  │                          │                    │
  ✅ 共享密钥 s=2             │                   ✅ 共享密钥 s=2
```

---

### 3.6 Eve 为什么无法破解？

Eve 拥有的信息：`p=23, g=5, A=8, B=19`

Eve 想得到共享密钥，需要知道 `a` 或 `b`。

```
从 A=8 反推 a：需要解 5^a mod 23 = 8，求 a
              → 这就是「离散对数难题」
```

当 `p` 是 2048 位大质数时，即使用超级计算机，求解也需要 **数百万年** 。

## 四、前向保密性（Forward Secrecy）

### ECDHE 的关键特性：临时密钥

**每次 TLS 连接都生成新的临时私钥，连接结束后立即从内存中删除。**

```
2024 年你访问银行：
  浏览器临时私钥 a₁ ──连接结束──▶ 从内存删除，永久消失
  服务器临时私钥 b₁ ──连接结束──▶ 从内存删除，永久消失
  共享会话密钥 s₁  ──连接结束──▶ 删除

2026 年服务器证书私钥泄露：
  黑客拥有：p, g, A₁, B₁, 加密通信数据, 证书私钥
  黑客缺少：a₁（已删除）或 b₁（已删除）
  无法计算 s₁ → 无法解密 2024 年的通信 ✅
```

### RSA vs ECDHE 前向保密对比

| 对比项                 | RSA 密钥交换                     | ECDHE 密钥协商                       |
| ---------------------- | -------------------------------- | ------------------------------------ |
| **私钥用途**     | 加密预主密钥（参与密钥协商）     | **仅用于签名验证身份**         |
| **会话密钥来源** | 浏览器单方面生成，用公钥加密传递 | 双方协商生成，任何一方都不能单独确定 |
| **私钥泄露影响** | 历史所有通信全部暴露             | **历史通信不受影响**           |
| **前向保密**     | ❌ 无                            | ✅ 有                                |
| **TLS 1.3 支持** | ❌ 已废弃                        | ✅ 强制使用                          |

> **类比理解前向保密：**
>
> RSA 就像用同一把主钥匙复制了所有房间的钥匙——主钥匙丢了，所有房间都不安全，包括过去住过的房间。
>
> ECDHE 就像每次入住酒店都换一把新锁——即使某把钥匙丢了，只影响当次入住，过去的房间完全无关。

## 五、ECDHE：实际使用的算法

### 为什么不直接用 DH，而用 ECDHE？

**ECDHE = Elliptic Curve Diffie-Hellman Ephemeral（椭圆曲线临时 DH）**

| 对比项               | 传统 DH                | ECDHE                            |
| -------------------- | ---------------------- | -------------------------------- |
| **数学基础**   | 模幂运算 `g^x mod p` | 椭圆曲线点乘 `x·G`            |
| **单向性来源** | 离散对数难题           | 椭圆曲线离散对数难题             |
| **密钥长度**   | 2048 位才够安全        | **256 位就达到同等安全性** |
| **计算速度**   | 较慢                   | **更快** ，适合移动设备    |

> **类比理解 ECDHE 的优势：**
>
> 传统 DH 就像用一个超大的密码本（2048 位）来保密，ECDHE 就像用一本小巧但更精妙的密码本（256 位）达到同样甚至更高的安全性。

```
├─────────────────────────────────────────────────────────────┤

│    用会话密钥进行对称加密（速度快）                         │

│                                                             │

```

## 七、TLS 版本差异与工程实践补充

### 7.1 TLS 1.2 vs TLS 1.3 的主要差异

| 对比项 | TLS 1.2 | TLS 1.3 |
|--------|---------|---------|
| 握手轮次 | 通常约 2-RTT（首次握手） | 1-RTT（首次握手），支持 0-RTT 恢复 |
| 密钥交换 | 可包含历史方案（如 RSA 密钥交换） | 强制 (EC)DHE，默认前向保密 |
| 加密套件 | 组合复杂、历史包袱重 | 套件简化，移除弱算法 |
| 安全基线 | 依赖配置质量 | 默认更现代、更安全 |

> 直观理解：TLS 1.3 握手更快、配置更少坑、安全默认值更高。

### 7.2 为什么 TLS 1.0/1.1 已废弃

核心原因不是“单一漏洞”，而是整体安全能力落后：

1. 依赖旧密码学组合，难满足现代安全基线。
2. 与已知攻击面（如历史上的降级与弱算法问题）兼容风险高。
3. 生态已转向 TLS 1.2+，继续保留旧版本只会扩大攻击面。

因此主流浏览器与合规标准已普遍禁用 TLS 1.0/1.1。

### 7.3 Session Resumption（会话恢复）

如果每次 HTTPS 连接都完整握手，延迟和 CPU 开销都较大。会话恢复用于减少重复协商成本。

常见方式：Session Ticket

1. 首次握手完成后，服务器给客户端一个会话票据（ticket）。
2. 下次连接客户端带上 ticket，请求恢复会话。
3. 双方基于 ticket 快速恢复，减少握手轮次与开销。

与前向保密（FS）的关系：

- 现代实现通常用恢复主密钥派生新的会话密钥，而不是复用旧会话密钥本身。
- 设计得当时，恢复机制可以兼顾性能和前向保密目标。
- 0-RTT 虽更快，但存在重放风险，通常只用于幂等请求。

### 7.4 证书固定（Certificate Pinning）

概念：客户端（尤其移动 App）预置服务器证书或公钥哈希，只接受“钉住”的证书。

作用：即便某个受信任 CA 体系被滥用或误签发，攻击者证书也可能因不匹配 pin 而被拒绝。

代价与风险：

1. 证书/公钥轮换时，若 pin 更新不及时，可能导致 App 大面积连接失败。
2. 运维策略更复杂，通常需要“主 pin + 备份 pin”并配合灰度更新。
3. 一旦发布错误 pin，往往需要强制更新 App 修复。

```
┌─────────────────────────────────────────────────────────────┐
│                   TLS 密钥协商核心要点                      │
├─────────────────────────────────────────────────────────────┤
│ 1. RSA 密钥交换的缺陷                                       │
│    私钥参与密钥协商 → 私钥泄露 → 历史通信全部暴露           │
│    TLS 1.3 已废弃                                           │

│    单向函数（模幂/椭圆曲线）使逆向计算几乎不可能             │

│                                                             │
│ 3. 前向保密的实现                                           │
│    每次连接生成新的临时私钥，连接结束后删除                  │

│                                                             │
│ 4. ECDHE 的优势                                             │
│    256 位密钥 ≈ 3072 位 RSA 的安全性                        │

│                                                             │

│    RSA 时代：既验证身份，又参与密钥协商                     │

└─────────────────────────────────────────────────────────────┘
```
