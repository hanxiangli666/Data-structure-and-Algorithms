## 一、面向对象核心概念

### 🔴 1.1 封装（Encapsulation）

 **定义** ：把数据和操作数据的方法包装在一起，对外只暴露必要的接口，隐藏内部实现细节。

 **类比** ：你开车不需要知道发动机怎么运作，只需要用方向盘和油门——这就是封装。

```java
// ❌ 没有封装：外部可以随意修改内部状态
class BankAccount {
    public double balance = 1000.0;  // 任何人都能直接改
}

// ✅ 封装良好：通过方法控制访问逻辑
class BankAccount {
    private double balance = 1000.0;  // 外部不可直接访问

    public void deposit(double amount) {
        if (amount > 0) balance += amount;  // 有逻辑保护
    }

    public double getBalance() {
        return balance;  // 只读
    }
}
```

 **为什么设计模式需要它** ：几乎所有模式都依赖封装——把变化的部分隐藏起来，让调用方只看到稳定的接口。

---

### 🔴 1.2 继承（Inheritance）

 **定义** ：子类自动获得父类的属性和方法，并可以在此基础上扩展或重写。

```java
class Animal {
    String name;
    void eat() { System.out.println(name + " is eating"); }
}

class Dog extends Animal {
    void bark() { System.out.println("Woof!"); }  // 扩展新方法
}

// Dog 既有 eat()（继承），又有 bark()（自己的）
Dog dog = new Dog();
dog.eat();   // 继承自 Animal
dog.bark();  // Dog 自己的方法
```

 **继承的问题** （重要！）：

* 父类变化会影响所有子类，耦合度高
* Java 只支持单继承，灵活性受限
* 继承层级太深时代码难以理解

> 这就是为什么设计模式笔记里反复强调" **组合优于继承** "。

---

### 🔴 1.3 多态（Polymorphism）——最关键的前置知识

 **定义** ：同一个接口/方法，在不同对象上有不同的表现。

 **类比** ：你说"叫一声"，猫叫"喵"，狗叫"汪"，鸭子叫"嘎"——同一个指令，不同的响应，这就是多态。

```java
// 父类/接口
interface Animal {
    void makeSound();
}

class Cat implements Animal {
    public void makeSound() { System.out.println("喵~"); }
}

class Dog implements Animal {
    public void makeSound() { System.out.println("汪！"); }
}

// ✅ 多态的威力：不需要知道具体是哪种动物
void makeAllAnimalsSound(List<Animal> animals) {
    for (Animal a : animals) {
        a.makeSound();  // 自动调用各自的实现
    }
}
```

 **为什么这是最重要的前置知识** ：

设计模式笔记里几乎每个模式的核心都依赖多态：

```java
// 观察者模式：不知道也不关心 b 是哪种具体观察者
for (Bidder b : bidders) {
    b.update(...);  // 多态！每个 b 各自响应
}

// 策略模式：context 不知道 strategy 是哪种具体策略
strategy.doExport(title, content);  // 多态！

// 装饰器模式：装饰器和被装饰者都是同一个接口
UserDao dao = new UserDaoCacheDecorator(new UserDaoImpl());  // 多态！
```

 **编译时多态 vs 运行时多态** ：

| 类型       | 触发时机                   | 例子                     |
| ---------- | -------------------------- | ------------------------ |
| 编译时多态 | 编译阶段确定调用哪个方法   | 方法重载（同名不同参数） |
| 运行时多态 | 程序运行时根据实际对象确定 | 方法重写（子类覆盖父类） |

> 设计模式主要利用的是 **运行时多态** 。

## 二、接口与抽象类

### 🔴 2.1 接口（Interface）

 **定义** ：只定义"能做什么"（方法签名），不实现"怎么做"的纯规范。

```java
interface Flyable {
    void fly();           // 只声明，不实现
    void land();          // 所有实现类必须提供具体实现
}

class Bird implements Flyable {
    public void fly()  { System.out.println("扇翅膀飞"); }
    public void land() { System.out.println("收翅膀降落"); }
}

class Airplane implements Flyable {
    public void fly()  { System.out.println("发动引擎起飞"); }
    public void land() { System.out.println("放起落架降落"); }
}
```

 **接口的核心价值** ：

* 一个类可以实现**多个**接口（解决Java单继承限制）
* 调用方只依赖接口，不依赖具体实现（这就是"面向接口编程"）
* 随时可以替换具体实现，调用方代码不需要改变

---

### 🔴 2.2 抽象类（Abstract Class）

 **定义** ：介于普通类和接口之间——可以有抽象方法（子类必须实现），也可以有具体方法（直接继承使用）。

```java
abstract class Shape {
    String color;  // 普通字段

    // 抽象方法：子类必须实现
    abstract double area();

    // 具体方法：直接继承，不需要重写
    void printInfo() {
        System.out.println("颜色: " + color + ", 面积: " + area());
    }
}

class Circle extends Shape {
    double radius;
    double area() { return Math.PI * radius * radius; }  // 必须实现
}
```

---

### 🔴 2.3 接口 vs 抽象类——必须搞清楚的区别

| 对比维度   | 接口                                  | 抽象类                             |
| ---------- | ------------------------------------- | ---------------------------------- |
| 实现数量   | 一个类可实现**多个**接口        | 一个类只能继承**一个**抽象类 |
| 字段       | 只能有常量（`static final`）        | 可以有普通字段                     |
| 构造函数   | 没有                                  | 有                                 |
| 方法       | 全是抽象的（Java 8后可有default方法） | 可以混合抽象和具体方法             |
| 表达的关系 | "能做什么"（行为契约）                | "是什么"（类型关系）               |

 **使用原则** ：

```
这个东西表达的是"一种行为/能力"？→ 用接口
  例：Flyable（能飞的）、Printable（能打印的）

这个东西表达的是"一种类型，且有公共实现可以复用"？→ 用抽象类
  例：Animal（动物基类，有公共的 eat() 实现）
```

 **在设计模式里** ：大多数模式用的是 **接口** ，因为接口更灵活，不限制继承关系。

---

### 🟡 2.4 面向接口编程（Programming to Interface）

这是设计模式笔记里最核心的编程思想，用一句话概括：

> **声明变量时用接口类型，而不是具体实现类型。**

```java
// ❌ 面向实现编程：紧耦合
ArrayList<String> list = new ArrayList<>();

// ✅ 面向接口编程：松耦合
List<String> list = new ArrayList<>();
// 将来想换成 LinkedList？只改右边一行，其他代码不用动
List<String> list = new LinkedList<>();
```

在设计模式中：

```java
// ✅ 发布者只认识接口
for (Bidder b : bidders) { ... }  // Bidder 是接口

// ✅ 上下文只认识策略接口
ExportStrategy strategy;  // 接口类型

// ✅ 装饰器持有的是接口
private final UserDao decoratedDao;  // 接口类型
```

## 三、组合 vs 继承

### 🔴 3.1 什么是组合

 **组合** ：一个类持有另一个类的实例作为字段，通过调用该实例的方法来复用功能。

```java
// 继承方式（"是一种"关系）
class LoggingUserDao extends UserDaoImpl {
    // 直接继承所有方法
}

// 组合方式（"有一个"关系）
class LoggingUserDao implements UserDao {
    private UserDao dao;  // 持有一个实例

    public LoggingUserDao(UserDao dao) {
        this.dao = dao;
    }

    public String getUserNameBy(int id) {
        System.out.println("查询用户: " + id);
        return dao.getUserNameBy(id);  // 委托给被持有的对象
    }
}
```

---

### 🔴 3.2 为什么"组合优于继承"

 **继承的三个核心问题** ：

**问题1：子类依赖父类的内部实现细节**

```java
class ParentList {
    int addCount = 0;
    void add(String item) { addCount++; /* ... */ }
    void addAll(List<String> items) {
        for (String item : items) add(item);  // 内部调用了 add()
    }
}

class ChildList extends ParentList {
    @Override
    void add(String item) {
        super.add(item);
        System.out.println("added: " + item);
    }
}

// 问题：调用 addAll(["a","b","c"]) 后，addCount 是多少？
// 答案取决于父类 addAll 的内部实现，子类作者根本不知道！
```

**问题2：继承暴露了不该暴露的方法**

```java
class Stack extends ArrayList {
    // Stack 只应该有 push/pop/peek
    // 但因为继承了 ArrayList，用户可以直接调用 add(0, item) 从底部插入
    // 破坏了 Stack 的语义！
}
```

 **问题3：类爆炸** （桥接模式笔记里的核心问题）

```
消息类型(3种) × 发送渠道(4种) = 12个类（用继承）
消息类型(3种) + 发送渠道(4种) = 7个类（用组合）
```

---

### 🟡 3.3 组合的优势

```java
// 组合的灵活性：运行时动态替换
class ReportExporter {
    private ExportStrategy strategy;

    // 运行时随时切换！
    public void setStrategy(ExportStrategy strategy) {
        this.strategy = strategy;
    }
}

// 继承做不到这一点——你不能在运行时换父类
```

 **记忆口诀** ：

> * 要复用 **行为** ，且两者是同类关系 → 考虑继承
> * 要复用 **功能** ，或需要灵活组合 → 用组合

## 四、SOLID 原则

这五个原则是设计模式的理论基础。笔记里反复提到"开闭原则"、"单一职责"，理解这五条后你会发现每个模式都在落地某条原则。

### 🟡 4.1 单一职责原则（Single Responsibility）

> 一个类只做一件事，只有一个引起它变化的原因。

```java
// ❌ 违反单一职责：一个类又管数据，又管格式化，又管打印
class Report {
    String data;
    String formatAsHtml()  { return "<p>" + data + "</p>"; }
    String formatAsCsv()   { return data + "\n"; }
    void printToConsole()  { System.out.println(data); }
    void saveToFile()      { /* ... */ }
}

// ✅ 各司其职
class Report       { String data; }
class HtmlFormatter { String format(Report r) { return "<p>" + r.data + "</p>"; } }
class ConsolePrinter{ void print(Report r)    { System.out.println(r.data); } }
```

 **在设计模式里** ：装饰器模式中，`UserDaoImpl` 只管数据库查询，`UserDaoCacheDecorator` 只管缓存，就是单一职责的体现。

---

### 🔴 4.2 开闭原则（Open/Closed Principle）

> **对扩展开放，对修改关闭。** 新增功能应该通过新增代码实现，而不是修改已有代码。

这是整套笔记里出现频率最高的原则。

```java
// ❌ 违反开闭原则：新增格式必须修改这个方法
String export(String content, String format) {
    if ("html".equals(format)) { /* ... */ }
    else if ("pdf".equals(format)) { /* ... */ }
    // 新增 LaTeX？改这里！引入 bug 风险++
}

// ✅ 符合开闭原则（策略模式）：新增格式 = 新增一个类，已有代码不动
interface ExportStrategy { String export(String content); }
class HtmlExporter  implements ExportStrategy { /* ... */ }
class PdfExporter   implements ExportStrategy { /* ... */ }
class LatexExporter implements ExportStrategy { /* 新增！完全独立 */ }
```

---

### 🟡 4.3 里氏替换原则（Liskov Substitution）

> 子类必须能够完全替换父类，替换后程序行为不变。

```java
class Bird {
    void fly() { System.out.println("flying..."); }
}

class Penguin extends Bird {
    // ❌ 企鹅不会飞！重写 fly() 抛异常违反里氏替换原则
    void fly() { throw new UnsupportedOperationException("I can't fly!"); }
}

// 如果代码里有：
void makeBirdFly(Bird bird) { bird.fly(); }
// 传入 Penguin 就崩了——违反了"子类可以替换父类"
```

 **实际意义** ：继承体系设计不好，会让运行时出现意外的异常或错误行为。

---

### 🟡 4.4 接口隔离原则（Interface Segregation）

> 不要让一个接口太胖——客户端不应该被迫依赖它不使用的方法。

```java
// ❌ 接口太胖：打印机不需要扫描，但被迫实现 scan()
interface MultiFunctionDevice {
    void print();
    void scan();
    void fax();
}

// ✅ 拆分成小接口
interface Printable { void print(); }
interface Scannable  { void scan(); }
interface Faxable    { void fax(); }

class SimplePrinter implements Printable { /* 只实现需要的 */ }
```

---

### 🔴 4.5 依赖倒置原则（Dependency Inversion）

> 高层模块不应该依赖低层模块，两者都应该依赖 **抽象（接口）** 。

这个原则直接解释了为什么设计模式里到处是接口：

```java
// ❌ 高层（Business）直接依赖低层（MySQLDatabase）
class BusinessLogic {
    MySQLDatabase db = new MySQLDatabase();  // 写死了 MySQL！
    void doWork() { db.query("..."); }
}

// ✅ 高层依赖抽象接口
interface Database { String query(String sql); }

class BusinessLogic {
    Database db;  // 依赖接口
    BusinessLogic(Database db) { this.db = db; }  // 注入具体实现
    void doWork() { db.query("..."); }
}

// 现在可以注入任何实现：MySQL、PostgreSQL、Mock（单元测试用）
new BusinessLogic(new MySQLDatabase());
new BusinessLogic(new MockDatabase());  // 测试时用 Mock
```

 **这就是"依赖注入（DI）"的核心思想** ，也是工厂模式系列的理论基础。

## 五、UML 类图基础

### 🟡 5.1 为什么要学 UML

设计模式的官方文档和大多数参考书都用 UML 类图描述结构。只需要认识几个符号，不需要精通画图。

---

### 🟡 5.2 必须认识的六种关系符号

```
1. 继承（Inheritance）
   子类 ──────────▷ 父类
   空心三角箭头，实线

2. 实现接口（Implementation）
   类 ─ ─ ─ ─ ─ ─▷ 接口
   空心三角箭头，虚线

3. 依赖（Dependency）
   A ─ ─ ─ ─ ─ ─→ B
   普通箭头，虚线（A 临时用到 B，如方法参数）

4. 关联（Association）
   A ──────────→ B
   普通箭头，实线（A 有一个 B 类型的字段）

5. 聚合（Aggregation）
   整体 ◇──────── 部分
   空心菱形（部分可以脱离整体独立存在）
   例：班级 ◇──── 学生

6. 组合（Composition）
   整体 ◆──────── 部分
   实心菱形（部分不能脱离整体存在）
   例：人 ◆──── 心脏
```

---

### 🟡 5.3 读一个实际例子

以策略模式为例：

```
ReportExporter ──────────────────→ ExportStrategy（接口）
      │                                    △
      │ 持有一个                           │ 实现
      │                         ┌──────────┼──────────┐
                        MarkdownExporter  HtmlExporter  LatexExporter
```

读法：

* `ReportExporter` 关联（持有）`ExportStrategy` 接口
* `MarkdownExporter`、`HtmlExporter`、`LatexExporter` 都实现了 `ExportStrategy`

## 六、Java 语言基础补充

### 🔴 6.1 访问修饰符

| 修饰符        | 本类 | 同包 | 子类 | 其他 |
| ------------- | ---- | ---- | ---- | ---- |
| `private`   | ✅   | ❌   | ❌   | ❌   |
| `protected` | ✅   | ✅   | ✅   | ❌   |
| `public`    | ✅   | ✅   | ✅   | ✅   |
| （默认）      | ✅   | ✅   | ❌   | ❌   |

设计模式中，`private` 和 `public` 是最常用的——字段基本都是 `private`，方法根据是否对外暴露选择 `public` 或 `private`。

---

### 🔴 6.2 static 关键字

```java
class Counter {
    static int count = 0;  // 类级别，所有实例共享
    int id;                // 实例级别，每个实例独有

    Counter() {
        count++;           // 每创建一个实例，共享计数器加1
        id = count;
    }
}

Counter a = new Counter();  // count = 1, a.id = 1
Counter b = new Counter();  // count = 2, b.id = 2
```

 **在设计模式里的用途** ：单例模式中 `getInstance()` 必须是 `static` 方法，因为调用它之前实例还不存在。

---

### 🔴 6.3 方法重写（Override）vs 方法重载（Overload）

```java
class Animal {
    void speak() { System.out.println("..."); }
    void speak(int times) { /* 重载：同名，不同参数 */ }
}

class Dog extends Animal {
    @Override
    void speak() { System.out.println("Woof!"); }  // 重写：子类改变父类方法的行为
}
```

设计模式主要利用**方法重写**来实现多态。

---

### 🟡 6.4 泛型（Generics）基础

笔记里的代码大量用到 `List<T>`、`Map<K, V>` 等泛型结构。

```java
// 没有泛型：类型不安全
List list = new ArrayList();
list.add("hello");
list.add(123);  // 编译不报错，运行时出问题

// 有泛型：类型安全
List<String> list = new ArrayList<>();
list.add("hello");
// list.add(123);  // 编译直接报错，更安全
```

只需要能**读懂**泛型代码即可，不需要深入学习泛型通配符等高级内容。

---

### 🟡 6.5 Lambda 表达式与函数式接口（Java 8+）

策略模式中，如果策略接口只有一个方法，可以用 Lambda 简化：

```java
// 传统写法
ExportStrategy strategy = new MarkdownExporter();

// Lambda 简化（策略接口只有一个方法时）
ExportStrategy strategy = (title, content) -> "# " + title + "\n\n" + content;
```

理解这个语法能让你读懂现代 Java 代码中更简洁的设计模式实现。

### 🟡 6.6 常用集合类

设计模式笔记里频繁出现这几个集合，需要知道基本用法：

// List：有序，可重复（观察者列表用它）
List<Observer> observers = new ArrayList<>();

## 八、软件工程基本思想

### 🟡 8.1 什么是耦合（Coupling）和内聚（Cohesion）

 **耦合** ：模块之间的依赖程度。**低耦合**是好事——一个模块的改变不影响其他模块。

 **内聚** ：一个模块内部的关联程度。**高内聚**是好事——模块只做一件事，做好它。

```
设计目标：低耦合 + 高内聚

耦合程度示例（从高到低）：
高耦合 → A 直接访问 B 的内部字段
        → A 调用 B 的具体方法
低耦合 → A 通过接口调用 B（设计模式追求的状态）
```

---

### 🟡 8.2 什么是"代码坏味道"（Code Smell）

理解这些"坏味道"，能帮你识别什么时候该用设计模式：

| 坏味道                 | 描述                       | 对应模式           |
| ---------------------- | -------------------------- | ------------------ |
| **if-else 地狱** | 一个方法里有大量条件分支   | 策略模式、状态模式 |
| **类爆炸**       | 继承导致类数量指数增长     | 桥接模式           |
| **散弹式修改**   | 新增一个功能需要改很多地方 | 观察者模式         |
| **平行继承体系** | 两套继承层次总是同步变化   | 桥接模式           |
| **数据泥团**     | 一堆参数总是一起出现       | 生成器模式         |

---

### 🟡 8.3 什么是重构（Refactoring）

> 在不改变外部行为的前提下，改善代码内部结构。

设计模式经常是重构的 **目标状态** ：你先写出能用的代码，然后识别出"坏味道"，再用对应的模式重构成更优雅的结构。

这解释了为什么笔记里每个模式都从"糟糕的做法"开始展示——不是故意写烂代码，而是模拟重构前的真实状态。

---

### 🟢 8.4 单元测试基础概念

理解设计模式为什么让代码"更容易测试"：

```java
// ❌ 紧耦合，难以测试
class BusinessLogic {
    MySQLDatabase db = new MySQLDatabase();  // 测试时必须连接真实数据库！
}

// ✅ 依赖接口，可以用 Mock 替换
class BusinessLogic {
    Database db;
    BusinessLogic(Database db) { this.db = db; }
}

// 测试时注入一个假的 Database，不需要真实数据库
Database mockDb = new MockDatabase();
BusinessLogic logic = new BusinessLogic(mockDb);
```

## 九、学习路径建议

### 推荐的学习顺序

```
第一阶段（1-2天）
    多态 + 接口 vs 抽象类 + 面向接口编程
    → 这三个搞清楚，笔记里80%的代码能看懂

第二阶段（1天）
    组合 vs 继承 + 开闭原则
    → 理解为什么模式这样设计，而不只是"看懂代码"

第三阶段（零散时间）
    其余SOLID原则 + UML + Java语法细节
    → 按需查阅，遇到不懂的再补

第四阶段（持续进行）
    读笔记 → 找自己写过的项目 → 尝试用一个模式重构 → 体会真实的痛点和收益
```

### 一个实用的自测标准

在开始看设计模式笔记之前，确认你能回答以下问题：

* [ ] `interface` 和 `abstract class` 你能说出3条区别
* [ ] 你能解释"为什么策略模式用接口而不是继承来实现算法切换"
* [ ] 你能说出"开闭原则"是什么，并举一个违反它的例子
* [ ] 你能读懂 `List<Observer> observers = new ArrayList<>()` 这行代码的每个部分

如果以上5条都能流畅回答，就可以直接开始看设计模式笔记了。
