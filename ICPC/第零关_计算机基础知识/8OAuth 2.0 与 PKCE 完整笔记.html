01

前线指挥部的困境

为什么移动端 App 和单页网页 (SPA) 守不住秘密？

在大战略推演中，最高统帅部（后端服务器）通常被重兵把守，绝对安全。因此，OAuth 2.0 敢放心地把终极机密 client_secret（通信密码本）放在后端服务器里。

但是，随着战局发展，我们派出了大量深入敌后的前线指挥部（这在代码里对应着运行在用户手机里的 iOS/Android App，或者浏览器里的 SPA）。

这些前线部队处于极其危险的环境中：手机 App 可以被黑客轻易反编译，浏览器里的代码按 F12 就能看得一清二楚。在安全术语中，它们被称为 Public Client。

致命威胁

如果我们把 client_secret 写进手机 App 的安装包里，敌军只要下载并拆解这个 App，就能拿到全局密码本，进而伪造任意请求。

02

半路杀出的间谍

如果没有 client_secret，授权码 (Auth Code) 将极其脆弱。

既然前线指挥部带不了机密密码本，那我们就干脆不带了！直接让统帅部把空投坐标（授权码 Auth Code）用明码电报发给前线指挥部。

这正是传统 App 最容易被黑客利用的漏洞：电报截获攻击。

🏰
大本营 (Auth Server)

呼叫前线 APP！用户已授权，这是你的提货凭证（授权码 Code）：XJ992。请接收：myapp://callback?code=XJ992

🦹
敌军间谍 (恶意 APP)

哈哈！我在用户的手机里也悄悄注册了 myapp:// 这个无线电频段（URI Scheme）。大本营发出来的授权码 XJ992 被我抢先截获了！

🦹
敌军间谍 (恶意 APP)

因为前线 APP 没有 client_secret，大本营根本区分不出谁是真谁是假。我现在就拿着这个截获的 Code 去换取真正的 Access Token，冒充长官调动资源！

03

PKCE：一次性战术口令

如何用数学魔法粉碎截获阴谋？

为了拯救深陷重围的前线 APP，安全专家们发明了 PKCE 协议。

它的核心思想是：既然静态密码本带不走，那我们就让前线指挥部每次请求时，现场生成一个绝密的“一次性口令”。这个口令分成了两部分：

  1. code_verifier (真口令)：绝密。前线死死攥在手里，不告诉任何人。
  2. code_challenge (谜面)：前线把真口令放进一个极其坚固的数学密码箱（SHA256算法）里锁上，然后把这个箱子大摇大摆地发给大本营。
请求授权参数 (带 PKCE) 
GET /authorize?
  client_id=frontline_app_01&
  response_type=code&
  code_challenge=wXyZ...aBcD&
  code_challenge_method=S256
大白话翻译

呼叫大本营，我是前线1号部队，请求授权码。


注意！这是我这次行动生成的数学密码箱（谜面）。

这个密码箱是用最顶级的 S256 算法打造的，没人能砸开它看到里面的真口令。

05

最高统帅的抉择

面对历史遗留的装备，你该如何做架构决策？

在配置 PKCE 时，系统中有一个参数叫做 code_challenge_method，它有两个选项：plain（平文）和 S256（加密）。

战术测验：为了兼容一些极其古老、连基础加密算法都不支持的旧电台设备，有人建议你把参数设为 `plain` (也就是密码箱直接等于真口令本身)。作为最高架构师，你会同意吗？
