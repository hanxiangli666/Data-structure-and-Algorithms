# 深入理解 JSON Web Token (JWT) 笔记

> **前置知识**：阅读本笔记前，建议先了解[非对称加密](./现代加密技术入门笔记.md)（公钥/私钥、数字签名）和 [Session-Cookie 机制](./深入理解Session和Cookie笔记.md)。
>
> **核心概览**：JWT 是一种**无状态**、**自包含**的认证令牌。服务器不需要存储任何用户信息，JWT 本身就携带了所有需要的数据，并通过数字签名保证不可伪造。

---

## 一、从 Session-Cookie 的痛点说起

### 1.1 Session-Cookie 认证回顾

传统的 Session-Cookie 认证流程：

```
① 用户登录成功
② 服务器创建 Session，存入 Redis:  abcd1234 → { user_id, permissions }
③ 服务器通过 Set-Cookie 把 sessionID 发给浏览器
④ 之后每次请求，浏览器带上 Cookie: sessionID=abcd1234
⑤ 服务器拿 sessionID 去 Redis 查询用户信息
```

**两个特征**：

| 特征 | 说明 |
|------|------|
| **有状态** | 服务器必须依赖 Redis/数据库存储 Session 映射，没有这个存储就无法认证 |
| **非自包含** | Cookie 中只有一个随机字符串 `abcd1234`，本身不携带任何用户信息 |

### 1.2 这带来什么问题？

> **类比**：Session-Cookie 就像游乐园的"寄存柜"模式——你入园时把行李存进柜子（Session 存入 Redis），拿到一个柜子号码牌（Session ID）。每次要用行李，都得回寄存处查柜子。如果寄存处关门了（Redis 宕机），你就拿不到行李了。

在分布式、微服务的场景下，这种"有状态"的设计会带来额外的复杂度——每台服务器都需要能访问同一个 Session 存储。

---

## 二、JWT 的核心思想

### 2.1 能不能不要"寄存柜"？

既然 Session-Cookie 的问题在于服务器要存东西，那我们**直接把用户信息放在请求头里**行不行？

```
Cookie: userInfo={"user_id":"test-user","permissions":["read"]}
```

这确实是**无状态**（服务器不用存东西）且**自包含**（请求自带用户信息）的。

**但有个致命问题：任何人都可以伪造！**

```
# 攻击者篡改请求头，冒充管理员
Cookie: userInfo={"user_id":"admin-user","permissions":["read","write","delete"]}
```

服务器无法判断这些数据是不是自己签发的，还是被客户端伪造的。

### 2.2 JWT 的解决方案：数字签名

JWT 的思路是：**把用户信息放在令牌中，但用私钥签名，防止伪造**。

> **类比**：JWT 就像游乐园的"手环"模式——入园时工作人员在你手环上写好你的信息（用户名、权限、有效期），然后盖上一个**防伪钢印**（数字签名）。之后你去每个项目，工作人员只要检查手环上的信息和钢印是否匹配就行，不需要回寄存处查任何东西。如果有人试图修改手环上的信息，钢印验证就会失败。

### 2.3 JWT 两大特性

| 特性 | 含义 | 对比 Session-Cookie |
|------|------|-------------------|
| **无状态（Stateless）** | 服务器**不需要**存储任何用户信息 | Session 方式需要 Redis 存储映射 |
| **自包含（Self-contained）** | JWT **本身**包含所有验证所需信息（用户 ID、权限等） | Session ID 只是一个无意义的随机字符串 |

## 三、JWT 认证流程

### 3.1 完整流程

```
① 用户提交账号密码登录
   浏览器 ──── POST /login ──────→ 服务器

② 服务器验证密码，查询用户信息
   构造 Payload: { user_id: "test-user", permissions: ["read"] }

③ 服务器用【私钥】对 Payload 签名，生成 JWT 令牌
   JWT = Base64(Header) + "." + Base64(Payload) + "." + Signature
   返回给客户端

④ 之后每次请求，客户端在请求头中带上 JWT
   Authorization: Bearer <JWT Token>

⑤ 服务器用【公钥】验证签名
   ✅ 签名有效 → 直接从 JWT 中解析出用户信息，处理请求
   ❌ 签名无效 → 拒绝请求（数据被篡改）
```

### 3.2 对比时序图

**Session-Cookie 方式**（有状态）：

```
浏览器               服务器              Redis
  │ POST /login        │                   │
  │───────────────────→│                   │
  │                    │  存 Session        │
  │                    │──────────────────→│
  │ Set-Cookie: sid    │                   │
  │←───────────────────│                   │
  │                    │                   │
  │ GET /api (带 sid)   │                   │
  │───────────────────→│                   │
  │                    │  查 Session        │
  │                    │──────────────────→│
  │                    │  返回用户信息       │
  │                    │←──────────────────│
  │ 响应数据            │                   │
  │←───────────────────│                   │
```

**JWT 方式**（无状态）：

```
浏览器               服务器
  │ POST /login        │
  │───────────────────→│
  │                    │  私钥签名生成 JWT
  │ 返回 JWT           │
  │←───────────────────│
  │                    │
  │ GET /api (带 JWT)   │
  │───────────────────→│
  │                    │  公钥验证签名 ✅
  │                    │  直接解析出用户信息（不需要查数据库！）
  │ 响应数据            │
  │←───────────────────│
```

> **关键区别**：JWT 方式中，服务器在验证阶段**完全不需要访问任何外部存储**——签名验证和信息提取都在本地完成。

## 四、JWT 的结构：三段式

一个 JWT 令牌由三部分组成，用 `.` 分隔：

```
<Header>.<Payload>.<Signature>
```

例如：

```
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9.
eyJ1c2VyX2lkIjoidGVzdC11c2VyIiwicGVybWlzc2lvbnMiOlsicmVhZCJdLC...
lZln8NiYZACSKVLSCzSTB8_VnDUp4WJHiuzO0CUrSw_Qagn...
```

每部分都是 **Base64 编码**的，可以直接解码出明文。

> ⚠️ **JWT 不是加密的！** JWT 看起来像乱码，但其实只是 Base64 编码，任何人都能解码看到内容。**永远不要在 JWT 中存放敏感信息**（如密码、银行卡号等）。
>
> **类比**：JWT 就像一封明信片——任何人都能读到上面的内容（Base64 可解码），但上面有你的签名钢印（数字签名），别人无法伪造。明信片不适合写秘密，但适合证明"这封信确实是你写的"。

### 4.1 Header（头部）

描述 JWT 的元数据：使用什么算法签名、令牌类型是什么。

```json
{
  "alg": "RS256",    // 签名算法：RSA + SHA256
  "typ": "JWT"       // 令牌类型：固定为 JWT
}
```

Base64 编码后：`eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9`

**解码验证**：

```bash
echo "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9" | base64 -d
# 输出: {"alg":"RS256","typ":"JWT"}
```

### 4.2 Payload（载荷）

存放实际需要传递的数据。

#### JWT 标准字段（均为可选）

| 字段 | 全称 | 含义 | 示例 |
|------|------|------|------|
| `jti` | JWT ID | 令牌唯一标识 | `"abc-123-def"` |
| `iss` | Issuer | 签发人/签发方 | `"labuladong.online"` |
| `sub` | Subject | 目标对象（该令牌面向谁） | `"user-12345"` |
| `aud` | Audience | 受众（谁可以使用该令牌） | `"my-app"` |
| `iat` | Issued At | 签发时间（Unix 时间戳） | `1758974578` |
| `nbf` | Not Before | 生效时间（此时间之前令牌无效） | `1758974578` |
| `exp` | Expiration Time | 过期时间（此时间之后令牌无效） | `1759147378` |

#### 自定义字段

除了标准字段，你可以自由添加**私有字段**：

```json
{
  "user_id": "test-user",          // 自定义：用户 ID
  "permissions": ["read"],          // 自定义：权限列表
  "iss": "labuladong.online",       // 标准字段：签发者
  "iat": 1758974578,                // 标准字段：签发时间
  "exp": 1759147378                 // 标准字段：过期时间
}
```

**验证时的检查**：
- `iss` 不匹配 → 令牌无效（不是我签发的）
- `exp` 已过期 → 令牌无效（过期了）
- 签名验证失败 → 令牌被篡改

### 4.3 Signature（签名）

签名是 JWT 的**核心安全机制**，保证 Header 和 Payload 没有被篡改。

**签名过程**：

```
签名输入 = Base64(Header) + "." + Base64(Payload)

签名 = RSA_Sign(私钥, SHA256(签名输入))

最终 JWT = Base64(Header) + "." + Base64(Payload) + "." + Base64(签名)
```

> 就是用私钥对 `<Header>.<Payload>` 这个字符串进行数字签名（先 SHA256 哈希，再 RSA 签名），把结果 Base64 编码后拼到最后。

**验证过程**：

```
收到 JWT → 拆分出 Header、Payload、Signature
→ 用公钥验证 Signature 是否与 Header.Payload 匹配
→ ✅ 匹配：数据可信，未被篡改
→ ❌ 不匹配：数据被篡改，拒绝
```

### 4.4 JWT 三部分总结

```
┌─────────────────────────────────────────────────────┐
│  Header（头部）                                       │
│  元数据：签名算法 + 令牌类型                            │
│  {"alg": "RS256", "typ": "JWT"}                      │
├─────────────────────────────────────────────────────┤
│  Payload（载荷）                                      │
│  实际数据：用户信息 + 标准字段                           │
│  {"user_id": "test-user", "exp": 1759147378, ...}    │
├─────────────────────────────────────────────────────┤
│  Signature（签名）                                    │
│  安全保障：私钥对 Header.Payload 签名                   │
│  防伪造、防篡改                                        │
└─────────────────────────────────────────────────────┘
```

## 五、在 jwt.io 上验证 JWT

可以在 [jwt.io](https://jwt.io) 网站上实际验证一个 JWT 令牌：

1. 在左侧粘贴 JWT 令牌字符串
2. 网站自动解码出 Header 和 Payload（因为只是 Base64 编码）
3. 在右下方粘贴对应的公钥
4. 网站验证签名 → 显示 **"Signature Verified"** 或 **"Invalid Signature"**

**亲手实验**：尝试修改 Payload 中的 `user_id` 或 `permissions`，签名验证会立即失败——因为数据被改了，但签名还是旧的，两者不匹配。

## 六、Session-Cookie vs JWT 全面对比

| 维度 | Session-Cookie | JWT |
|------|---------------|-----|
| **状态** | 有状态（服务器需存储 Session） | 无状态（服务器不存任何东西） |
| **自包含性** | 非自包含（Cookie 只存 Session ID） | 自包含（令牌本身携带用户信息） |
| **验证方式** | 拿 Session ID 去查 Redis/数据库 | 本地用公钥验证签名即可 |
| **服务器存储** | 需要 Redis 等存储 | 不需要任何存储 |
| **扩展性** | 多服务器需共享 Session 存储 | 天然支持分布式（任何有公钥的服务器都能验证） |
| **注销/撤销** | 容易（删除 Session 即可） | 困难（JWT 签发后无法主动撤销，只能等过期） |
| **令牌大小** | 小（Cookie 只存一个 ID） | 大（包含完整用户信息 + 签名） |
| **安全风险** | Session 劫持（Cookie 被盗） | 令牌泄露（JWT 被盗则在过期前一直有效） |

> **类比总结**：
> - **Session-Cookie** = 寄存柜模式：你拿号码牌，行李在柜子里，每次用都要回柜台查。柜台关了就完蛋。
> - **JWT** = 手环模式：信息直接写在手环上并盖钢印，任何工作人员都能现场验证，不需要回柜台。但手环丢了在过期前谁捡到都能用。

## 七、知识点速查

| 概念 | 一句话解释 |
|------|-----------|
| **JWT** | 一种自包含的令牌，用 Base64 编码 + 数字签名实现无状态认证 |
| **无状态** | 服务器不需要存储任何用户信息，验证完全在本地完成 |
| **自包含** | 令牌本身携带所有认证所需信息（用户 ID、权限、过期时间等） |
| **Header** | JWT 第一段，描述签名算法和令牌类型 |
| **Payload** | JWT 第二段，存放用户信息和标准字段（iss、exp 等） |
| **Signature** | JWT 第三段，用私钥对 Header.Payload 签名，防篡改 |
| **RS256** | RSA + SHA256 签名算法，用私钥签名、公钥验证 |
| **Base64** | 编码方式（不是加密！），任何人都能解码 |
| **Bearer Token** | HTTP 请求头 `Authorization: Bearer <JWT>` 的格式 |
| **jwt.io** | 在线工具，可以解码和验证 JWT 令牌 |

## 八、JWT 使用注意事项

| 注意事项 | 说明 |
|---------|------|
| **不要存敏感信息** | JWT 只是 Base64 编码，不是加密，任何人都能解码读取 |
| **设置合理的过期时间** | JWT 签发后无法主动撤销，过期时间不宜太长 |
| **使用 HTTPS** | 防止 JWT 在传输过程中被截获 |
| **私钥严格保密** | 私钥泄露 = 任何人都能签发有效 JWT |
| **验证所有标准字段** | 不仅验证签名，还要检查 `iss`、`exp` 等字段 |
