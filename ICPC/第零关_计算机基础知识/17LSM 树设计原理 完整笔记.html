01

传统档案室的崩溃

当每秒涌入 100 万条日志时，B+ 树为什么会扛不住？

假设你正在为监控系统或物联网开发一个数据库，面临着“海量写入、不准丢失”的极端挑战。在存储引擎的选型上，存在两种截然不同的流派：

传统 B+ 树：精细的档案室

像 MySQL InnoDB 这样的 B+ 树，就像一个极其严谨的档案室。每次来一份新文件（插入数据），理货员都要跑到迷宫般的书架深处，找到那份文件绝对正确的位置并塞进去。这叫**随机写**。

致命弱点：为了腾出空间，书架经常要大挪移（页分裂）。如果每秒来 100 万份文件，理货员会因为在货架间疲于奔命而直接累死 。

LSM 树：极速分拣枢纽

LSM 树（如 RocksDB / Cassandra 用的结构）改变了游戏规则。它像一个现代化的双十一分拣中心。快递车来了，直接把包裹倾泻到大厅正中央的巨大收集筐里。这叫**顺序写**。

核心优势：不找位置，先堆起来再说。用极其狂暴的写入速度吃下所有流量，至于“整理”，那是夜班工人的事 。

1. WAL (Write-Ahead Log): 监控探头与登记簿

包裹进门前，先在纸本上按顺序飞速写下一笔记录。万一突然停电，供电恢复后，只要查阅 WAL，就能知道停电前收了哪些包裹，保证数据绝对不丢 。

🛒
2. Memtable: 大厅中央的分拣筐

登记完后，包裹被扔进大厅的巨大收集筐里。这个筐是放置在计算机高速**内存**里的数据结构（如跳表）。因为不用去磁盘找位置，所以写入速度飙到极限 。

📦
3. SSTable: 塑封的不可变集装箱

当收集筐（Memtable）满了，它就会被瞬间打包，直接拖进仓库（磁盘），变成一个塑封好的 SSTable。这种“写满就封存”的方式避免了任何原地的随机修改 。

03

夜班整理与退件贴纸

既然不准修改，那我们怎么删除或更新数据？

在 LSM 的世界里，有一个铁律：**落入集装箱 (SSTable) 的数据绝对不可更改**。那如果你想删除或者更新一条旧数据怎么办？

答案是：**用写代替删**。你想删除包裹 A，系统绝不会去翻旧集装箱找它，而是直接在当前的收集筐里扔进去一个写着“取消 A”的新包裹，这个假包裹被称为 墓碑 (Tombstone) 。

👷‍♂️
夜班整理工 (Compaction 线程)

唉，白班那些人只管往仓库里堆集装箱 (SSTable)，现在仓库快塞满了，而且有很多同一个包裹的更新版和退件贴纸 (Tombstone) 散落在不同的箱子里。

👷‍♂️
夜班整理工 (Compaction 线程)

我要开启 Compaction (合并压缩) 了！我把 L0 层那几个小箱子拆开，把它们按顺序合并归类。如果发现某个包裹已经有了一张“退件贴纸”，我就把它们俩一起扔进垃圾桶，彻底释放空间 ！

👷‍♂️
夜班整理工 (Compaction 线程)

不过老板，我有个抱怨。为了保持仓库整洁，同一个包裹我可能在几天的夜班里被反复搬来搬去、写进新箱子里。用户明明只写入了 1 次，但硬盘实际上被我写了 8 次！这 写放大 (Write Amplification) 太严重了，SSD 硬盘寿命烧得很快啊 ！

04

大海捞针与智能探测仪

文件堆积如山，如何快速找回一个包裹？

LSM 树的读取路径非常“坎坷”。如果有人来取件，你必须先找大厅的收集筐 (Memtable)，没有的话，再去翻仓库里最新的 L0 层集装箱，然后再去 L1、L2 层找 。为了避免把成千上万个集装箱全撬开看一遍，我们需要一个外部黑科技。

每个集装箱在封存时，外面都贴了一个 布隆过滤器 (Bloom Filter) 智能探测标签 。

布隆过滤器的探测逻辑 
function findPackage(container, packageId) {
  const probe = container.bloomFilter.scan(packageId);
  
  if (probe === 'DEFINITELY_NOT') {
    // 探测仪发绿光
    return 'Skip this container'; 
  } 
  
  if (probe === 'MAYBE_INSIDE') {
    // 探测仪发黄光
    const data = openContainerAndSearch(container);
    return data ? data : 'False Alarm';
  }
}
大白话翻译

拿着扫描仪对着集装箱外壳“滴”一下。


如果探测仪明确回答：**“绝对不在里面！”**

太好了！连开箱的力气都省了（省了一次极慢的磁盘 I/O 操作），直接去查下一个箱子 。


如果探测仪犹豫地回答：**“可能在里面。”**

那没办法，只能把沉重的集装箱撬开（读取磁盘），去里面翻找。有可能真的找到了，也有可能是探测仪误报（假阳性，False Positive） 。

数据库架构师的抉择

没有完美的结构，只有最适合的场景。

实战架构选型：公司正在开发一个物联网（IoT）项目，数百万个传感器每秒钟都会源源不断地上报温度和湿度数据（全是写操作）。但是，业务端每天只需要偶尔生成一张昨天的温度曲线图（读操作极少）。如果你是首席架构师，为了支撑这种极端的高吞吐量写入，你应该为时序数据库选择哪种底层存储引擎？
