# LSM 树设计原理 完整笔记

---

## 一、背景：磁盘数据结构 vs 内存数据结构

### 为什么磁盘数据结构更复杂？

| 对比项             | 内存数据结构                     | 磁盘数据结构                   |
| ------------------ | -------------------------------- | ------------------------------ |
| **创建方式** | `new`一下就行，OS 自动管理布局 | 必须手动设计每个字节存什么     |
| **读取效率** | 随机访问，非常快                 | 每次只能读取固定大小块，相对慢 |
| **实现难度** | 相对简单                         | 复杂（B+ 树代码极难读懂）      |

> 💡 了解磁盘数据结构的**原理和时间复杂度**即可，不必深究代码实现。

## 二、两大存储结构流派

### 2.1 数据可变 vs 数据不可变

| 特性               | B 树（数据可变）              | LSM 树（数据不可变）      |
| ------------------ | ----------------------------- | ------------------------- |
| **写入方式** | 原地修改节点                  | 只能在尾部追加            |
| **读性能**   | ⭐⭐⭐ 强（有序树结构，logN） | ⭐⭐ 中（需要多层查找）   |
| **写性能**   | ⭐⭐ 中（分裂/合并/锁机制）   | ⭐⭐⭐ 强（顺序追加）     |
| **空间利用** | ⭐⭐⭐ 高（原地更新，不浪费） | ⭐⭐ 中（历史版本占空间） |
| **难点**     | 平衡性维护 + 并发控制         | Compact 操作 + 读取优化   |
| **适用场景** | **读多写少**            | **写多读少**        |

> **类比理解——可变 vs 不可变：**
>
> **B 树（可变）** 就像一本活页笔记本——你可以随时翻到某一页修改内容，但翻页和重新排版（维护平衡）很费劲。
>
> **LSM 树（不可变）** 就像流水账日记本——你只能在最后一页继续写，不能修改前面的内容。找东西要翻很多页，但写起来飞快。

---

### 2.2 有序 vs 无序

> **核心规律：有序度越高 → 读性能越强，但写入维护成本越高，写性能越差。**

```
全局有序（B 树）：读性能最强，写性能受限
局部有序（LSM 树）：写性能最强，读性能通过层级结构优化
完全无序：写入最快，读取最慢（线性扫描）
```

---

## 三、LSM 树的三个核心组件

LSM 树本质上是一种 **存储方案** ，而不是单一数据结构，由三个组件协同工作：

```
写入数据
    │
    ├──────────────────────────┐
    ↓                          ↓
log 文件（WAL）            memtable（内存）
（持久化操作日志）          （有序缓存，红黑树/跳表）
（防止断电数据丢失）              │
                               │ 满了之后
                               ↓
                          SSTable（磁盘）
                        （有序不可变文件）
```

### 3.1 memtable（内存有序缓存）

* **数据结构：** 红黑树或跳表（**有序**内存数据结构）
* **作用：** 缓存新写入的数据，并按**键的大小自动排序**
* **触发条件：** 当 memtable 达到一定大小时，自动转化为 SSTable 刷入磁盘

> **类比理解：**
>
> memtable 就像餐厅收银台旁的便签纸——服务员先把新订单写在便签上（快），等便签写满了，再整理归档进订单文件夹（SSTable）。

---

### 3.2 log 文件（WAL，预写日志）

* **作用：** 数据写入 memtable 的**同时**也写入 log 文件，用于**灾难恢复**
* **生命周期：** memtable 成功刷盘为 SSTable 后，对应的 log 就可以删除了

```
断电场景：
  memtable 中的数据（内存）→ 全部丢失 ❌
  log 文件（磁盘）→ 完整保留 ✅
  重启后：读取 log → 重放操作 → 恢复数据
```

---

### 3.3 SSTable（Sorted String Table）

* **本质：** 磁盘上的一个特殊格式文件，数据按**键的大小有序排列**
* **类比：** 一个有序数组，但存在磁盘上
* **不可变：** 一旦写入，内容永不修改
* **LSM 树 = 若干 SSTable 的集合**

---

## 四、LSM 树的写入流程

```
set(key, val) 写入流程：
  步骤 1：写入 log 文件（持久化，防断电丢失）
  步骤 2：写入 memtable（内存有序结构）
  步骤 3：memtable 满了 → 转化为 SSTable → 刷入磁盘（第 0 层）
```

### 增删改的实现（仅靠 set / get 两个 API）

| 操作           | 实现方式                                |
| -------------- | --------------------------------------- |
| **新增** | `set(key, val)`，key 不存在           |
| **更新** | `set(key, new_val)`，key 已存在，覆盖 |
| **删除** | `set(key, null)`，写入"墓碑"标记      |
| **查询** | `get(key)`，找最近一次操作记录        |

> 查询某个 key 时， **只需找到最近一次的操作记录** ，就知道当前值。

---

## 五、SSTable 的层级组织（Leveled Compaction）

### 层级结构

```
L0（第0层，最新数据）：[SST_a][SST_b][SST_c]
            ↓ compact
L1（第1层）：[SST_d][SST_e][SST_f][SST_g]
            ↓ compact
L2（第2层，最旧数据）：[SST_h]...[SST_z]（容量最大）
```

* **越上层：数据越新，容量上限越小**
* **越下层：数据越旧，容量上限越大**
* **同一层内：各 SSTable 的 key 范围不重叠** （compact 保证）

---

### Compact（压实）操作

**触发条件：** 某一层的 SSTable 数量超过上限

**操作过程：**

```
从当前层和下一层选出 key 范围有重叠的 SSTable
    ↓
合并（多路归并排序，类似合并多个有序链表）
    ↓
去除失效的历史记录（只保留最新操作）
    ↓
生成一个更大的 SSTable，移到下一层
```

> **类比理解：**
>
> Compact 就像定期整理抽屉——把散乱的便签（多个小 SSTable）合并成一本整洁的笔记（大 SSTable），同时划掉已经作废的内容（历史操作），释放空间。

**Compact 解决了两个问题：**

| 问题                                                | 解决方式                               |
| --------------------------------------------------- | -------------------------------------- |
| **空间放大** （同一 key 存了 100 次历史记录） | Compact 时删除失效的旧记录，只保留最新 |
| **读取效率低** （需要翻遍所有记录）           | 层级结构 + 有序性，支持二分查找        |

---

## 六、LSM 树的读取流程

### `get(key)` 查找过程

```
步骤 1：先查 memtable（内存中最新数据，直接查有序结构）
    ↓ 没找到
步骤 2：从上到下遍历各层（L0 → L1 → L2 → ...）
    │
    ├── 每层：用二分查找找到 key 范围包含目标 key 的 SSTable
    ├── 用布隆过滤器快速判断 key 是否在这个 SSTable 中
    └── 存在：在 SSTable 内部再次二分查找，找到对应 val
    ↓ 找到即返回（最新的就是正确答案）
```

### 为什么可以用二分查找？

1. **同一层内 SSTable 的 key 不重叠** → 可以二分定位到哪个 SSTable
2. **SSTable 内部 key 有序** → 可以在文件内部二分查找

> **类比理解：**
>
> 就像在图书馆找书——先根据"分类标签"（层级和 key 范围）定位到哪个书架（SSTable），再在书架内按字母顺序（有序 key）找到具体的书。布隆过滤器就像书架上贴的快速索引卡，让你一眼判断这本书在不在这个书架上。

## 七、LSM 树 vs B+ 树 全面对比

| 对比维度             | B+ 树（MySQL）       | LSM 树（RocksDB/Cassandra） |
| -------------------- | -------------------- | --------------------------- |
| **数据可变性** | 可变（原地修改）     | 不可变（只追加）            |
| **写入方式**   | 随机写（维护树结构） | 顺序追加（高效）            |
| **读取效率**   | O(logN)，极快        | 多层查找，较慢              |
| **写入效率**   | 受限（分裂/合并/锁） | 极快（内存 + 顺序写）       |
| **空间利用**   | 高效（原地更新）     | 存在放大（需 compact）      |
| **维护难点**   | 平衡性 + 并发锁      | Compact 策略 + 读优化       |
| **适用场景**   | 读多写少（OLTP）     | 写多读少（日志、时序数据）  |

## 八、知识总结

```
┌─────────────────────────────────────────────────────────────┐
│                     LSM 树核心要点                          │
├─────────────────────────────────────────────────────────────┤
│ 1. 核心思想                                                 │
│    牺牲部分读性能，换取极高的写入性能                        │
│    数据不可变 + 顺序追加 = 写入飞快                         │
│                                                             │
│ 2. 三大组件                                                 │
│    log：WAL，防断电丢数据                                   │
│    memtable：内存有序缓存，写入入口                         │
│    SSTable：磁盘有序文件，最终存储形式                      │
│                                                             │
│ 3. 写入流程                                                 │
│    写 log → 写 memtable → 满了刷盘为 SSTable（L0 层）       │
│                                                             │
│ 4. Compact 解决两个问题                                     │
│    消除历史版本，解决空间放大                               │
│    维护层级有序性，支持二分查找                             │
│                                                             │
│ 5. 读取流程                                                 │
│    memtable → L0 → L1 → ... → 二分 + 布隆过滤器            │
│    越靠上层越新，找到即返回                                 │
└─────────────────────────────────────────────────────────────┘
```
