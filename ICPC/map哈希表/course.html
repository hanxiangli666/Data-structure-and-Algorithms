01

映射的威力

为什么你需要一个能瞬间回答问题的"智能字典"？

先从一道竞赛题开始

给你 10 万个单词，问 "apple" 出现了几次。暴力做法：每次查询都把数组扫一遍，10 万次查询就是 100 亿次操作——必然超时。

1
暴力 O(n)：扫整个数组

每次查询耗时与数据量成正比。数据越多，越慢。

2
哈希表 O(1)：直接跳到答案

把"问题"（键）和"答案"（值）绑定存储。查询时无需比较，直接定位。无论有多少数据，始终只需一步。

💡 核心思想

键值存储 把"问题"和"答案"绑在一起，查找时绕过所有比较，直接跳到答案。

02

unordered_map 全操作

每一个方法，读懂它在做什么

声明、插入、访问

unordered_map 像一个可以用任意字符串编号的储物柜——不管有多少个格子，找到正确格子永远只需一步。

CODE — unordered_map.cpp 
unordered_map<string, int> mp;

mp["apple"]  = 3;
mp["banana"] = 5;
mp["cherry"] = 2;

cout << mp.size()  << "\n";
cout << mp.empty() << "\n";
cout << mp["apple"] << "\n";
PLAIN ENGLISH

创建一个哈希表 mp：键是字符串，值是整数

插入三个键值对，就像填表格：名字 → 数量

（apple=3, banana=5, cherry=2，共 3 条）

打印键值对数量 → 输出 3

打印是否为空 → 输出 0（false，不为空）

通过键 "apple" 直接取值 → 输出 3

程序在和 unordered_map "对话"

每次你操作 mp，背后就像发送了一条消息。注意最后那条——

运行中 — main.cpp
U

删除、遍历、清空

CODE — unordered_map.cpp 
// erase：删除某个键
mp.erase("banana");

if (!mp.count("banana")) {
    cout << "banana 已删除\n";
}

// 遍历：顺序随机！
for (auto& [key, val] : mp) {
    cout << key << " -> " << val << "\n";
}

mp.clear();
PLAIN ENGLISH

从哈希表中移除 "banana" 键和它对应的值

count() 检查 "banana" 是否还存在（0=不存在）

打印确认：banana 已删除

结构化绑定 auto& [key, val] 同时获取键和值

⚠️ 顺序随机——每次运行输出顺序可能不同

清空整个哈希表，回到初始空状态

⚠️ 竞赛注意

unordered_map 遍历顺序不可预测。题目若要求有序输出，必须改用 map（第 4 章讲）。

03

[] 的隐形陷阱

最常见的 WA 来源——一行看起来安全的代码，悄悄污染了你的数据

访问 = 创建

大多数语言里，访问不存在的键会报错。C++ 的 map/unordered_map 不一样：

💣 危险行为

mp["grape"]

如果 "grape" 不存在，这行代码会自动创建 "grape" → 0，并让 size() 悄悄加 1。

CODE — 两个文件相同的陷阱 
mp["apple"]  = 3;
mp["banana"] = 5;
mp["cherry"] = 2;

// ⚠️ grape 不存在！
cout << mp["grape"] << "\n";
cout << mp.size() << "\n";
PLAIN ENGLISH

插入 apple=3, banana=5, cherry=2，共 3 个键值对

访问 "grape"——它不存在！

输出：0（自动创建，默认值 0）

输出：4——等等，我们只插入了 3 个！

"grape" 被 [] 悄悄插入，size 变成 4 了

安全模式：先问，再取

用 count() 在访问前确认键存在，不会产生任何副作用：

CODE — 安全写法 
// count：判断键是否存在（返回 0 或 1）
if (mp.count("apple")) {
    cout << mp["apple"] << "\n";
}

if (!mp.count("durian")) {
    cout << "durian 不存在\n";
}
PLAIN ENGLISH

count() 检查 "apple" 是否存在，返回 1 = 存在，不创建任何东西

确认存在后，再用 [] 取值 → 输出 3，安全

! 取反：count() 返回 0 时条件为真，即"不存在时执行"

输出确认消息，全程 size() 没有任何变化

❌
危险写法

if (mp["key"] > 0) { ... }

// 若 key 不存在，已被创建！

✅
安全写法

if (mp.count("key") &&
mp["key"] > 0) { ... }
// 先确认存在，再取值

💡
例外：统计频次

mp[word]++

// 此时自动创建→0 是有意为之

04

map：自动排序的兄弟

当你需要按键的顺序输出时，它是唯一选择

图书馆书架 vs 随机仓库

unordered_map 像随机仓库——你知道存了什么，但货架顺序不可预测。

map 像图书馆书架——所有书（键）按字母顺序摆好，随时可以按序浏览。

🌳
底层：红黑树

红黑树 在每次插入时自动调整结构，保证所有键始终有序

⏱️
速度：O(log n)

O(log n) 比 unordered_map 慢，但换来了自动排序的能力

🔤
遍历：自动升序

始终按键升序——字符串按字典序，整数按数值大小

同样的插入，不同的输出顺序

CODE — map.cpp 
map<string, int> mp;

mp["banana"] = 5;
mp["apple"]  = 3;
mp["cherry"] = 2;

for (auto& [key, val] : mp) {
    cout << key << " -> " << val << "\n";
}
// 输出：apple->3, cherry->2, grape->0
// （按字母顺序，不是插入顺序）
PLAIN ENGLISH

声明有序映射——只把 unordered_map 改成 map

按任意顺序插入 banana, apple, cherry

map 内部自动按键排序：apple < banana < cherry

遍历时，按字典序输出（不是插入顺序）

输出顺序：apple→3，cherry→2，grape→0

这是和 unordered_map 最大的区别

💡 竞赛技巧

题目要求"按字典序输出所有键值对"？直接用 map 存储，遍历时自动有序，不需要额外 sort 步骤。

切换只需改一个词

好消息：map 和 unordered_map 的所有操作写法完全一致。

1
声明时换一个词

unordered_map<K,V> → map<K,V>

2
[] 的陷阱同样存在

map 的 [] 访问不存在的键也会自动创建！用 count() 先检查。

3
其余操作完全相同

size(), empty(), count(), erase(), clear(), 遍历——所有写法不变。

竞赛实战：选哪个？

在比赛中 30 秒内做出正确决策

性能对比一览

特性unordered_mapmap
底层结构哈希表红黑树
查找/插入速度O(1) 平均O(log n)
遍历顺序随机（不可预测）按键升序排列
最坏时间O(n)（哈希碰撞）O(log n) 始终稳定
内存稍高（哈希桶开销）稍低
竞赛推荐默认选它！需要排序时选
🏆 竞赛经验

大多数题目对输出顺序无要求，直接用 unordered_map 速度更快。只有明确要求有序输出（或题目隐含字典序）时，才切换到 map。
